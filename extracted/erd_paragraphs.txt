[0042] (Heading 1) 2. Database Architecture Overview
[0043] () PreOne हा PostgreSQL 16 वर build आहे. PostgreSQL ची choice खालील reasons ने आहे: ACID compliance, native UUID support, jsonb with GIN indexing, native enum types, row-level security (RLS) for multi-tenancy, partial + expression indexes, point-in-time recovery (PITR), logical replication for read replicas, आणि strong ecosystem (Prisma, pgAdmin, psql, pgvector for AI). PostgreSQL 16 च्या UUID v7 support ने time-sortable primary keys मिळतात जे multi-region writes सह सहज handle करतात.

[0065] (Heading 2) 2.2 Multi-Tenancy Model
[0066] () PreOne हा shared-schema multi-tenant model वापरतो — सर्व tenants एकच database + schema share करतात, प्रत्येक row मध्ये school_id column tenant discriminator म्हणून काम करतो. हा model chosen कारण tenant count 10,000+ असेल आणि per-tenant database maintain करणे operationally expensive होईल.
[0067] () Alternative options considered: schema-per-tenant (rejected — 10k+ schemas degrade catalog performance), database-per-tenant (rejected — operationally unmanageable at scale). Shared schema with RLS हा sweet spot आहे: query performance good, backup/restore simple, migration uniform.
[0068] () Row-Level Security (RLS) policies enforce tenant isolation at database level. Application connections set school_id via session variable (SET app.school_id = ?); RLS policies automatically filter rows. Cross-tenant access requires platform_admin role + explicit bypass.
[0069] (Heading 2) 2.3 Connection Pooling
[0070] () PgBouncer transaction-mode pooling वापरतो — 10,000+ concurrent application connections ला ~100 database connections शी multiplex करतो. Pool size per-tenant cap configured केला आहे (max 10 concurrent transactions per tenant) ज्यामुळे noisy-neighbor problem prevent होतो.
[0071] (Heading 2) 2.4 Backup + DR
[0072] () Backup strategy: PITR via WAL archiving (5-minute RPO), daily logical backup (pg_dump) per tenant for granular restore, weekly full cluster snapshot, monthly restore drill. RTO 30 minutes for single-tenant restore, 2 hours for full cluster restore. Cross-region DR replica maintains async replication; failover run-book documented separately.

[0101] (Heading 1) 4. Data Type Standards
[0102] () Data type choice हे storage efficiency, query performance, आणि correctness लाखो rows मध्ये affect करतो. PreOne मध्ये खालील type standards enforce केले जातात — दर type चे specific use case + size characteristics document केले आहेत.
[0103] (Heading 2) 4.1 Type Reference
[0104] (Heading 2) 4.2 Money Handling
[0105] () Money columns नेहमी integer cents मध्ये store होतात (e.g., amount_cents) किंवा numeric(12,2) मध्ये. Float types (real, double precision) हे money साठी FORBIDDEN आहेत कारण rounding errors compound होतात. Integer cents preferred कारण integer arithmetic float च्या comparison अधिक fast आणि exact आहे. Display time ला divide by 100 करून decimal show करावे.
[0106] (Heading 2) 4.3 Timestamps
[0107] () सर्व timestamp columns ने timestamptz (timestamp with timezone) वापरावे. PostgreSQL internally UTC मध्ये store करतो पण display timezone session setting नुसार adjust होतो. Multi-region deployment मध्ये timestamp without tz columns ambiguous होतात आणि subtle bugs introduce करतात. Created_at आणि updated_at साठी DEFAULT now() वापरा; updated_at साठी trigger-based auto-update.
[0108] (Heading 2) 4.4 JSON vs JSONB
[0109] () jsonb हा default choice आहे — binary storage, GIN-indexable, faster query. json (text-storage) हे केवळ order-preserving need असल्यास वापरा (जे दुर्मिळ आहे). PreOne मध्ये config columns, flexible metadata, आणि arrays सर्व jsonb मध्ये store होतात.
[0110] (Heading 2) 4.5 Enums vs Lookup Tables
[0111] () Closed status sets (≤10 values, जे कधीच change होणार नाहीत) साठी native PG enum वापरा (AttendanceStatus, InvoiceStatus). Open or evolving sets साठी lookup table वापरा (inventory_categories, fee_heads). Enum values delete होऊ शकत नाहीत — त्यामुळे जर कधी value remove करायचा असेल तर lookup table better आहे.

[0125] () PgBouncer pooler ला tenant-aware config केले आहे. [0130] (Heading 1) 6. Indexing Strategy
[0131] () Indexing हे database performance चा single most important factor आहे. PreOne मध्ये indexing strategy documented, deliberate, आणि measured आहे. प्रत्येक index चा specific query pattern serve करतो — नको ते indexes write performance degrade करतात आणि storage waste होतो.
[0132] (Heading 2) 6.1 Index Type Reference
[0133] (Heading 2) 6.2 Common Index Patterns in PreOne
[0134] () •  Foreign key columns always indexed — PostgreSQL does NOT auto-index FKs.
[0135] () •  Partial index WHERE deleted_at IS NULL on soft-delete tables — active rows only.
[0136] () •  Partial index WHERE is_active = true on status-flag tables — active records only.
[0137] () •  Composite index (school_id, branch_id, ...) — tenant + scope pruning.
[0138] () •  BRIN on time-series tables (audit_log, sms_log, trip_log) — 90% smaller than btree.
[0139] () •  GIN on jsonb config columns — @> containment queries.
[0140] () •  Unique indexes on natural keys (student_code, invoice_number, phone) — enforce business uniqueness.
[0141] (Heading 2) 6.3 Index Maintenance
[0142] () Index bloat monitoring weekly run होतो — pages_reclaimed आणि fragmentation ratio track होतो. REINDEX CONCURRENTLY scheduled monthly for high-write tables (audit_log, sms_log, attendances). Auto-vacuum tuned per-table — high-write tables get aggressive autovacuum settings; low-write tables default.

[0145] (Heading 2) 7.1 Constraint Types Used
[0146] (Heading 2) 7.2 FK ON DELETE Behaviors
[0147] () FK ON DELETE behavior हा data integrity चा critical decision आहे. PreOne मध्ये खालील rules enforce होतात:
[0148] () •  RESTRICT — parent table deletion blocked (default for school, branch, student, user).
[0149] () •  CASCADE — child rows auto-deleted (for junction tables, e.g., role_permission, user_role).
[0150] () •  SET NULL — child FK set to NULL (for soft references, e.g., approved_by, created_by).
[0151] () •  Hard deletes दुर्मिळ — soft delete (deleted_at) preferred. RESTRICT केवळ schema-migration safety साठी.
[0152] (Heading 2) 7.3 CHECK Constraints for Enum Validation
[0153] () Native PG enums type-safe आहेत पण migration burden जास्त (value add करण्यास migrate करावे लागते). PreOne मध्ये hybrid approach: critical enums (AttendanceStatus, InvoiceStatus) native enum; evolving enums varchar + CHECK constraint. CHECK constraint faster migrate करता येतो.
[0154] (Heading 2) 7.4 Exclusion Constraints
[0155] () Time-overlapping entities (room bookings, staff shifts, classroom assignments) साठी exclusion constraints enforce केले आहेत. 例: room_booking table मध्ये EXCLUDE USING gist (room_id WITH =, tsrange(start_at, end_at) WITH &&) — एकाच room मध्ये overlapping booking prevent होते.
[0156] (Heading 1) 8. Audit, Soft-Delete & Versioning
[0157] () Data integrity + forensic audit + DPDP compliance यांसाठी audit + soft-delete + versioning patterns standardize केले आहेत. या section मध्ये implementation patterns document आहेत.
[0158] (Heading 2) 8.1 Soft Delete Pattern
[0159] () Hard deletes दुर्मिळ आहेत — प्रत्येक business table मध्ये deleted_at timestamptz NULL column आहे. Row 'delete' म्हणजे UPDATE ... SET deleted_at = now(). Restore म्हणजे UPDATE ... SET deleted_at = NULL. Permanent delete हा admin tool द्वारे manual — दर permanent delete audit_log मध्ये record होतो.
[0160] () -- Soft delete trigger (simplified)
[0161] () CREATE TRIGGER soft_delete_before_delete
[0162] () BEFORE DELETE ON students
[0163] () FOR EACH ROW EXECUTE FUNCTION soft_delete_handler();
[0165] () CREATE FUNCTION soft_delete_handler() RETURNS TRIGGER AS $$
[0166] () BEGIN
[0167] () UPDATE students SET deleted_at = now() WHERE id = OLD.id;
[0168] () RETURN NULL;  -- suppress actual DELETE
[0169] () END;
[0170] () $$ LANGUAGE plpgsql;
[0171] (Heading 2) 8.2 Audit Log Pattern
[0172] () audit_log table हा immutable forensics record आहे — कधीच UPDATE/DELETE होत नाही. प्रत्येक CREATE/UPDATE/DELETE म्हणजे INSERT into audit_log. Trigger-based logging हा default; application-layer logging केवळ cross-service operations साठी. audit_log monthly partitioned (1M+ rows/day at scale) — BRIN index on occurred_at.
[0173] () Audit log columns: user_id (actor), action (CREATE/UPDATE/DELETE/LOGIN/...), module, entity, entity_id, old_values (jsonb), new_values (jsonb), ip_address, user_agent, request_id (distributed tracing), occurred_at.
[0174] (Heading 2) 8.3 Optimistic Concurrency
[0175] () Critical entities (Student, Invoice, Payment, Application) मध्ये version integer column आहे जो प्रत्येक UPDATE ला increment होतो. Application UPDATE ... SET version = version + 1 WHERE id = ? AND version = ? execute करते; जर 0 rows affected तर concurrent modification detect होते आणि user ला retry prompt केले जाते. हे pattern lost-update anomalies prevent करते.
[0176] (Heading 2) 8.4 History Tables
[0177] () Specific high-value entities (Student, Invoice, FeePlan, Application) साठी dedicated history tables आहेत — student_history, invoice_history, etc. ह्या tables append-only आहेत आणि प्रत्येक significant lifecycle event snapshot store करतात (admission, promotion, graduation, transfer for students; issue, payment, refund, write-off for invoices). History tables analytical queries साठी optimize केले आहेत.

[3671] (Heading 2) 27.1 Naming Conventions
[3672] () •  Table name (snake_case) → Prisma model (PascalCase): students → Student, audit_log → AuditLog.
[3673] () •  Column name (snake_case) → Prisma field (camelCase): school_id → schoolId, created_at → createdAt.
[3674] () •  @@map annotation on model to specify table name.
[3675] () •  @map annotation on field to specify column name.
[3676] () •  @@unique for composite unique constraints.
[3677] () •  @@index for composite indexes.
[3678] () •  Enum types defined at top of schema file; values UPPER_SNAKE_CASE.

[3720] (Heading 1) 28. Migration Strategy (v2 → v3)
[3721] () v2 → v3 migration हा large-scale schema evolution आहे — 142 tables पासून 307 tables पर्यंत, plus multi-tenancy layer hardening, partitioning introduction, आणि DPDP compliance fields. या chapter मध्ये phased migration plan document आहे.
[3722] (Heading 2) 28.1 Migration Waves
[3723] () Migration 12 waves मध्ये execute होतो — दर wave 1-2 weeks मध्ये complete होतो. दर wave ची deploy + verify + rollback plan documented आहे.
[3724] (Heading 2) 28.2 Rollback Plan
[3725] () दर wave साठी rollback plan documented आहे — schema snapshot before migration, dual-write window where possible, feature flag to revert to v2 logic. Total rollback window 30 days from W11 cutover. Post-W12, rollback requires data loss (v3-only columns).
[3726] (Heading 2) 28.3 Verification
[3727] () दर wave post-deploy verification: (a) row count parity, (b) query performance regression test, (c) application smoke test suite, (d) audit log completeness check, (e) RLS isolation test (cross-tenant query must return 0 rows).

[3732] () •  Single-row restore: PITR WAL replay to staging → extract row → INSERT into production (manual review).
[3733] () •  Single-tenant restore: pg_dump from daily backup → restore to staging → extract tenant rows → INSERT/UPDATE into production.
