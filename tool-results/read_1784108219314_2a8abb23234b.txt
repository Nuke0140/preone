    65→[00069] [Normal] Access token expiry झाल्यावर client refresh token वापरून new access token fetch करतो. Refresh token rotation — प्रत्येक refresh मध्ये old refresh token blacklist, new refresh token issue. ह्यामुळे stolen refresh token quickly useless होतो.

    68→[00072] [Heading 2] 4.5 Token Lifecycle
    69→[00073] [Normal] Token lifecycle खालील steps मध्ये managed होतो:
    70→[00074] [Normal] •  User logs in via OTP/password → server issues accessToken (15min) + refreshToken (30day)
    71→[00075] [Normal] •  Client uses accessToken for all API calls in Authorization: Bearer <token> header
    72→[00076] [Normal] •  When accessToken expires, client calls POST /auth/refresh with refreshToken → new accessToken
    73→[00077] [Normal] •  Refresh token is rotated on each use (one-time use) — old refresh token blacklisted
    74→[00078] [Normal] •  User logs out → POST /auth/logout → both tokens blacklisted server-side
    75→[00079] [Normal] •  After 30 days of inactivity → refreshToken expires → full re-authentication required
    76→[00080] [Normal] •  Compromised token? Admin can force-invalidate all sessions for a user via POST /users/{id}/sessions/invalidate

   154→[00164] [Heading 2] 10.2 Pagination Rules
   155→[00165] [Normal] Page-based pagination साठी rules:
   156→[00166] [Normal] •  Default pageSize: 20. Max pageSize: 100. Larger sizes rejected with VALIDATION_001.
   157→[00167] [Normal] •  Page numbering is 1-indexed (page=1 is first page). page=0 returns 400.
   158→[00168] [Normal] •  For datasets >10k rows, prefer cursor-based pagination (?cursor=abc&limit=20) for performance.
   159→[00169] [Normal] •  OFFSET pagination becomes O(n) slow at deep pages — cursor is O(1).
   160→[00170] [Normal] •  Response meta always includes: page, pageSize, total, totalPages, hasMore, nextCursor.
   161→[00171] [Normal] •  Empty result set returns 200 with data:[] and meta.total:0 (NOT 404).
   162→[00172] [Normal] •  Sorting + pagination: sort must be deterministic (include id as tiebreaker).
   163→[00173] [Heading 2] 10.3 Cursor-Based Pagination (Large Datasets)
   164→[00174] [Normal] Datasets >10k rows साठी cursor-based pagination recommended आहे. OFFSET pagination O(n) slow होते deep pages वर — cursor O(1) राहतो. Cursor opaque string आहे (client ला decode करायची गरज नाही), server-side last-seen-ID encode करतो. Request: ?cursor=abc&limit=20. Response: meta.nextCursor field.

   191→[00204] [Heading 1] 13. Versioning
   192→[00205] [Heading 2] 13.1 Versioning Policy
   193→[00206] [Normal] URI path versioning वापरले जाते — /api/v1/, /api/v2/. हा approach clients साठी सर्वात intuitive आहे (URL ने version visible), routing सोपा आहे (load balancer prefix route), आणि debugging easier आहे (logs मध्ऍे version visible).
   194→[00207] [Heading 2] 13.2 Versioning Rules
   195→[00208] [Normal] Versioning strict rules follow करतात — long-term client compatibility साठी:
   196→[00209] [Normal] •  URI path versioning: /api/v1/, /api/v2/. Most intuitive for clients, easiest to route.
   197→[00210] [Normal] •  Within a major version, NO breaking changes — only additive (new fields, new endpoints, new optional params).
   198→[00211] [Normal] •  Breaking changes (rename field, remove field, change type, change semantics) require new major version.
   199→[00212] [Normal] •  New major version has 12-month overlap with prior version before sunset.
   200→[00213] [Normal] •  Deprecation headers sent on deprecated endpoints: Sunset: Wed, 30 Jun 2027 00:00:00 GMT; Deprecation: true; Link: </api/v2/students>; rel="successor-version".
   201→[00214] [Normal] •  Version support policy: current major + 1 prior major. v3 release → v1 sunset.
   202→[00215] [Normal] •  Migration guide published with each major release: field mapping, behavior changes, SDK updates.
   203→[00216] [Normal] •  Versioning is per-API-surface, not per-endpoint — all endpoints under /api/v1/ move together.
   204→[00217] [Heading 2] 13.3 Breaking vs Non-Breaking Changes
   205→[00218] [Normal] Breaking changes (new major version required):
   206→[00219] [Normal] •  Remove a field from response
   207→[00220] [Normal] •  Rename a field (e.g., studentName → firstName)
   208→[00221] [Normal] •  Change field type (e.g., string → number)
   209→[00222] [Normal] •  Change field semantic (e.g., amount in rupees → paise)
   210→[00223] [Normal] •  Change required-ness of a request field (optional → required)
   211→[00224] [Normal] •  Change error code (e.g., AUTH_001 → AUTH_BAD)
   212→[00225] [Normal] •  Change default behavior (e.g., default pageSize 20 → 50)
   213→[00226] [Normal] Non-breaking changes (within same major version OK):
   214→[00227] [Normal] •  Add new field to response
   215→[00228] [Normal] •  Add new optional field to request
   216→[00229] [Normal] •  Add new endpoint
   217→[00230] [Normal] •  Add new enum value (clients should handle unknown enums gracefully)
   218→[00231] [Normal] •  Add new error code (clients should fallback to message)
   219→[00233] [Heading 1] 14. Idempotency
   220→[00234] [Heading 2] 14.1 Idempotency-Key Header
   221→[00235] [Normal] Idempotency-Key header (UUID v4) POST/PUT/PATCH methods वर supported आहे. हा header server ला सांगतो की हा request retry आहे — same key + same payload आला तर cached response return करा. Payment, refund, admission submission, file upload यांसारख्या critical endpoints साठी MANDATORY आहे.
   222→[00236] [Heading 2] 14.2 Idempotency Rules
   223→[00237] [Normal] •  Idempotency-Key header (UUID v4) supported on POST, PUT, PATCH methods.
   224→[00238] [Normal] •  MANDATORY for: payments, refunds, admissions submission, file uploads, invoice generation.
   225→[00239] [Normal] •  Server stores key + response hash for 24 hours in Redis.
   226→[00240] [Normal] •  Same Idempotency-Key + same payload → returns cached response (HTTP 200/201 + identical body).
   227→[00241] [Normal] •  Same Idempotency-Key + DIFFERENT payload → returns 409 CONFLICT_001 (key reuse with different body).
   228→[00242] [Normal] •  Scope: per-user, per-endpoint. Different users can use same key without conflict.
   229→[00243] [Normal] •  Client MUST regenerate key for genuinely new requests, not retry same key for different operations.
   230→[00244] [Normal] •  Network failure / timeout: client retries with SAME key — server deduplicates automatically.
   231→[00245] [Normal] •  After 24h TTL, key expires — same key can be reused for new operation (uncommon, but safe).
   232→[00246] [Heading 2] 14.3 Example: Payment with Idempotency
   233→[00247] [Normal] Network failure झाला तर client समान Idempotency-Key ने retry करतो — server automatically deduplicates. ह्यामुळे duplicate payment होण्याचा risk zero.
