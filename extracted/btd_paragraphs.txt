[0203] () Modules एकमेकांशी direct imports करू शकत नाहीत — हे strict rule आहे. Cross-module communication खालील patterns through होते:
[0204] () •  Integration Events (Async): Redis Stream पर event publish करून consumer module subscribe करतो. Most common pattern — eventual consistency acceptable.

[0208] () •  Read Model Projection: Module दुसऱ्या module चा read-only data ची copy ठेवते via event subscription (CQRS pattern). Avoids cross-module queries.

[0464] (Heading 1) 12. CQRS
[0465] () CQRS (Command Query Responsibility Segregation) हा pattern reads आणि writes ला separate करतो. PreOne मध्ये CQRS फक्त complex workflows मध्ये वापरला जाईल, प्रत्येक CRUD endpoint मध्ये नाही. हा ADR-012 चा decision आहे — simple CRUD operations साठी CQRS overhead justify होत नाही, पण multi-aggregate workflows (admissions → enrollment → invoice) साठी command/query separation clarity देते.
[0466] (Heading 2) 12.1 When to Use CQRS
[0467] () CQRS use करण्याचे clear criteria आहेत. हे criteria ADR-012 मध्ये documented आहेत आणि architect च्या approval नंतर वापरावे लागतात:
[0468] () •  Multi-aggregate workflows (3+ aggregates coordinated) — Admission approval involves Application + Student + Invoice.
[0469] () •  Read/write asymmetry — heavy reads with complex filters, simpler writes (e.g., student search).
[0470] () •  Independent scaling of reads — read replicas, denormalised projections (e.g., dashboard KPIs).
[0471] () •  Long-running business processes with state machine — application lifecycle, fee collection cycle.
[0472] () •  Event sourcing requirements — financial ledger, audit trail.
[0473] () Simple CRUD (single aggregate, no workflow) साठी plain application service पुरेसा आहे — CreateStudentHandler सारखा command handler unnecessary complexity जोडतो.

[0520] () •  Event names: past-tense verb (StudentCreated, not CreateStudent).
[0521] () •  Events are immutable — once published, never modified.

[0528] (Heading 1) 14. Integration Events
[0529] () Integration Events हे cross-bounded-context communication चा mechanism आहे. [0530] (Heading 2) 14.1 Integration Event Catalog
[0531] (Heading 2) 14.2 Integration Patterns
[0532] () Integration events च्या two delivery patterns आहेत — sync (HTTP) आणि async (Redis Stream). Pattern choice ADR-015 मध्ये per-integration basis decide केला जातो. मुख्य criteria: काय consumer चा immediate response पाहिजे आहे?
[0533] () •  Sync (HTTP): Producer HTTP call करतो consumer च्या internal endpoint वर, response waits. Used when producer ला consumer चा result पाहिजे (e.g., Admissions → Identity for student ID).
[0534] () •  Async (Redis Stream): Producer event Redis Stream वर publish करतो, consumer खाली subscribe करतो. Used when eventual consistency acceptable (e.g., PaymentReceived → Communication).
[0535] () •  Saga Pattern: Long-running multi-step transactions ज्यात प्रत्येक step own transaction आहे. Compensating actions rollback साठी. Used for admission → enrollment → invoice flow.
[0536] () •  Outbox Pattern: Producer event DB outbox table मध्ये same transaction मध्ये writes करतो. Background publisher outbox ला drain करून Redis Stream वर push करतो. Reliability सुनिश्चित करतो.
[0537] (Heading 2) 14.3 Integration Rules
[0538] () •  Integration events are versioned (.v1, .v2) — backward-compatible schema evolution
[0539] () •  Sync integration = HTTP call within same request — must have timeout + circuit breaker
[0540] () •  Async integration = Redis Stream — at-least-once delivery; consumer must be idempotent
[0541] () •  Producer publishes to outbox table; publisher worker drains to Redis Stream
[0542] () •  Consumer maintains last-read position in Redis (consumer group offset)
[0543] () •  Retry policy: 3 retries with exponential backoff (1s, 5s, 25s); then DLQ
[0544] () •  Poison messages (failed 3x) routed to DLQ for manual inspection
[0545] () •  Schema changes require new version — never modify existing event schema in-place
[0546] () •  Event payload includes schema URL for runtime validation by consumer
[0547] () •  All events include: eventId (UUID), occurredAt (ISO), tenantId, branchId, userId

[0605] (Heading 1) 17. Transactions
[0606] () Transactions हे data consistency चा foundation आहेत. PreOne चा core principle: 'Aggregate = Transaction Boundary'. एका aggregate चा save हा एक transaction आहे — multiple aggregates एकाच transaction मध्ये save होणे anti-pattern आहे. जर multi-aggregate operation हवा असेल तर Saga pattern वापरा — प्रत्येक step own transaction, rollback via compensating action.
[0607] (Heading 2) 17.1 Transaction Patterns
[0608] () वेगवेगळ्या scenarios साठी वेगवेगळे transaction patterns वापरले जातात. खालील table सर्व patterns ची catalog करते:
[0609] (Heading 2) 17.2 Transaction Rules
[0610] () •  Aggregate = Transaction Boundary — never split one aggregate across transactions
[0611] () •  No cross-aggregate transactions — use saga or domain events instead
[0612] () •  Always set transaction timeout (default 5s; longer for batch)
[0613] () •  Never call external services (HTTP, S3, email) within transaction — do after commit
[0614] () •  Rollback on any exception — let it propagate to global filter
[0615] () •  Transaction isolation level: READ COMMITTED (default); SERIALIZABLE only for critical billing
[0616] () •  Max transaction duration: 5s; longer = suspect deadlock — alert
[0617] () •  Test transactions under load to detect deadlocks (Testcontainers + concurrent Vitest)
[0618] () •  Log transaction begin/commit/rollback with traceId for observability

[0647] (Heading 1) 18. Validation
[0648] () Validation हे defense-in-depth strategy आहे. PreOne मध्ये 6 validation layers आहेत — प्रत्येक layer वेगवेगळ्या type ची invalid input catch करते. यामुळे invalid data कधीच database पर्यंत पोहोचत नाही. Layer-by-layer validation fail-fast principle follow करते — जितक्या लवकर error catch होईल तितका clear error message.
[0649] (Heading 2) 18.1 Validation Layers
[0650] (Heading 2) 18.2 Validation Flow
[0651] () Request चा flow खालील stages through जातो. प्रत्येक stage वेगवेगळ्या type ची validation perform करते. जर early stage मध्ये fail झाले तर later stages skip होतात — यामुळे wasted work कमी होते:
[0652] () 1. Request arrives at controller
[0653] () 2. Throttler guard checks rate limit (429 if exceeded)
[0654] () 3. JwtAuthGuard authenticates (401 if invalid token)
[0655] () 4. PermissionsGuard checks authorization (403 if denied)
[0656] () 5. ValidationPipe runs DTO class-validator (422 if invalid)
[0657] () 6. Zod validation pipe runs schema (422 if invalid)
[0658] () 7. Idempotency check (409 if duplicate)
[0659] () 8. Handler invoked
[0660] () 9. Domain validation throws DomainException (400)
[0661] () 10. Business rule check throws BusinessException (409)
[0662] () 11. Repository save - DB constraint violation throws ConflictException (409)
[0663] () 12. Success - ResponseDto returned (200/201)
[0664] () Each layer has a specific responsibility and a specific error code on failure. This separation makes debugging easier — the error code tells you which layer failed, and the field-level details array pinpoints the exact field.
[0665] (Heading 2) 18.3 Validation Best Practices
[0666] () •  Validate at the boundary — never trust external input.
[0667] () •  Fail fast — cheapest check first (rate limit before DB query).
[0668] () •  Return specific error codes — not generic 'INVALID_INPUT'.
[0669] () •  Field-level errors for ValidationException — array of { field, code, message }.
[0670] () •  Never swallow exceptions — let them propagate to global filter.
[0671] () •  Log validation failures at warn level — useful for UX improvement.
[0672] () •  Domain layer validates via value object constructors — fail on construction.
[0673] () •  Business rules in specifications — composable, testable.

[0914] () •  Use Result<T, E> type for fallible operations (no thrown exceptions in domain layer)

[0916] () •  JSDoc on every public API (controller, service, repository method)
[0917] () •  No console.log — use Pino logger
[0918] () •  No Date — use ISO-8601 string or custom DateTime VO
[0919] () •  No Math.random for IDs — use UUID v7 (time-ordered)
[0920] () •  No setTimeout for business logic — use BullMQ delayed jobs
[0921] () •  No direct env access — use ConfigService
[0922] () •  No new Date().getFullYear() — use injected Clock service for testability
