[Heading 1] 7. User Roles & Permissions
[Normal] PreOne चा Role-Based Access Control (RBAC) model 8 primary roles define करतो आणि प्रत्येक role ची per-module permissions specify करतो. हे model ADR-007 (Identity & Authorization) मध्ये freeze केले गेले आहे आणि ते multi-tenant architecture चा core security mechanism आहे.
[Normal] RBAC चा primary principle हा आहे: "least privilege" — प्रत्येक user ला फक्त त्याच्या role ला आवश्यक असलेले permissions मिळतात. Permissions चे चार levels आहेत: Read (view only), Create (add new), Update (edit existing), Delete (remove). काही actions (e.g., refund approval, fee plan edit) additional approval workflows द्वारे governed आहेत जे role permission पलीकडे आहेत.

[Heading 2] 7.3 Permission Principles
[Normal] RBAC implementation खालील principles follow करतो. हे principles design review चा framework आहेत — जेव्हा permission ambiguity येतो तेव्हा हे principles tie-breaker म्हणून वापरले जातात:
[Normal] •  Least Privilege: Default deny; explicit allow. User ला फक्त त्याच्या role ला आवश्यक असलेले minimum permissions.
[Normal] •  Need-to-Know: Data access सीमित — teacher ला फक्त त्याच्या class चा data, parent ला फक्त त्याच्या child चा data.
[Normal] •  Separation of Duties: Critical actions ला दोन वेगळे roles — e.g., fee refund request ≠ refund approval (different persons).
[Normal] •  Approval Workflow: Role permission पलीकडे high-value actions (refund, salary edit, fee plan change) ला additional approval chain.
[Normal] •  Audit Every Action: Permission check result (allow/deny) चा audit log; denial चा reason captured.
[Normal] •  Context-Aware: Time, location, device, IP — contextual factors role permission ला override करू शकतात (e.g., after-hours access alert).
[Normal] •  Temporary Elevation: Emergency elevation mechanism with auto-expiry + audit + manager notification.
[Normal] •  Delegation: Approver out-of-office असल्यास delegation allowed with original approver notified + audit.

[Heading 1] 13. Traceability Matrix (RTM)
[Normal] Requirement Traceability Matrix (RTM) हा PreOne च्या requirements engineering चा final quality gate आहे. यात प्रत्येश requirement चा complete trace chain documented आहे — Vision → BR → FR → BRC → AC → Test Case → Implementation. हे traceability ensure करते की कोणताही requirement orphaned नाही आणि कोणताही implementation un-traced नाही.
[Normal] RTM चा primary use case: (1) Impact analysis — BR change केला तर कोणते FRs, BRC rules, ACs affected होतील; (2) Coverage analysis — कोणत्या BRs ची ACs पूर्ण झाली आहेत; (3) Audit compliance — ISO 27001 / SOC 2 audits मध्ये traceability mandatory; (4) Onboarding — new team members ला context मिळतो. खालील master table 30 BRs चा trace chain दर्शवतो.
[Heading 2] 13.1 Requirements Flow
[Normal] PreOne च्या requirements hierarchy खालीलप्रमाणे आहे. ही hierarchy top-down आहे — वरचा layer खालच्या layer ला drive करतो, आणि खालचा layer वरच्या layer चा verification provide करतो:
[Normal] Vision  →  Business Requirement (BR)  →  Functional Requirement (FR)  →  Business Rule (BRC)  →  Acceptance Criteria (AC)  →  Test Case (TC)  →  Implementation

[Heading 2] 13.4 Traceability Process
[Normal] RTM चा maintenance खालील process द्वारे होतो. हा process software development lifecycle चा embedded step आहे — separate RTM update activity नाही:
[Normal] •  Requirement Creation: नवीन BR/FR/AC created करताना traceability links mandatory. बिना trace chain कोणताही requirement merged होणार नाही.
[Normal] •  Sprint Planning: Each sprint RTM review करतो — orphan requirements identified, missing ACs flagged.
[Normal] •  Code Review: PR template मध्ये "Related AC" field mandatory — code without AC reference rejected.
[Normal] •  QA Cycle: Test cases traceable to ACs; AC without test case flagged; coverage report auto-generated.
[Normal] •  Release Sign-off: Release readiness check: 100% P0 BRs covered, ≥95% P1 covered, 0 orphan ACs.
[Normal] •  Quarterly Audit: RTM completeness audit by Product team; gaps remediated within sprint.

[Normal] •  Annual Major Release: Once a year, major version bump (v2.0). Triggers re-baseline of BRC + companion docs.
[Normal] •  Emergency Change: Critical compliance / security changes can be pushed as hotfix with CPO + Compliance Officer approval.
