## 2. Problems Found & Fixes Applied

### 2.1 YAML Syntax (Tasks 1, 2)

#### Problem P-1 — Malformed flow mapping with `${{ }}` expression [Critical]

**Location:** Lines 29, 48, 94 (original file)
**Original:**
```yaml
- uses: pnpm/action-setup@v4
  with: { version: ${{ env.PNPM_VERSION }} }
```

**Issue:** In YAML flow-mapping context (`{ ... }`), the sequence `${{ ... }}` is technically invalid per the YAML 1.2 spec. The first `{` after `$` is interpreted as the start of a nested flow mapping, then the parser fails to find a matching `,` or `}`. GitHub Actions' Go YAML parser tolerates this in practice, but stricter tools (`yamllint`, Python `pyyaml`, `actionlint` strict mode) reject it. This makes the file un-portable and prevents local linting.

**Fix:** Convert flow mappings to standard block mappings:
```yaml
- uses: pnpm/action-setup@v4
  with:
    version: ${{ env.PNPM_VERSION }}
```

**Files affected:** Lines 29, 48, 94 → expanded to block form.

### 2.2 Branch Filters (Task 3)

#### Problem P-2 — Push branch filter does not match actual branch naming convention [Critical]

**Location:** Line 7 (original)
**Original:**
```yaml
push:
  branches: [main, develop, 'feature/**']
```

**Issue:** The repo's actual branch convention is `feat/wave-XX-<name>` (see `git branch -a`: `feat/wave-4-admissions-attendance`, `feat/wave-16-websocket-gateways`, etc.). The glob `feature/**` matches branches starting with `feature/`, **not** `feat/`. As a result, pushes to **any** feature branch in the repo's history have **never** triggered CI on push — only PR-targeted events fire. Dependabot branches (`dependabot/npm_and_yarn/...`) are also not covered.

**Fix:** Add `feat/**` and `dependabot/**` globs while keeping `feature/**` for backward compatibility:
```yaml
push:
  branches: [main, develop, 'feat/**', 'feature/**', 'dependabot/**']
```

### 2.4 Artifact Upload (Task 8)

#### Problem P-4 — No artifact upload for coverage or security scan results [Important]

**Location:** `unit-tests` and `build-scan` jobs (original)

**Issue:** Coverage reports were generated by `vitest --coverage` but only sent to Codecov — never saved as GitHub artifacts. If Codecov is down or token missing, coverage data is lost. Similarly, Trivy SARIF was uploaded to GitHub Security but not saved as an artifact for offline review.

**Fix:** Added `actions/upload-artifact@v4` steps:
- After `test:unit`: upload `apps/api/coverage/unit/` as `coverage-unit-${{ github.sha }}` (retention 14d)
- After Trivy scan: upload `trivy-results.sarif` as `trivy-results-${{ github.sha }}` (retention 30d)

Both use `if: always()` so artifacts are preserved even on test/scan failures.

#### Problem P-5 — Codecov action missing `fail_ci_if_error` and `token` [Important]

**Location:** `unit-tests` job (original)
**Original:**
```yaml
- uses: codecov/codecov-action@v4
  with:
    files: ./apps/api/coverage/unit/lcov.info
    flags: unit
```

**Issue:**
1. Codecov v4 **requires** `token: ${{ secrets.CODECOV_TOKEN }}` for private repos (the PreOne repo is private). Uploads silently fail without it.
2. Without `fail_ci_if_error: false`, any Codecov outage or token misconfiguration breaks the entire CI run — even though coverage is informational, not gating.

**Fix:**
```yaml
- uses: codecov/codecov-action@v4
  with:
    files: ./apps/api/coverage/unit/lcov.info
    flags: unit
    fail_ci_if_error: false
    token: ${{ secrets.CODECOV_TOKEN }}
```

### 2.5 Security Scan Jobs (Task 10)

#### Problem P-6 — Snyk scan blocks pipeline if `SNYK_TOKEN` is unset [Critical]

**Location:** `build-scan` job (original)
**Original:**
```yaml
- name: Snyk security scan
  uses: snyk/actions/node@master
  env:
    SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
  with:
    args: --severity-threshold=high
```

**Issue:** Snyk requires a valid token. If `SNYK_TOKEN` is not configured in repo secrets (new forks, emergency CI runs, etc.), the step fails, the job fails, and `deploy-staging` is blocked. Security scanning should be **advisory**, not a deploy blocker for the staging environment — especially with Trivy as a backup scan.

**Fix:** Add `continue-on-error: true` to the Snyk step. Trivy (which runs unconditionally) still provides SARIF-based security reporting. Note: this preserves the existing pipeline shape — Snyk is not removed, just made non-blocking.

#### Problem P-7 — Trivy action pinned to floating `@master` [Important]

**Location:** `build-scan` job (original)
**Original:** `uses: aquasecurity/trivy-action@master`

**Issue:** Floating tags like `@master` can introduce breaking changes without warning. Supply-chain attacks against the upstream repo would also propagate immediately. GitHub's security hardening guide explicitly recommends pinning to a version tag or commit SHA.

**Fix:** Pin to `aquasecurity/trivy-action@0.24.0` (stable release). Did not pin to a commit SHA — that would be more secure but harder to maintain; version tag is the conventional compromise.

> Note: `snyk/actions/node@master` was left unchanged — Snyk's actions repo does not publish version tags, only `master`. Pinning to a SHA is a deferred recommendation (see §3).

#### Problem P-8 — Trivy SARIF upload missing `if: always()` [Important]

**Location:** `build-scan` job (original)
**Original:**
```yaml
- uses: github/codeql-action/upload-sarif@v3
  with: { sarif_file: trivy-results.sarif }
```

**Issue:** If the Trivy step itself fails (image not built, network error), the SARIF file is not produced, and the upload step also fails. The `with: { ... }` flow mapping has the same YAML portability issue as P-1.

**Fix:** Expanded to block mapping with `if: always()`:
```yaml
- name: Upload Trivy SARIF to GitHub Security
  uses: github/codeql-action/upload-sarif@v3
  if: always()
  with:
    sarif_file: trivy-results.sarif
```

### 2.6 Deployment Jobs (Task 11)

#### Problem P-9 — `deploy-production` references `:prod` image tag that is never pushed [Critical]

**Location:** Lines 162-166 (`deploy-staging`) and Line 191 (`deploy-production`) — original
**Original staging:**
```bash
docker tag preone-api:${{ github.sha }} ${{ secrets.ECR_REGISTRY }}/preone-api:staging
docker push ${{ secrets.ECR_REGISTRY }}/preone-api:staging
```
**Original production:**
```bash
kubectl set image deployment/preone-api-green preone-api=${{ secrets.ECR_REGISTRY }}/preone-api:prod -n api
```

**Issue:** Staging only pushes `:staging`. Production references `:prod`, which **does not exist in ECR**. The `kubectl set image` command would succeed (kubectl doesn't validate remote image existence), but the subsequent `kubectl rollout status` would hang and eventually fail with `ImagePullBackOff`. This bug would only manifest on the **first production deploy** — a worst-case scenario for silent CI bugs.

**Fix:** Updated `deploy-staging` to also tag and push `:prod`:
```bash
docker tag preone-api:${{ github.sha }} ${{ secrets.ECR_REGISTRY }}/preone-api:staging
docker tag preone-api:${{ github.sha }} ${{ secrets.ECR_REGISTRY }}/preone-api:prod
docker push ${{ secrets.ECR_REGISTRY }}/preone-api:staging
docker push ${{ secrets.ECR_REGISTRY }}/preone-api:prod
```

Both tags point to the same SHA digest — `:prod` is just an alias until the next staging deploy promotes a different SHA. This preserves blue-green semantics: production always deploys the same image that staging just verified.

#### Problem P-10 — `deploy-production` missing AWS credentials setup [Critical]

**Location:** `deploy-production` job (original)
**Original:**
```yaml
steps:
  - name: Await manual approval
    run: echo "✓ Production deployment approved"
  - name: Blue-green deploy to production EKS
    run: |
      aws eks update-kubeconfig --name preone-prod --region ap-south-1
      ...
```

**Issue:** `aws eks update-kubeconfig` requires AWS credentials. The `deploy-staging` job has a `aws-actions/configure-aws-credentials@v4` step, but `deploy-production` does not. The production deploy would fail immediately with `Unable to locate credentials`.

**Fix:** Added the same `aws-actions/configure-aws-credentials@v4` step to `deploy-production` (after the approval echo, before the deploy step).

#### Problem P-11 — `deploy-production` (and `deploy-staging`) missing `kubectl` setup [Critical]

**Location:** Both deploy jobs (original)

**Issue:** `ubuntu-latest` runners do **not** ship with `kubectl` pre-installed. Both deploy jobs call `kubectl` directly, which would fail with `command not found`. The `deploy-staging` job has the same issue.

**Fix:** Added `azure/setup-kubectl@v4` step to both `deploy-staging` and `deploy-production`, before the deploy commands.

#### Problem P-12 — `deploy-staging` and `deploy-production` not protected from concurrent runs [Important]

**Location:** Workflow-level `concurrency` block (original)
**Original:**
```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

**Issue:** This single concurrency group covers **all** jobs, including deploys. If two pushes land on `main` in quick succession, the second push cancels the first — including in-progress deploys. Canceling a `kubectl rollout` mid-flight leaves the cluster in an inconsistent state (partial rollout, mismatched replicas).

**Fix:** Added per-deploy concurrency groups with `cancel-in-progress: false`:
```yaml
deploy-staging:
  concurrency:
    group: deploy-staging
    cancel-in-progress: false

deploy-production:
  concurrency:
    group: deploy-production
    cancel-in-progress: false
```

Deploys now queue instead of canceling each other. The workflow-level group still cancels redundant CI jobs (lint/test/scan) on new pushes.

### 2.8 Docker Build (Task 9)

#### Problem P-14 — Docker build step itself is correct; Dockerfile has a separate latent bug [Informational]

**Location:** `build-scan` job + `infra/docker/api/Dockerfile`

**Workflow step (correct):**
```yaml
- uses: docker/build-push-action@v6
  with:
    context: .
    file: infra/docker/api/Dockerfile
    push: false
    tags: preone-api:${{ github.sha }}
    load: true
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

The step itself is well-formed: `load: true` makes the built image available to subsequent steps (Trivy scan), `cache-from`/`cache-to: type=gha` uses GitHub Actions cache for layer reuse.

**However**, the Dockerfile at line 18 references `packages/shared/package.json`:
```dockerfile
COPY packages/shared/package.json ./packages/shared/
```

The `packages/shared/` directory does not exist in the repo (only `packages/database/` exists). This causes the Docker build to **fail** with `COPY failed: file does not exist`. This is a Dockerfile bug, not a workflow bug — out of scope for this CI review. See §3 (Recommendations).
