    42→[0044] (Heading 1) 2. Infrastructure Overview
    43→[0045] (Heading 2) 2.1 High-Level Architecture
    44→[0046] () PreOne ची infrastructure completely containerized आहे आणि Kubernetes द्वारे orchestrated. ही architecture खालील principles follow करते: infrastructure-as-code (Terraform + Helm), GitOps (GitHub as source of truth), immutable infrastructure (containers + declarative K8s manifests), defense-in-depth (multiple security layers), आणि observability-first (metrics + logs + traces from day one). पूर्ण pipeline — code commit पासून production deployment पर्यंत — automated आहे आणि minimal manual intervention required आहे.
    45→[0047] () Pipeline चा flow समजून घेणे critical आहे. Developer code push करतो → GitHub repository मध्ये change detect होतो → GitHub Actions CI/CD pipeline triggered होतो → code lint, test, build, security scan होतो → Docker image build होतो → GitHub Container Registry (GHCR) मध्ये push होतो → Kubernetes cluster मध्ये deploy होतो → health checks pass होतात → smoke tests run होतात → traffic new deployment वर route होतो. हे सर्व ~20 minutes मध्ये complete होते.

    89→[0092] (Heading 1) 3. Docker Architecture
    90→[0093] (Heading 2) 3.1 Containerization Strategy
    91→[0094] () Docker हे PreOne चे containerization standard आहे. प्रत्येक service (API, Web, Worker, Mobile Build, NGINX) ची स्वतंत्र Docker image असते ज्यामुळे services independently build, version, deploy, आणि scale होऊ शकतात. Docker images multi-stage builds द्वारे optimized आहेत — builder stage मध्ये dev dependencies + build tools वापरून production artifact तयार केला जातो, runtime stage मध्ये फक्त production artifact + minimal runtime copied होतो. हे approach final image size कमी करते आणि attack surface कमी करते.
    92→[0095] () Docker images के सर्व aspects standardized आहेत: base image (node:20-alpine for runtime, node:20 for build), non-root user (uid 1000) for security, HEALTHCHECK instruction embedded, OCI standard labels (org.opencontainers.image.{version,revision,source}), reproducible builds via .dockerignore + lock files, layer caching for faster rebuilds, आणि image signing via cosign (Sigstore) for supply chain security.

   154→[0174] (Heading 1) 4. Kubernetes Architecture
   155→[0175] (Heading 2) 4.1 Cluster Topology
   156→[0176] () PreOne चे Kubernetes cluster multi-AZ (across 3 availability zones) deployed आहे ज्यामुळे single AZ failure साठी resilience मिळते. Cluster मध्ये multiple node groups आहेत: system node group (CoreDNS, CNI, Ingress Controller), application node group (API, Web, Worker pods), data node group (PostgreSQL, Redis — stateful, larger instances), आणि monitoring node group (Prometheus, Loki — storage-heavy). हे separation ensure करते की heavy data workloads application workloads ला impact करणार नाहीत.
   157→[0177] () Cluster version: Kubernetes v1.29 (latest stable). Managed via EKS (AWS Elastic Kubernetes Service). Worker nodes: AWS Graviton (ARM64) for cost optimization; spot instances for non-critical workloads (workers, monitoring); on-demand for critical workloads (API, Web, DB). Cluster autoscaler automatically adds/removes nodes based on pending pods.
   158→[0178] (Heading 2) 4.2 Namespace Architecture
   159→[0179] () Cluster मध्ये namespaces द्वारे logical isolation आहे. प्रत्येक namespace चे specific purpose आहे आणि NetworkPolicies enforce करतात की फक्त allowed namespaces एकमेकांशी communicate करू शकतात. Default deny policy द्वारे defense-in-depth सुनिश्चित होते.

   179→[0200] () PreOne च्या cluster मध्ये विविध workload types वापरले जातात based on workload characteristics:
   180→[0201] () •  Deployment — stateless workloads (API, Web, NGINX). Rolling updates + HPA scaling.

   272→[0294] (Heading 1) 5. CI/CD Pipeline (GitHub Actions)
   273→[0295] (Heading 2) 5.1 Pipeline Overview
   274→[0296] () PreOne ची CI/CD pipeline GitHub Actions द्वारे powered आहे. Pipeline completely automated आहे — developer code push केल्यापासून production deployment पर्यंत कोणताही manual step नाही (except production approval gate). Pipeline stages द्वारे quality gates enforce होतात — एकही stage fail झाल्यास pipeline stop होतो आणि deployment blocked होतो. हे approach production quality सुनिश्चित करतो आणि broken code production मध्ये reach होणार नाही याची ग्यारंटी देतो.
   275→[0297] () Pipeline design principles: fast feedback (lint + unit tests < 5 min), fail-fast (stop on first failure), reproducible (lock files + pinned action versions), observable (status checks visible in PR), recoverable (auto-rollback on health check failure). Pipeline runs on every push (feature branch) + every PR (against main). Production deploy फक्त main branch merge नंतर + manual approval.

   504→[0535] (Heading 1) 7. PostgreSQL High Availability
   505→[0536] (Heading 2) 7.1 HA Architecture
   506→[0537] () PostgreSQL HA PreOne चा critical infrastructure component आहे. Database failure म्हणजे complete platform outage, त्यामुळे database HA design सर्वात जास्त careful आहे. PreOne Patroni-based HA solution वापरतो जो etcd cluster द्वारे leader election handle करतो. Cluster मध्ये 1 primary + 2 replicas आहेत. Primary सर्व writes handle करतो, replicas सर्व reads (via PgBouncer) handle करतात. Automatic failover < 30 seconds मध्ये complete होतो — primary failure detect → new primary election → connection rerouting via PgBouncer.
   507→[0538] () HA architecture चे key components: Patroni (HA manager), etcd (distributed consensus for leader election), PgBouncer (connection pooling + read/write splitting), pgBackRest (backup + WAL archiving), postgres_exporter (Prometheus metrics). सर्व components containerized आहेत आणि StatefulSet द्वारे managed होतात ज्यामुळे stable network identity + persistent volumes ensure होतात.

   528→[0560] (Heading 2) 7.4 Connection Pooling Strategy
   529→[0561] () PgBouncer हे PreOne चे connection pooling layer आहे जो PostgreSQL connections च्या exhaustion ला prevent करतो. PgBouncer sidecar pattern मध्ये deploy झालेला आहे — प्रत्येक Postgres pod मध्ये PgBouncer container alongside run होतो. PgBouncer transaction-mode pooling वापरतो ज्यामुळे connection per transaction असते, न की connection per session. हे approach connection count कमी करते (1000 → ~50 actual Postgres connections) आणि performance improve होते.
   530→[0562] () Read/write splitting द्वारे read replicas चा efficient use होतो. PgBouncer configured आहे ज्यामुळे queries with /* replica */ comment hint read replicas ला route होतात, बाकी सर्व queries primary ला route होतात. Reports dashboard आणि analytics endpoints read replicas वापरतात ज्यामुळे primary वर जोर कमी पडतो. PgBouncer प्रत्येक pool साठी max 1000 client connections + 50 server connections allow करतो.

   532→[0565] (Heading 2) 8.1 Role of Redis
   533→[0566] () Redis हे PreOne चा second most critical infrastructure component आहे Postgres नंतर. Redis multiple roles handle करतो — cache, session store, rate limiter, OTP cache, BullMQ queue backend, distributed locks, real-time pub/sub, feature flags, idempotency store, आणि geo indexing. हे सर्व use cases single Redis cluster द्वारे handled होतात पण separate logical databases (db 0-9) द्वारे isolation आहे. Redis monitoring via redis_exporter → Prometheus → Grafana dashboard with alerts on memory usage, connection count, hit rate, replication lag.

   707→[0754] (Heading 1) 11. Backup Strategy
   708→[0755] (Heading 2) 11.1 Backup Philosophy
   709→[0756] () Backups हे PreOne च्या disaster recovery चा foundation आहेत. Backup strategy खालील principles follow करते: 3-2-1 rule (3 copies, 2 different media, 1 offsite), automated (no manual backups), tested (daily restore tests), encrypted (at rest + in transit), versioned (multiple point-in-time restores), आणि documented (runbook for every backup type). PreOne च्या backup stack मध्ये Velero (K8s resources + PVs), pgBackRest (Postgres), Redis BGSAVE (Redis), S3 cross-region replication (object storage), आणि GitHub mirror (source code) included आहेत.
   710→[0757] () Backup verification critical आहे — untested backup हा no backup च समान आहे. PreOne मध्ये daily automated restore tests run होतात जे QA cluster मध्ये restore करतात आणि smoke tests execute करतात. Restore test failure trigger करतो PagerDuty alert. Quarterly full DR drill execute होतो जिथे complete production cluster restore simulate केला जातो.

   750→[0799] (Heading 1) 12. Disaster Recovery
   751→[0800] (Heading 2) 12.1 DR Strategy
   752→[0801] () Disaster Recovery (DR) हे PreOne च्या business continuity चा critical component आहे. DR strategy दोन pillars वर built आहे: RPO (Recovery Point Objective — max acceptable data loss) आणि RTO (Recovery Time Objective — max acceptable downtime). PreOne चे targets: RPO ≤ 15 minutes, RTO ≤ 60 minutes. हे targets aggressive आहेत पण preschool domain मध्ये parent communication + fee operations critical असल्यामुळे justified आहेत.
   753→[0802] () DR architecture multi-layered आहे. Layer 1: Multi-AZ deployment (3 AZs) — single AZ failure handle करते. Layer 2: Automated failover (Patroni for Postgres, Sentinel for Redis) — single node failure handle करते. Layer 3: Cross-region replication (S3, ECR) — region failure handle करते. Layer 4: Multi-region standby (v1.0: active-passive; v2.0: active-active) — complete region failure handle करते. Each layer ची testing regular असते — daily automated tests + quarterly full DR drills.
   754→[0803] (Heading 2) 12.2 DR Metrics
   755→[0804] (Heading 2) 12.3 DR Runbook (High-Level)
   756→[0805] () Scenario: Primary Region Complete Failure
   757→[0807] () Phase 1: Detection (0-5 minutes)
   758→[0808] () - Cloudflare health check fails for 3 consecutive checks
   759→[0809] () - Alert triggered to on-call engineer via PagerDuty
   760→[0810] () - On-call acknowledges + starts incident bridge
   761→[0812] () Phase 2: Decision (5-15 minutes)
   762→[0813] () - On-call verifies primary region is genuinely down
   763→[0814] () - On-call escalates to DevOps Lead + Engineering Manager
   764→[0815] () - Decision: failover to DR region OR wait for primary recovery
   765→[0816] () - Decision criteria: estimated primary recovery > 30 min -> failover
   766→[0818] () Phase 3: Failover (15-45 minutes)
   767→[0819] () - Update Cloudflare DNS to point to DR region load balancer
   768→[0820] () - Promote DR Postgres replica to primary (via Patroni)
   769→[0821] () - Scale up DR Kubernetes cluster (min 3 nodes -> 10 nodes)
   770→[0822] () - Deploy latest Helm charts to DR cluster
   771→[0823] () - Verify health checks pass
   772→[0825] () Phase 4: Verification (45-55 minutes)
   773→[0826] () - Smoke tests against DR endpoints
   774→[0827] () - Verify data consistency (last 15 min may be lost - RPO)
   775→[0828] () - Monitor error rates, latency, queue backlog
   776→[0829] () - Communicate to customers via status page
   777→[0831] () Phase 5: Stabilization (55-60 minutes)
   778→[0832] () - Traffic fully routed to DR region
   779→[0833] () - On-call hands off to follow-the-sun team
   780→[0834] () - Post-incident review scheduled within 48 hours
   781→[0836] () Phase 6: Recovery (post-incident)
   782→[0837] () - Primary region restored from latest backup
   783→[0838] () - Data reconciled (DR primary -> restored primary)
   784→[0839] () - Traffic routed back to primary region
   785→[0840] () - DR cluster scaled back down to standby

   818→[0875] (Heading 1) 14. Security
   819→[0876] (Heading 2) 14.1 Security Architecture
   820→[0877] () PreOne ची security architecture defense-in-depth model वर based आहे — multiple security layers द्वारे protection. Single layer compromise होण्याचे case मध्ये other layers protection provide करतात.    891→[0949] (Heading 1) 15. Scalability
   892→[0950] (Heading 2) 15.1 Scalability Strategy
   893→[0951] () PreOne ची scalability strategy horizontal scaling वर based आहे — more pods + more nodes, न की bigger machines. Horizontal scaling preferred कारण ते granular, cost-effective, आणि stateless-friendly आहे. Stateless API pods द्वारे any pod can serve any request — यामुळे HPA (Horizontal Pod Autoscaler) effectively scale करू शकतो. Cluster Autoscaler pending pods detect करतो आणि automatically new nodes add करतो. हे सर्व automated आहे — manual scaling intervention required नाही.
   894→[0952] () Scalability targets: API 3-30 pods, Web 2-10 pods, Worker 2-15 pods, cluster 3-20 nodes. HPA CPU + memory + custom metrics (request latency, queue depth) द्वारे scaling decisions घेतो. Cluster Autoscaler pending pods detect करतो (pods unschedulable due to resource constraints) आणि new nodes provision करतो. Scale-down द्वारे underutilized nodes automatically removed होतात (after 10 min of low utilization).
   895→[0953] (Heading 2) 15.2 Scalability Tactics
   896→[0954] (Heading 2) 15.3 HPA + Cluster Autoscaler Flow
   897→[0955] () 1. Traffic increases -> API pods CPU > 70%
   898→[0956] () 2. HPA detects high CPU via Prometheus metrics
   899→[0957] () 3. HPA increases replicas (3 -> 5)
   900→[0958] () 4. K8s scheduler tries to schedule 2 new pods
   901→[0959] () 5. If insufficient node capacity -> pods Pending
   902→[0960] () 6. Cluster Autoscaler detects Pending pods
   903→[0961] () 7. Cluster Autoscaler provisions new node (1-2 min)
   904→[0962] () 8. New pods scheduled on new node
   905→[0963] () 9. New pods pass readiness check
   906→[0964] () 10. NGINX routes traffic to new pods
   907→[0965] () 11. CPU usage normalizes
   908→[0967] () Scale-down flow (reverse):
   909→[0968] () 1. Traffic decreases -> CPU < 30%
   910→[0969] () 2. HPA decreases replicas (5 -> 3)
   911→[0970] () 3. Pods gracefully terminate (SIGTERM)
   912→[0971] () 4. NGINX removes from upstream
   913→[0972] () 5. If node underutilized -> Cluster Autoscaler drains
   914→[0973] () 6. Node removed after 10 min
