=== Table 3 ===
Term | Definition
ABAC | Attribute-Based Access Control. Authorization model that uses attributes (user, resource, environment) to make access decisions. PreOne reserves schema space for ABAC (ADR-015) but does not implement it in v1.0.
Action Library | Centralized collection of 10 primitive actions (View, Create, Edit, Delete, Approve, Reject, Export, Import, Print, Share) defined once and referenced by module bundles (ADR-014).
Aggregate Root | DDD concept. The root entity of an aggregate that enforces consistency boundaries. All modifications to the aggregate go through the root. Example: FeeInvoice is an aggregate root containing FeeInvoiceItem entities.
Assignment | A link between a user, a role, and a scope (branch, class, section, activity, route, shift). Assignments determine what data a user can access within a module (ADR-017).
Bounded Context | DDD concept. A boundary within which a ubiquitous language and domain model apply. PreOne has 13 bounded contexts, one per business domain (ADR-005, ADR-029).
Business Bundle | Level 1 of the 3-level permission bundle. Groups module bundles by business domain (Finance, Inventory, Academics, etc.) (ADR-013).
Business Domain | Top-level architecture partition. PreOne has 13 business domains (ADR-005).
Cache Invalidation | Process of marking cached data as stale so it is rebuilt from the source. PreOne uses version-based invalidation (ADR-022) with event-driven staggered rebuild (ADR-032).
Cache Stampede | Failure mode where many cache misses occur simultaneously, overwhelming the source database with concurrent rebuilds. Prevented by single-flight pattern and staggered rebuild (ADR-032).
Clean Architecture | Architectural pattern that separates concerns into layers: Presentation, Application, Domain, Infrastructure. Dependencies point inward toward domain (ADR-030).
Context-Based Access | Authorization model where permissions are unioned across roles and data scope is computed per-module based on the user's role assignments for that module (ADR-010).
CQRS | Command Query Responsibility Segregation. Pattern that separates read operations from write operations, enabling independent optimization. PreOne uses CQRS where justified (ADR-030).
DDD | Domain-Driven Design. Approach to software development that models the domain explicitly using aggregates, entities, value objects, and domain events (ADR-029).
Domain Event | A fact about a state change in a domain, published to other domains for loose coupling. Example: StudentCreatedEvent published by Admission domain (ADR-036).
Effective Access Resolver | Single service that computes a user's effective permissions and scopes by resolving their roles, assignments, and permission bundles. Cached in Redis (ADR-016).
Fail-Closed | Authorization failure mode where access is denied when the system cannot determine permissions. PreOne uses fail-closed with LKG snapshot fallback (ADR-031).
Feature Flag | Configuration that enables or disables a feature without code deployment. PreOne uses 3-level flags: platform, school, plan (ADR-037).
Hybrid Assignment Model | Assignment schema with a master assignment table plus scope-specific child tables (assignment_branch, assignment_class, etc.) preserving referential integrity (ADR-017).
LKG | Last Known Good. A snapshot of a user's last known permissions, used as fallback when the primary cache (Redis) is unavailable (ADR-031).
Modular Monolith | Deployment architecture where the application is a single deployable unit with strong module boundaries. Modules can be extracted to microservices later if scale demands (ADR-028).
Module Bundle | Level 2 of the 3-level permission bundle. Groups action bundles by module (Fee Collection, Invoices, Receipts) (ADR-013).
Multi-Tenant | Architecture where a single software instance serves multiple customers (tenants). PreOne uses shared database with tenant_id column (ADR-001).
Permission Bundle | Pre-packaged set of permissions assignable to a role in one action. PreOne uses 3-level bundles (ADR-013).
RBAC | Role-Based Access Control. Authorization model where permissions are assigned to roles, and users gain permissions via role assignments. PreOne's core authorization model (ADR-010 to ADR-016).
RLS | Row-Level Security. PostgreSQL feature that enforces row-level access control via policies. PreOne uses RLS as defense-in-depth for tenant isolation (ADR-025).
Scope | Data boundary for a user's access within a module. Examples: class scope (Jr.KG B), branch scope (Mumbai HQ), school scope (whole school) (ADR-011).
Scope-Per-Role-Per-Module | Algorithm that computes a user's data scope for a module by unioning the scopes of all their roles that grant access to that module (ADR-011).
Single-Flight | Pattern where multiple concurrent requests for the same resource share a single in-flight operation, preventing thundering herd (ADR-032).
Soft Delete | Marking a record as deleted (is_deleted=true) without physically removing it. PreOne uses tiered soft delete policy (ADR-034).
System Role | A role with is_system=true that bypasses tenant isolation but goes through permission checks and audit logging. Examples: SUPER_ADMIN, SUPPORT_ENGINEER, MIGRATION_TOOL (ADR-026).
Tenant | A customer organization in a multi-tenant SaaS. In PreOne, each School is a tenant (ADR-001).
TTL | Time To Live. Cache expiry duration. PreOne uses TTL as safety net but relies on version-based invalidation for correctness (ADR-022).
Ubiquitous Language | DDD concept. Shared vocabulary between domain experts and code. PreOne enforces this per domain (ADR-029).
User Group | HR/organizational classification for users (Teaching Staff, Office, Operations, Management, Support). Doubles as workflow routing input (ADR-008, ADR-009).
User-Role Separation | Pattern where users and roles are separate entities linked by many-to-many relationship, enabling multi-role users (ADR-007).
Version-Based Invalidation | Cache invalidation strategy where a global version number tracks changes; cache entries with stale versions are rebuilt on next access (ADR-022).
