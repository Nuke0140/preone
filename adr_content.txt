[Heading 2] ADR-001: Multi-Tenant SaaS Architecture
[Normal] Context
[Normal] PreOne targets the Indian preschool market with a SaaS product that must serve thousands of schools on a single platform. Each school is an independent business entity with its own students, staff, fees, and academic operations. Sharing infrastructure reduces per-school cost, but data isolation between schools is non-negotiable because student records, medical information, and financial data carry legal and ethical obligations under the Indian DPDP Act 2023.
[Normal] The product must scale from a small standalone preschool with five staff members to a multi-branch chain with hundreds of employees, all on the same codebase. Tenant isolation must therefore be enforced at the data layer, not just at the application layer, to prevent cross-school data leakage in the event of a query bug or migration error.
[Normal] Options Considered
[List Paragraph] Database-per-tenant: Each school gets its own PostgreSQL database. Strongest isolation, but operationally expensive at 1000+ schools — connection pooling, migrations, backups, and observability multiply linearly with tenant count.
[List Paragraph] Schema-per-tenant: Shared database, separate schemas per school. Moderate isolation, but migration scripts must iterate across hundreds of schemas and schema count caps in PostgreSQL make this impractical beyond ~500 tenants.
[List Paragraph] Shared database with tenant_id column: Single schema, every table carries a tenant_id (school_id) column, every query filters on it. Cheapest to operate, weakest isolation in raw SQL, but enforces isolation via repository-level auto-injection.
[Normal] Decision
[Normal] Adopt shared database with mandatory tenant_id column on every aggregate root. Tenant isolation is enforced at three layers: (1) repository-level auto-injection of WHERE tenant_id = ? into every query, (2) application-level validation that no cross-tenant foreign key is ever created, and (3) row-level security policies on PostgreSQL as a defense-in-depth measure.
[Normal] The tenant_id is named school_id throughout the schema to align with the business vocabulary of PreOne, where the school is the tenant boundary. Multi-branch schools share the same school_id but differentiate via branch_id.
[Normal] Rationale
[Normal] Operational cost is the decisive factor. At 1000+ schools, database-per-tenant requires 1000 separate connection pools, 1000 backup jobs, and 1000 migration runs per release — operationally unsustainable for a startup engineering team. Schema-per-tenant hits PostgreSQL's practical schema limit (~500) before market saturation.
[Normal] Shared database with auto-injected tenant_id filters is the industry-standard pattern used by Shopify, GitHub, and Salesforce for multi-tenant SaaS. The risk of cross-tenant leakage is mitigated by never letting developers write raw tenant_id filters manually — the repository layer enforces it transparently, and any query that bypasses the repository throws a runtime error in CI.
[Normal] PostgreSQL Row-Level Security (RLS) provides defense-in-depth: even if a query bug bypasses the repository filter, RLS policies reject rows whose tenant_id does not match the current session's set tenant context.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Single database to back up, monitor, and migrate — operational cost is constant regardless of tenant count.
[List Paragraph] Cross-tenant analytics (platform-level dashboards, Super Admin views) are trivial — no federation needed.
[List Paragraph] Tenant onboarding is a single INSERT into the School table, not a database creation script.
[Heading 3] Negative / Trade-offs
[List Paragraph] A single query bug can leak data across tenants — mitigated by repository auto-injection and RLS.
[List Paragraph] Noisy-neighbor risk: one heavy query from a large tenant can slow others — mitigated by query timeout policies and per-tenant rate limiting.
[List Paragraph] Migration risk is amplified: a bad migration affects all tenants at once — mitigated by ADR-033 (Migration Identity) with dry-run support.
[Normal] Implementation Notes
[Normal] Every aggregate root table must include the following mandatory columns in addition to its primary key and business columns:
[Normal] The repository pattern enforces tenant_id injection. Developers never write WHERE clauses manually — the BaseRepository class appends tenant_id from the request context automatically.
[Normal] -- Mandatory columns on every aggregate root table
[Normal] school_id      UUID NOT NULL REFERENCES school(id),
[Normal] branch_id      UUID NULL REFERENCES branch(id),  -- NULL for school-level aggregates
[Normal] created_by     UUID NOT NULL REFERENCES "user"(id),
[Normal] updated_by     UUID NOT NULL REFERENCES "user"(id),
[Normal] created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
[Normal] updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
[Normal] -- Index on (school_id, ...) is mandatory for every query path
[Normal] CREATE INDEX idx_student_school ON student(school_id, branch_id);
[Normal] -- PostgreSQL Row-Level Security as defense-in-depth
[Normal] ALTER TABLE student ENABLE ROW LEVEL SECURITY;
[Normal] CREATE POLICY tenant_isolation ON student
[Normal]   USING (school_id = current_setting('app.tenant_id')::uuid);

[Heading 2] ADR-002: Multi-School, Multi-Branch, Multi-Academic-Year Isolation
[Normal] Context
[Normal] PreOne is a multi-tenant SaaS where each tenant is a School. Within a school, there can be multiple Branches (geographically separate campuses). Within a branch, operations are organized by Academic Year — student admissions, fee structures, attendance, and academic records all carry academic_year context.
[Normal] Data from one academic year must remain queryable but must not pollute the current year's views. For example, a student's promotion history shows all years, but the daily operations dashboard shows only the current academic year. Similarly, fee invoices from 2024-2025 must remain accessible for audit and refunds even when the school is operating in 2025-2026.
[Normal] Options Considered
[List Paragraph] Separate schemas per academic year: Each year gets its own schema. Strong isolation but breaks cross-year queries (promotion history, cumulative attendance) and complicates year-end promotion logic.
[List Paragraph] Soft isolation via academic_year_id column: All operational tables carry academic_year_id. Current-year views filter on is_current=true on AcademicSession. Cross-year queries explicitly include the column.
[List Paragraph] Archive previous years to cold storage: Move past years to a separate archive database. Reduces operational table size but makes historical queries slow and breaks audit trail continuity.
[Normal] Decision
[Normal] Adopt Option 2: every operational table (attendance, fee_invoice, daily_sheet, observation, assessment, etc.) carries a mandatory academic_year_id column. The AcademicSession table tracks which year is is_current=true per branch. Default query filters automatically apply academic_year_id = current_session, but explicit queries can override for historical access.
[Normal] Year-end promotion is a transactional operation that creates new StudentAllocation rows for the new academic year while preserving historical allocations. Promotion never deletes or overwrites past-year data.
[Normal] Rationale
[Normal] Academic year isolation is a business requirement, not just an optimization — fee invoices, attendance records, and report cards from past years are legally required to be retained and queryable for at least 7 years per Indian education regulations.
[Normal] Soft isolation with academic_year_id keeps all data in one schema, simplifying cross-year analytics (cumulative growth reports, attendance trends, fee collection history) while still enabling current-year filtering by default. This matches how teachers and administrators actually use the system — they live in the current year but occasionally need historical context.
[Normal] Schema-per-year and archive-to-cold-storage approaches both break the natural query patterns of school operations. Year-end promotion would require complex data movement, and historical queries would require federation logic that adds latency and complexity.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Single schema for all years — operational simplicity for backups, migrations, and observability.
[List Paragraph] Cross-year queries (promotion history, cumulative attendance) are standard SQL with no federation.
[List Paragraph] Year-end promotion is a transaction, not a data migration — atomic and reversible.
[Heading 3] Negative / Trade-offs
[List Paragraph] Operational tables grow large over time (10+ years of attendance, fees) — mitigated by table partitioning on academic_year_id for high-volume tables.
[List Paragraph] Default query filters must always include academic_year_id — mitigated by repository pattern that injects it automatically.
[Normal] Implementation Notes
[Normal] Every operational table includes academic_year_id as a mandatory foreign key. The repository layer auto-injects the current academic_year_id from branch context. Tables projected to exceed 10M rows per year (attendance, daily_sheet, photo) are partitioned by academic_year_id using PostgreSQL declarative partitioning.
[Normal] -- AcademicSession tracks current year per branch
[Normal] CREATE TABLE academic_session (
[Normal]   id UUID PRIMARY KEY,
[Normal]   school_id UUID NOT NULL REFERENCES school(id),
[Normal]   branch_id UUID NOT NULL REFERENCES branch(id),
[Normal]   name VARCHAR(20) NOT NULL,  -- e.g., '2025-2026'
[Normal]   start_date DATE NOT NULL,
[Normal]   end_date DATE NOT NULL,
[Normal]   is_current BOOLEAN NOT NULL DEFAULT FALSE,
[Normal]   status VARCHAR(20) NOT NULL,  -- PLANNED/ACTIVE/CLOSING/CLOSED/ARCHIVED
[Normal]   UNIQUE(branch_id, name)
[Normal] );
[Normal] -- Operational tables reference academic_year_id
[Normal] CREATE TABLE student_attendance (
[Normal]   id UUID PRIMARY KEY,
[Normal]   school_id UUID NOT NULL,
[Normal]   branch_id UUID NOT NULL,
[Normal]   academic_year_id UUID NOT NULL REFERENCES academic_session(id),

[Heading 2] ADR-003: Module-Based Architecture (Not Role-Based)
[Normal] Context
[Normal] Early ERP systems organized their navigation by user role — a Teacher login showed teacher-only menus, an Admin login showed admin menus, a Parent login showed parent menus. This approach locks modules to roles: adding a new role (e.g., HR Manager) requires re-architecting navigation, and a user who holds multiple roles must switch logins or see a confusing merged menu.
[Normal] PreOne must serve small preschools where one staff member may hold 4-5 roles (Class Teacher + Activity Teacher + Inventory Incharge + Reception) due to manpower constraints. A role-based navigation model would force role switching or produce a chaotic menu. The architecture must decouple modules from roles entirely.
[Normal] Options Considered
[List Paragraph] Role-based navigation: Each role has its own menu structure. Switching roles requires re-login or role switcher. Simple for single-role users, broken for multi-role users.
[List Paragraph] Module-based navigation with role-driven visibility: All modules exist as flat peers. A user sees a module if any of their roles grants access to it. Within a module, role determines what actions are available.
[List Paragraph] Custom dashboard per user: Each user configures their own menu. Maximum flexibility but minimum standardization — onboarding and training become school-specific.
[Normal] Decision
[Normal] Adopt Option 2: Module-Based Architecture. There are 13 master modules (Dashboard, CRM & Admission, Student Management, Parent Management, Staff Management, Academic, Daily Operations, Communication, Finance, Transport, Inventory, Reports, Settings). Modules are flat — there is no role-based navigation tree.
[Normal] Role determines: (1) which modules appear in a user's navigation, (2) what actions (View/Create/Edit/Delete/Approve) are available within each module, (3) what data scope (which classes, branches, students) the user sees within each module. A user with multiple roles sees the union of modules from all their roles, with action permissions unioned per module.
[Normal] Rationale
[Normal] Module-based architecture scales when roles change. Adding a new role (HR Manager, Librarian, Owner) requires only defining the role's module access and permissions — no navigation restructuring. This is critical for a SaaS product where every school has slightly different role definitions.
[Normal] The multi-role user case (Pooja as Class Teacher + Activity Teacher + Inventory Incharge) is handled naturally — Pooja sees Today, My Class, Observations, Curriculum, Inventory, and Reports modules in her navigation. She does not need to switch roles to access Inventory; the system grants access based on her role union.
[Normal] This pattern aligns with industry-standard ERPs (SAP, Oracle, Workday) where modules are first-class citizens and roles are permission bundles. It also matches the Teacher Portal design already specified for PreOne, where 7 operational modules (Today, My Class, Observations, Talk, Curriculum, Insights, Me) are flat and role determines visibility.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] New roles can be added without touching module code — only permission matrix changes.
[List Paragraph] Multi-role users get unified navigation without role switching.
[List Paragraph] Modules can be developed and tested independently — module teams don't block on role definitions.
[Heading 3] Negative / Trade-offs
[List Paragraph] Navigation can become long for users with many roles — mitigated by grouping modules by Business Domain in the UI (Foundation, Operations, Academic, Finance, etc.).
[List Paragraph] Permission matrix grows large (13 modules × 11 roles × 6 actions = 858 cells) — mitigated by ADR-013 Permission Bundles that pre-package common permission sets.
[Normal] Implementation Notes
[Normal] The 13 master modules are defined as a static enumeration in the codebase. Each module is implemented as a DDD bounded context with its own controllers, services, repositories, and database tables. Modules communicate via domain events (ADR-036), never via direct database access to another module's tables.
[Normal] The frontend navigation is generated dynamically from the Effective Access Resolver output (ADR-016) — the resolver returns a list of (module, allowed_actions) tuples, and the UI renders navigation accordingly.

[Heading 2] ADR-004: 3-Layer Architecture (Business / Module / Role)
[Normal] Context
[Normal] PreOne's architecture must serve multiple audiences: product managers think in business domains (Admissions, Academics, Finance), engineers build in modules (Student Management, Fee Collection), and administrators configure in roles (Teacher, Principal, Accountant). A flat architecture that mixes these concerns produces confused codebases where business logic, module implementation, and role permissions are tangled together.
[Normal] The Teacher Portal already separates operational modules (Today, My Class, Observations, etc.) from role-based visibility. This pattern must extend platform-wide so that adding a new role never requires touching module code, and adding a new module never requires touching role definitions.
[Normal] Options Considered
[List Paragraph] Single-layer (role-centric): Roles directly contain modules and permissions. Adding a role means redefining its module list. Tight coupling, brittle to change.
[List Paragraph] 2-layer (module + role): Modules are flat, roles grant module access. No business domain grouping — modules are an unstructured list.
[List Paragraph] 3-layer (business domain + module + role): Business Domains group modules. Modules contain features. Roles grant module+action access. Each layer is independently evolvable.
[Normal] Decision
[Normal] Adopt Option 3: 3-Layer Architecture with Business Domains at the top, Modules in the middle, and Roles at the bottom.
[Normal] Layer 1 (Business Domains): 13 conceptual groupings — Marketing & CRM, Admission, Student Lifecycle, Academics, Daily Operations, Inventory & Procurement, Communication, Finance, HR, Administration, Reports & Analytics, Settings, Platform Management.
[Normal] Layer 2 (Modules): Concrete functional units within a domain. E.g., Finance domain contains Fee Structure, Invoices, Payments, Receipts, Expenses, Reports modules.
[Normal] Layer 3 (Roles): Permission bundles that grant (module, action) pairs. Roles never directly reference business domains — they only see modules.
[Normal] Rationale
[Normal] Three layers provide clean separation of concerns. Product managers evolve business domains (e.g., splitting Administration into Transport and Compliance) without touching role definitions. Engineers evolve modules (e.g., adding a new fee report submodule) without touching business domain structure. Administrators evolve roles (e.g., adding a Transport Manager role) without touching module code.
[Normal] Each layer is independently evolvable because each layer only references the layer directly below it. Roles reference modules, not domains. Modules are grouped into domains for navigation and reporting only — there is no hard dependency from module to domain.
[Normal] This pattern is used by enterprise ERPs (SAP's module-fication, Workday's domain-driven design) and matches the Teacher Portal design where Today, My Class, etc. are operational modules within the Teaching business domain.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Layers evolve independently — domain restructuring does not break roles, role changes do not break modules.
[List Paragraph] Clear ownership: product owns domains, engineering owns modules, admins own roles.
[List Paragraph] Matches Teacher Portal's existing flat-module + role-visibility design.
[Heading 3] Negative / Trade-offs
[List Paragraph] Three layers add conceptual overhead for new engineers — mitigated by clear documentation in this ADR.
[List Paragraph] Risk of layers drifting out of sync (e.g., module removed but role still references it) — mitigated by referential integrity in the permission table and CI tests.

[Heading 2] ADR-005: 13 Business Domains
[Normal] Context
[Normal] PreOne's product scope covers the entire preschool operating lifecycle — from marketing leads to admissions, daily operations, academics, finance, HR, and platform management. The product must be organized into coherent business domains so that engineering teams can own bounded contexts, product managers can roadmap features within their domain, and the schema can be partitioned cleanly.
[Normal] An ad-hoc collection of features without domain boundaries produces a tangled codebase where a Fee change touches Student, where a Daily Sheet change touches Inventory, and where no one owns the orphan features that fall between the cracks. Domains must be explicit and exhaustive — every feature must belong to exactly one domain.
[Normal] Options Considered
[List Paragraph] Fewer, larger domains (6-8): Coarse-grained grouping like 'Operations', 'Academics', 'Finance'. Easier to navigate but teams become large and ownership is diffuse.
[List Paragraph] More, smaller domains (20+): Fine-grained micro-domains. Excellent ownership clarity but cross-domain workflows become complex and DDD bounded contexts proliferate.
[List Paragraph] 13 balanced domains: Domain count tuned to preschool operations — neither too coarse (Finance stays one domain) nor too fine (HR stays one domain even with payroll + recruitment + training). Each domain maps naturally to a DDD bounded context.
[Normal] Decision
[Normal] Adopt Option 3: 13 Business Domains as the top-level architecture partition. The domains are: (1) Marketing & CRM, (2) Admission, (3) Student Lifecycle, (4) Academics, (5) Daily Operations, (6) Inventory & Procurement, (7) Communication, (8) Finance, (9) Human Resources, (10) Administration, (11) Reports & Analytics, (12) Settings, (13) Platform Management.
[Normal] Each domain is a DDD bounded context with its own schema namespace, its own engineering team (or sub-team), its own release cadence, and its own domain events. Cross-domain communication is via events (ADR-036) only — never via direct database access.
[Normal] Rationale
[Normal] The 13 domains emerged from analyzing PreOne's v2 database schema (142 models) and naturally grouping them by business concern. Each domain has 5-20 models, which is the sweet spot for DDD bounded context size — large enough to have meaningful aggregates, small enough for one team to own.
[Normal] Inventory is a separate domain (ADR-027) rather than a sub-module of Administration because preschool inventory (toys, Montessori materials, art supplies) is operationally critical and deserves first-class treatment. This was a deliberate upgrade from typical school ERPs that bury inventory as an admin afterthought.
[Normal] Platform Management (SaaS layer) is a separate domain because it has fundamentally different concerns from school operations — multi-school management, subscription billing, storage usage, feature flags. Mixing platform concerns with school concerns would couple two different scaling axes.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Clear ownership — each domain has a dedicated team and a clear schema boundary.
[List Paragraph] Independent release cadence — Finance can release weekly without coordinating with Academics.
[List Paragraph] Schema partitioning — each domain's tables can be migrated, optimized, and scaled independently.
[List Paragraph] Onboarding — new engineers join one domain and master it before crossing boundaries.
[Heading 3] Negative / Trade-offs
[List Paragraph] 13 domains may be too many for a small startup team — mitigated by one team owning 2-3 related domains initially.
[List Paragraph] Cross-domain workflows (e.g., Admission → Student → Finance) require event choreography — mitigated by ADR-036 Domain Events.
[Normal] Implementation Notes
[Normal] Each domain is implemented as a NestJS module with its own controllers, services, repositories, and entity definitions. Domains are deployed as a single modular monolith (ADR-028) initially, with the option to extract high-traffic domains (Finance, Daily Operations) into separate services later.
[Normal] Domain event publishers and consumers are wired via an in-process event bus initially, with the option to swap to a message broker (RabbitMQ, Kafka) when the modular monolith is split.

[Heading 2] ADR-006: 11 Final Roles
[Normal] Context
[Normal] PreOne must serve preschools ranging from 5-staff small standalone setups to 200-staff multi-branch chains. The role system must accommodate both ends without re-architecture. Too few roles force uncomfortable role-sharing (one person wearing many hats); too many roles create administrative overhead and permission matrix bloat.
[Normal] The Teacher Portal already defines 4 operational roles (Class Teacher, Activity Teacher, Coordinator, Principal) and the Parent Portal defines Parent as a primary role. The platform layer requires Super Admin. A complete role enumeration must cover all 13 business domains without leaving any operational gap.
[Normal] Options Considered
[List Paragraph] 8 roles (minimal): Super Admin, School Admin, Principal, Teacher, Parent, Accountant, Reception, Inventory. Missing Coordinator, Activity Teacher, Transport Manager, HR — gap in real-world coverage.
[List Paragraph] 15+ roles (fine-grained): Splits Teacher into Class Teacher/Activity Teacher/Substitute; splits Admin into Admin/Owner/Manager. High fidelity but permission matrix becomes unwieldy for small schools.
[List Paragraph] 11 roles (balanced): Super Admin, School Admin, Principal, Coordinator, Class Teacher, Activity Teacher, Reception/Admission, Accounts, Inventory/Store Keeper, Transport Manager, HR, Parent, Student (Future). Total 13 including Parent and Student-Future.
[Normal] Decision
[Normal] Adopt Option 3: 11 staff roles + Parent + Student (Future) = 13 total. The 11 staff roles cover all 13 business domains without overlap. Student is marked Future — not implemented in v1.0 but reserved in the schema for future learning activities for older preschoolers.
[Normal] Roles are defined as system roles (is_system=true) that cannot be deleted by school admins. Schools can create custom roles by copying a system role and modifying permissions, but the 11 base roles are immutable.
[Normal] Rationale
[Normal] The 11 roles map cleanly to the 13 business domains. School Admin and Super Admin cover Settings + Platform Management. Principal covers Reports & Analytics. Coordinator is a mid-level teaching supervisor role that exists in mid-to-large preschools. Activity Teacher is separated from Class Teacher because art/music/dance/sports are distinct skill sets in preschools and deserve dedicated permissions.
[Normal] Transport Manager and Inventory/Store Keeper are dedicated roles because these functions involve physical assets (buses, stock) that require accountability. Bundling them into Admin would create accountability gaps — when stock goes missing, the audit trail must show who had inventory permissions.
[Normal] HR is a separate role because HR functions (payroll, performance reviews, recruitment) involve sensitive data that should not be visible to School Admin by default. In small schools, the same person may hold HR + Admin roles (handled by ADR-007 User-Role Separation), but the role boundaries remain clean for audit purposes.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Complete coverage of all 13 business domains without role gaps.
[List Paragraph] Audit-friendly — each role has clear accountability for its domain.
[List Paragraph] Scales from 5-staff schools (one person holds 4-5 roles) to 200-staff chains (each role is a dedicated person).
[Heading 3] Negative / Trade-offs
[List Paragraph] 11 roles may seem like a lot for small preschool administrators — mitigated by ADR-013 Permission Bundles that pre-package role templates.
[List Paragraph] Role definitions are static (system roles) — schools wanting custom roles must use the copy-and-modify pattern, not edit system roles.
[Normal] Implementation Notes
[Normal] The 11 system roles are seeded on platform bootstrap and cannot be deleted. Each system role has a default permission bundle that grants access to relevant modules. Schools can create custom roles by copying a system role and modifying the permission bundle.

[Heading 2] ADR-007: User-Role Separation
[Normal] Context
[Normal] Most Indian school ERPs collapse User and Role into a single concept — a user record has a role field, and changing roles means updating the user. This model breaks for preschools where one person holds multiple roles due to manpower constraints. Kavita, a school admin at a small preschool, may also handle Reception, Accounts, and HR — the system must reflect this reality without creating four separate logins.
[Normal] Separating User from Role also future-proofs the schema. When a teacher is promoted to Coordinator, the system records a role history (Teacher until June 2025, Coordinator from July 2025) rather than overwriting the role field. This history is essential for audit and for permission evaluation of historical actions.
[Normal] Options Considered
[List Paragraph] Single role per user (role field on user): users.role_id = X. Simple but cannot represent multi-role users. Promotion overwrites history.
[List Paragraph] Multiple roles via comma-separated field: users.role_ids = 'X,Y,Z'. Query-unfriendly, no temporal validity, no role history.
[List Paragraph] Many-to-many UserRoles table with temporal validity: user_roles table links users to roles with valid_from/valid_to. Supports multi-role, role history, and time-bound assignments.
[Normal] Decision
[Normal] Adopt Option 3: User-Role Separation via a many-to-many user_roles table. A user can hold any number of roles simultaneously. Each user_role row has valid_from and valid_to timestamps, enabling role history and time-bound role assignments (e.g., Pooja is Reception for the morning shift only, 9 AM to 12 PM).
[Normal] One role per user is marked as primary_role_id on the user record — this determines the default dashboard after login. Additional roles are loaded by the Effective Access Resolver (ADR-016) and their permissions are unioned.
[Normal] Rationale
[Normal] User-Role Separation is the industry-standard pattern for enterprise identity systems (Microsoft Active Directory, Okta, AWS IAM). It treats users as identities and roles as capabilities — the two have different lifecycles and different governance processes.
[Normal] The multi-role use case is not an edge case in Indian preschools — it is the norm. A 5-staff preschool cannot afford dedicated Reception, Accounts, HR, and Admin staff. The system must embrace this reality, not fight it.
[Normal] Temporal validity (valid_from/valid_to) supports role history for audit. When investigating who approved a fee waiver in March 2025, the system can resolve Pooja's roles at that point in time, even if she has since changed roles.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Multi-role users handled naturally — no role switching or multiple logins.
[List Paragraph] Role history preserved for audit and compliance.
[List Paragraph] Time-bound role assignments (shift-based, contract-based) supported.
[List Paragraph] Role deletion does not affect users — only the user_role link is severed.
[Heading 3] Negative / Trade-offs
[List Paragraph] Permission evaluation requires joining user_roles, role_module_permission, and role tables — mitigated by ADR-021 Permission Caching.
[List Paragraph] Primary role must be explicitly designated — mitigated by UI prompting during role assignment.
[Normal] Implementation Notes
[Normal] The user_roles table is the link table. It includes temporal validity fields (valid_from, valid_to) and a status field for soft states (ACTIVE, SUSPENDED). The primary role is denormalized onto the user record for fast default-dashboard resolution at login time.

[Heading 2] ADR-008: User Groups Only (Drop Role Groups)
[Normal] Context
[Normal] An initial proposal suggested two grouping concepts: User Groups (organizational classification like Teaching Staff, Office, Operations) and Role Groups (grouping of roles like Teaching Group = Class Teacher + Activity Teacher + Coordinator). The intent was to use User Groups for HR reporting and Role Groups for permission bundle assignment.
[Normal] On review, Role Groups add a redundant layer. A role is already a capability bundle — grouping roles into meta-bundles produces three levels of indirection (User → Role Group → Roles → Permissions) when two levels (User → Roles → Permissions) suffice. Role Groups would not add capability, only complexity.
[Normal] Options Considered
[List Paragraph] Both User Groups and Role Groups: Two parallel grouping concepts. More flexibility but redundant — Role Group's job is already done by Permission Bundles (ADR-013).
[List Paragraph] User Groups only: One grouping concept for organizational classification. Roles are flat — no meta-grouping. Permission Bundles handle the role-permission assignment optimization.
[List Paragraph] Role Groups only (no User Groups): Drop organizational classification. Loses HR reporting capability (Teaching Staff count, Office Staff count, etc.).
[Normal] Decision
[Normal] Adopt Option 2: User Groups only. User Groups are an HR/organizational classification (Teaching Staff, Office, Operations, Management, Support) that serves workflow routing, dashboard defaults, and batch operations. Roles remain flat — there is no Role Group concept in the schema.
[Normal] Permission Bundle optimization (ADR-013) replaces what Role Groups would have done — bundling common permission sets for easy role assignment. Role Groups would have been a poor man's Permission Bundles, and the real Permission Bundles make them redundant.
[Normal] Rationale
[Normal] User Groups and Roles serve different purposes. User Groups answer 'who is this person in the organization?' (HR classification). Roles answer 'what can this person do in the system?' (capability). Conflating them produces confusion — a Reception staff member who is also handling Accounts has the User Group 'Office' but two distinct Roles.
[Normal] Role Groups would have introduced a third concept that overlaps with Permission Bundles. The 3-Level Permission Bundle (Business → Module → Action) already provides the 'group of permissions' abstraction that Role Groups would have provided at the role level. Adding Role Groups would force the system to maintain two parallel bundling mechanisms.
[Normal] User Groups as workflow routing inputs (ADR-009) is the killer feature — when an incident is reported, the system routes a notification to the Operations Group; when a leave request is submitted, it routes to the user's reporting manager within their User Group. This makes User Groups an active workflow participant, not just an HR label.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Single grouping concept — simpler mental model for admins and engineers.
[List Paragraph] User Groups serve double duty as HR classification and workflow routing input.
[List Paragraph] No redundant Role Group layer to maintain or migrate.
[Heading 3] Negative / Trade-offs
[List Paragraph] Schools wanting to group roles (e.g., 'all teaching roles get a default bundle') must use Permission Bundles — which is actually the correct abstraction anyway.

[Normal] User Groups are stored in a user_groups table with many-to-many relationship to users via user_user_groups. A user can belong to multiple groups (e.g., Pooja is in both Teaching Staff and Operations groups). Groups are seeded with 5 defaults and schools can add custom groups.

[Heading 2] ADR-009: User Groups as Workflow Engine Input
[Normal] Context
[Normal] User Groups were originally conceived as an HR classification label — Teaching Staff, Office, Operations, etc. The label would be useful for reporting (count of teaching staff) but would not actively drive system behavior.
[Normal] On review, this underutilizes User Groups. Enterprise workflow engines use organizational groupings as routing inputs — when an incident is reported, route to Operations; when a fee is overdue, route to Finance; when a leave request is submitted, route to the user's reporting manager (resolved via group hierarchy). User Groups should be active workflow participants, not passive labels.
[Normal] Options Considered
[List Paragraph] User Groups as HR labels only: Passive classification for reporting. No workflow integration. Simplest but underutilizes the data.
[List Paragraph] User Groups as workflow routing inputs: Active participation — workflow engine queries User Groups to determine notification recipients, approval chains, and default dashboard layouts.
[List Paragraph] Both, but with separate HR and Workflow group tables: Two parallel grouping systems — one for HR, one for workflow. Maximum flexibility but redundant and confusing.
[Normal] Decision
[Normal] Adopt Option 2: User Groups are active workflow routing inputs. The workflow engine queries User Groups to determine: (1) notification recipients for events (incident → Operations Group), (2) approval chains (leave request → reporting manager in user's group), (3) default dashboard layouts (Teaching Staff group → Teacher dashboard), (4) batch operations (broadcast message to Support Staff group).
[Normal] User Groups become a first-class workflow entity, not just an HR tag. Group membership changes trigger workflow re-evaluation (e.g., when Pooja leaves the Operations group, she stops receiving incident notifications).
[Normal] Rationale
[Normal] Workflow routing is the most common reason enterprise systems need organizational groupings. Without workflow integration, User Groups are decorative — admins fill them in but the system never uses them. With workflow integration, User Groups become operational infrastructure that drives daily system behavior.
[Normal] The four workflow use cases (notification routing, approval chains, dashboard defaults, batch operations) cover the majority of real-world preschool workflows. Hardcoding these routes per role would be brittle (role changes break routing); routing via User Groups is resilient (group membership can change without touching workflow definitions).
[Normal] This pattern aligns with industry workflow engines (Camunda, Bonita, Microsoft Power Automate) where organizational units are first-class routing inputs. PreOne's workflow engine will use the same pattern at smaller scale.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] User Groups actively drive system behavior — admins have incentive to keep them accurate.
[List Paragraph] Workflow routing is decoupled from role definitions — role changes do not break routing.
[List Paragraph] Four workflow use cases covered: notifications, approvals, dashboards, batch operations.
[Heading 3] Negative / Trade-offs
[List Paragraph] Group membership changes must trigger workflow re-evaluation — adds complexity to the membership update API.
[List Paragraph] Misconfigured groups can break workflows (e.g., empty Operations Group means incident notifications go nowhere) — mitigated by validation that prevents empty system groups.
[Normal] Implementation Notes
[Normal] The workflow engine subscribes to group membership change events. When a user is added to or removed from a group, the engine re-evaluates pending workflows that depend on that group's membership. Workflow definitions reference groups by name (e.g., route_to: 'operations') and the engine resolves names to user lists at runtime.

[Heading 2] ADR-010: Context-Based Access (Role Switch UI Only)
[Normal] Context
[Normal] When a user holds multiple roles (Pooja is Class Teacher + Activity Teacher + Inventory Incharge), two questions arise: (1) Should the user explicitly switch roles to access different modules? (2) Should permissions be the union of all roles, or scoped to the currently-active role? Early enterprise systems forced explicit role switching — the user clicked a dropdown, selected 'Inventory Incharge', and only then could access the Inventory module.
[Normal] Role switching creates friction. Pooja, mid-task in the Today module (Class Teacher context), wants to glance at Inventory stock levels. Forcing a role switch is a context-breaking interaction. The system should know from her role union that she has Inventory access and apply the correct scope automatically based on the module she's accessing.
[Normal] Options Considered
[List Paragraph] Explicit role switching: User selects active role from a dropdown. Only that role's permissions apply. Friction for multi-role users; impossible to access Inventory while in Class Teacher mode.
[List Paragraph] Union of all roles, broadest scope wins: All role permissions unioned; data scope is broadest among roles. Security risk — Inventory Incharge role's whole-school scope leaks into Class Teacher queries.
[List Paragraph] Union of permissions, scope-per-role-per-module: Permissions unioned across all roles. Data scope is determined per-module by finding all roles that grant access to that module and unioning their assignments.
[Normal] Decision
[Normal] Adopt Option 3: Context-Based Access. Permissions are unioned across all of the user's roles. Data scope is computed per-module at query time: for each module the user accesses, the system finds all roles granting access to that module, retrieves their assignments, and unions the scopes.
[Normal] Role switching exists only as UI convenience — a user can filter their dashboard to show 'only Class Teacher view' or 'only Inventory view', but this is a presentation choice, not a security boundary. The user always has their full role union's permissions, regardless of UI role switcher state.
[Normal] Rationale
[Normal] Context-Based Access is the natural model for multi-role users. Pooja accessing the Inventory module sees whole-school inventory because her Inventory Incharge role grants that scope. Pooja accessing the Today module sees only Jr.KG B because her Class Teacher role is scoped to that class. The system determines scope from the module context, not from an explicit role selection.
[Normal] Explicit role switching (Option 1) is a usability regression — it forces the user to think about 'which role am I in?' instead of just doing their work. Union-with-broadest-scope (Option 2) is a security regression — it leaks scope from one role into another. Context-Based Access avoids both pitfalls.
[Normal] This pattern matches how Microsoft Azure and AWS IAM handle multi-role users — permissions are unioned, scopes are determined by the resource being accessed, and role switching is a UI convenience only.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Multi-role users get seamless access without role switching.
[List Paragraph] Data scope is always correct per module — no scope leakage between roles.
[List Paragraph] UI role switcher is optional, not mandatory — users who never switch roles still get correct behavior.
[Heading 3] Negative / Trade-offs
[List Paragraph] Permission evaluation is more complex than single-role — mitigated by ADR-016 Effective Access Resolver.
[List Paragraph] Caching is essential because per-module scope computation is expensive — mitigated by ADR-021 Permission Caching.

The UI role switcher is a frontend-only filter that does not call the resolver again.

[Heading 2] ADR-011: Scope-Per-Role-Per-Module
[Normal] Context
[Normal] ADR-010 established that data scope is computed per-module, not globally. The remaining question is the exact algorithm: when a user holds multiple roles that all grant access to the same module, how are their scopes combined? Pooja holds Class Teacher (scope: Jr.KG B) and Activity Teacher (scope: Art + Music activities) — both roles grant access to the Observations module. When Pooja views Observations, what does she see?
[Normal] A naive implementation might pick the broadest scope (whole school), but that would leak Activity Teacher's broad activity scope into Class Teacher's class-scoped queries. A stricter implementation might require the user to pick which role's scope to use, but that reintroduces explicit role switching. The correct algorithm must union scopes from all relevant roles without leaking scope between roles.
[Normal] Options Considered
[List Paragraph] Broadest scope wins: Pick the broadest scope among all roles granting module access. Pooja sees all observations school-wide. Simple but insecure — scope leaks across roles.
[List Paragraph] Scope intersection: Find the intersection of scopes across roles. Pooja sees only observations that are both in Jr.KG B AND in Art/Music. Likely empty — too strict.
[List Paragraph] Scope union per role, then union across roles: For each role granting module access, gather its scope. Union all scopes across all relevant roles. Pooja sees Jr.KG B observations + Art observations + Music observations.
[Normal] Decision
[Normal] Adopt Option 3: Scope-Per-Role-Per-Module. For each module the user accesses, the system finds all roles that grant access to that module. For each such role, it retrieves the role's assignments (which classes, branches, activities, etc.). All these scopes are unioned to produce the effective scope for that module.
[Normal] Critical rule: scopes are unioned, but they are never 'promoted'. A class-scoped role does not become school-scoped just because another role has school scope. Pooja's Class Teacher scope (Jr.KG B) and Activity Teacher scope (Art + Music activities) are unioned — she sees observations from Jr.KG B, Art, and Music. She does not see observations from other classes or other activities.
[Normal] Rationale
[Normal] Scope-Per-Role-Per-Module is the only algorithm that respects both multi-role reality and security boundaries. Pooja, as Class Teacher for Jr.KG B and Activity Teacher for Art + Music, legitimately needs to see observations from all three contexts — she teaches in all three. But she does not need to see observations from Sr.KG A or from Sports class, because she has no role there.
[Normal] The 'never promote scope' rule prevents the most dangerous failure mode — a low-scope role gaining high-scope access because the user happens to also hold a high-scope role. This would violate the principle of least privilege and create audit nightmares (Pooja viewing Sr.KG A observations would be inexplicable from her role assignments).
[Normal] This algorithm is what enterprise document management systems (SharePoint, Box) use for permission inheritance — each role contributes its scope, scopes are unioned, but a role's scope never expands beyond what was explicitly assigned.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Multi-role users see exactly what they need — no more, no less.
[List Paragraph] No scope leakage between roles — security boundary is preserved.
[List Paragraph] Audit trail is clear: every observation Pooja sees can be traced to a specific role assignment.
[Heading 3] Negative / Trade-offs
[List Paragraph] Scope computation is more complex than naive broadest-scope — mitigated by caching (ADR-021).
[List Paragraph] Edge case: a role with 'whole school' scope (e.g., Inventory Incharge) effectively grants that scope for the Inventory module only, not for other modules — implementation must enforce per-module scope isolation.
[Normal] Implementation Notes
[Normal] The resolver algorithm iterates through the user's active roles, filters to those granting the requested module, retrieves each role's assignments, and unions the scopes. The result is cached per (user, module) pair in Redis with version-based invalidation.
[Normal] async function resolveModuleScope(userId: string, module: Module): Promise<Scope[]> {
[Normal]   const cacheKey = `user:${userId}:scope:${module}`;
[Normal]   const cached = await redis.get(cacheKey);
[Normal]   if (cached) return JSON.parse(cached);
[Normal]   // 1. Get user's active roles
[Normal]   const roles = await userRoleRepo.findActiveRoles(userId);
[Normal]   // 2. Filter to roles granting this module
[Normal]   const moduleRoles = roles.filter(r =>
[Normal]     r.permissions.some(p => p.module === module && p.can_view)
[Normal]   );
[Normal]   // 3. For each such role, get its assignments
[Normal]   const scopes: Scope[] = [];
[Normal]   for (const role of moduleRoles) {
[Normal]     const assignments = await assignmentRepo.findActiveByUserRole(userId, role.id);
[Normal]     for (const assignment of assignments) {
[Normal]       scopes.push(...assignment.scopes);
[Normal]     }
[Normal]   }
[Normal]   // 4. Cache and return
[Normal]   await redis.setex(cacheKey, 3600, JSON.stringify(scopes));
[Normal]   return scopes;
[Normal] }

[Heading 2] ADR-012: Permission UNION + Data Scope from Assignment UNION
[Normal] Context
[Normal] ADR-010 and ADR-011 established the high-level algorithm: permissions are unioned across roles, data scope is unioned per-module across roles. This ADR formalizes the exact formula and the explicit prohibition against 'broadest scope wins' as a fallback.
[Normal] The formula must be unambiguous because implementers under deadline pressure tend to simplify scope computation to 'broadest scope wins' (which is one line of code) instead of proper per-role scope union (which requires iterating assignments). This ADR makes the prohibition explicit so code reviews can reject broadest-scope shortcuts.
[Normal] Options Considered
[List Paragraph] Permission intersection, scope broadest: User can do X only if ALL roles grant X. Scope is broadest. Maximum security but blocks legitimate multi-role workflows.
[List Paragraph] Permission union, scope broadest: User can do X if ANY role grants X. Scope is broadest. Convenient but insecure — broadest scope leaks into narrow-scope queries.
[List Paragraph] Permission union, scope union per role per module: User can do X if ANY role grants X. Scope for module M is the union of (assignments of roles granting M). Correct balance of usability and security.
[Normal] Decision
[Normal] Adopt Option 3. The formal formula is:
[Normal] EffectiveAccess(user, module) = {
[Normal]   permissions: UNION(permissions(role) for role in user.roles if role grants module),
[Normal]   scopes: UNION(assignment.scopes for assignment in user.assignments where assignment.role grants module)
[Normal] }
[Normal] Explicit prohibition: 'Broadest scope wins' is FORBIDDEN as an implementation shortcut. Every module's scope must be computed by iterating the user's roles that grant that module and unioning their assignments.
[Normal] Rationale
[Normal] Permission union (Option 3) is correct because a user holding both Class Teacher and Coordinator roles should have the union of their permissions — Coordinator's 'approve leave' permission should not be blocked just because Class Teacher doesn't have it.
[Normal] Scope union per role per module (Option 3) is correct because scope is a property of the (role, module) combination, not of the user globally. Pooja's Inventory Incharge scope (whole school) applies only to the Inventory module — it does not leak into her Class Teacher queries in the Daily Operations module.
[Normal] The explicit prohibition against 'broadest scope wins' exists because that shortcut is the single most common implementation mistake in multi-role RBAC systems. It silently leaks scope across roles and produces audit results that cannot be explained from role assignments.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Correct behavior for all multi-role combinations — no scope leakage.
[List Paragraph] Audit trail is explainable — every accessible record traces to a specific (role, assignment) pair.
[List Paragraph] Formula is implementable in one repository method, testable in isolation, and cacheable.
[Heading 3] Negative / Trade-offs
[List Paragraph] More complex than broadest-scope shortcut — mitigated by Effective Access Resolver (ADR-016) centralizing the logic.
[List Paragraph] Performance cost of scope iteration — mitigated by Redis caching (ADR-021).
[Normal] Implementation Notes
[Normal] The Effective Access Resolver implements the formula as a single method. The method is the only place in the codebase where scope computation happens — all repositories call this method and apply the returned scope as a query filter.
[Normal] // THE formula — implemented once, called everywhere
[Normal] async function resolveEffectiveAccess(userId: string): Promise<EffectiveAccess> {
[Normal]   const roles = await userRoleRepo.findActiveRoles(userId);
[Normal]   const modules = new Map<Module, { permissions: Set<Action>, scopes: Scope[] }>();
[Normal]   for (const role of roles) {
[Normal]     const rolePerms = await rolePermissionRepo.findByRole(role.id);
[Normal]     for (const perm of rolePerms) {
[Normal]       if (!perm.can_view) continue;  // skip modules without view permission
[Normal]       const module = perm.module;
[Normal]       if (!modules.has(module)) {
[Normal]         modules.set(module, { permissions: new Set(), scopes: [] });
[Normal]       }
[Normal]       const entry = modules.get(module)!;
[Normal]       // Permission UNION
[Normal]       if (perm.can_view)    entry.permissions.add('view');
[Normal]       if (perm.can_create)  entry.permissions.add('create');
[Normal]       if (perm.can_edit)    entry.permissions.add('edit');
[Normal]       if (perm.can_delete)  entry.permissions.add('delete');
[Normal]       if (perm.can_approve) entry.permissions.add('approve');
[Normal]       if (perm.can_export)  entry.permissions.add('export');
[Normal]       // Scope UNION (only scopes from THIS role's assignments)
[Normal]       const assignments = await assignmentRepo.findActive(userId, role.id);
[Normal]       for (const assignment of assignments) {
[Normal]         entry.scopes.push(...assignment.scopes);
[Normal]       }
[Normal]     }
[Normal]   }
[Normal]   return { userId, version: await getVersion(), modules: Object.fromEntries(modules) };
[Normal] }
[Normal] // ❌ FORBIDDEN — broadest scope shortcut
[Normal] function broadestScopeShortcut(roles: Role[]): Scope {
[Normal]   // This is FORBIDDEN by ADR-012. Code review must reject this pattern.
[Normal]   return roles.some(r => r.scope === 'school') ? { type: 'school' } : { type: 'class' };
[Normal] }

[Heading 2] ADR-013: 3-Level Permission Bundle (Business → Module → Action)
[Normal] Context
[Normal] PreOne has 13 modules × 6 actions × 11 roles = 858 permission cells in the role-module-permission matrix. Assigning permissions one cell at a time when creating a new role is tedious and error-prone. Enterprise RBAC systems solve this with permission bundles — pre-packaged permission sets that can be assigned to a role in one action.
[Normal] A flat bundle (one level) is too coarse — a 'Teacher Bundle' would either grant all Teacher permissions (over-permissive) or require fine-grained override (defeats the bundle purpose). A 3-level bundle (Business → Module → Action) provides the right balance: high-level bundles for quick role creation, mid-level for module-specific tuning, low-level for fine adjustments.
[Normal] Options Considered
[List Paragraph] Flat bundles (one level): Pre-package permission sets like 'Teacher Bundle', 'Finance Bundle'. Quick assignment but coarse — cannot tune within a bundle.
[List Paragraph] 2-level bundles (Module → Action): Bundle at module level. 'Fee Collection Bundle: View + Create + Edit'. Better, but still requires assembling many module bundles per role.
[List Paragraph] 3-level bundles (Business → Module → Action): Business domain bundles contain module bundles, which contain action references. 'Finance Bundle → Fee Collection Bundle → View, Create, Receipt'. Maximum reuse.
[Normal] Decision
[Normal] Adopt Option 3: 3-Level Permission Bundle. Level 1 is Business Bundle (Finance, Inventory, Academics, Communication). Level 2 is Module Bundle (Fee Collection, Invoices, Receipts). Level 3 is Action Bundle — references to reusable Action Library entries (View, Create, Edit, Delete, Approve, Export).
[Normal] Roles are assigned bundles, not individual permissions. A new role is created by selecting business bundles, then optionally trimming/extending at the module and action level. The 11 system roles (ADR-006) come pre-wired with their default bundle assignments.
[Normal] Rationale
[Normal] 3-level bundles match how administrators think about roles. A principal creating a new 'Office Manager' role thinks in terms of business areas (Finance, HR, Operations) first, then specific modules (Fee Collection, Payroll), then specific actions (View only, no Approve). The 3-level structure mirrors this thought process.
[Normal] The Action Library (ADR-014) provides the reusable Level 0 primitives that Level 3 references. Without this, every module would redefine 'View' and 'Create' actions, producing duplication. The Action Library centralizes these primitives.
[Normal] This pattern is used by AWS IAM (policies → statements → actions) and Microsoft Azure RBAC (role definitions → permission sets → actions). PreOne's 3-level structure is a simplified version of these enterprise patterns, tuned for preschool operations.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Role creation is fast — assign business bundles, trim at module level if needed.
[List Paragraph] Action definitions are reusable — View, Create, etc. defined once, referenced everywhere.
[List Paragraph] Permission matrix is auditable at any level — by business domain, by module, or by action.
[Heading 3] Negative / Trade-offs
[List Paragraph] 3 levels add conceptual complexity — mitigated by UI that hides levels until needed.
[List Paragraph] Bundle updates affect all roles using the bundle — mitigated by versioning (bundles are immutable; updates create new versions).
[Normal] Implementation Notes
[Normal] The bundle hierarchy is stored in three tables: business_bundle, module_bundle, and action_bundle. The action_bundle table references action_library entries (ADR-014). Roles are linked to module bundles via role_module_bundle, and business bundles are linked to module bundles via business_module_bundle.

[Heading 2] ADR-014: Reusable Action Library (Level 0)
[Normal] Context
[Normal] Without a reusable action library, every module redefines the same actions — Fee Collection has its own View/Create/Edit, Inventory has its own View/Create/Edit, Attendance has its own View/Create/Edit. This produces duplication, makes action-level auditing impossible (which 'View' action was used?), and complicates ABAC conditions (each 'View' would need its own condition logic).
[Normal] A reusable action library centralizes action definitions. 'View' is defined once, referenced by every module bundle that needs view permission. ABAC conditions (ADR-015) attach to the action reference, not to the action itself, enabling module-specific conditions on the same action primitive.
[Normal] Options Considered
[List Paragraph] Per-module action definitions: Each module defines its own View/Create/Edit/etc. Duplication, no shared ABAC, audit unclear.
[List Paragraph] Global action library, no per-module override: One View, one Create, etc. Clean but no way to add module-specific conditions.
[List Paragraph] Global action library with per-reference conditions: Actions defined once globally. Module bundles reference actions with optional condition_json for module-specific rules.
[Normal] Decision
[Normal] Adopt Option 3: Reusable Action Library at Level 0. The action_library table contains 10 primitive actions: View, Create, Edit, Delete, Approve, Reject, Export, Import, Print, Share. Module bundles reference these actions via action_bundle (ADR-013). Per-reference condition_json enables module-specific ABAC rules on the same primitive.
[Normal] Rationale
[Normal] Action reuse eliminates duplication and makes audit clean. When the audit log shows 'Pooja used View action on observation X', the action primitive is unambiguous — there is only one View action in the system.
[Normal] Per-reference condition_json enables future ABAC without bloating the action library. Fee Collection's 'View' reference can have condition {maxAmount: 5000} while Inventory's 'View' reference has condition {branchOnly: true} — same primitive, different rules.
[Normal] The 10 primitive actions cover all CRUD + workflow + I/O operations needed by PreOne. New actions (e.g., 'Archive', 'Restore') can be added to the library as needed, but the 10 are sufficient for v1.0.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Action definitions centralized — no duplication.
[List Paragraph] Audit log references unambiguous action primitives.
[List Paragraph] ABAC conditions attach to action references, enabling module-specific rules without forking the action library.
[Heading 3] Negative / Trade-offs
[List Paragraph] Adding a new action requires updating the action library, not just a module — minor friction, prevents accidental action proliferation.
[Normal] Implementation Notes
[Normal] The action_library table is seeded with 10 primitives on platform bootstrap. Module bundles reference actions via action_bundle rows, which carry optional condition_json for ABAC-lite.

[Heading 2] ADR-015: ABAC-Ready Schema (Condition JSON, Future)
[Normal] Context
[Normal] Attribute-Based Access Control (ABAC) extends RBAC with conditional rules — 'user can approve fees up to ₹5,000 only', 'user can edit daily sheet only on the same day', 'user can view medical records only for students in their assigned class'. PreOne v1.0 does not implement ABAC — RBAC with scope-per-role-per-module (ADR-011) covers v1.0 requirements.
[Normal] However, retrofitting ABAC later requires schema changes that are painful in production. The schema must reserve space for ABAC conditions now, even if the rules engine is not built. This ADR defines the schema reservation; the rules engine itself is deferred to a future version.
[Normal] Options Considered
[List Paragraph] Pure RBAC for v1.0, add ABAC columns later: Schema migration in production to add condition columns. Risky and disruptive.
[List Paragraph] ABAC columns now, no rules engine: Schema has condition_json columns, but no rules engine consumes them. Field is nullable, ignored by v1.0 authorization logic.
[List Paragraph] ABAC columns now + minimal rules engine for v1.0: Schema has condition_json, simple JSON-based rules engine for the most critical conditions. Scope creep risk.
[Normal] Decision
[Normal] Adopt Option 2: ABAC-ready schema with condition_json columns on action_bundle (ADR-014), but no rules engine in v1.0. [Normal] Conditions are stored as JSONB for query flexibility. The schema is documented so future implementers know the intended structure, but the v1.0 resolver treats condition_json as opaque.
[Normal] Rationale
[Normal] Retrofitting schema changes in a multi-tenant SaaS is risky — every tenant's data must be migrated, and downtime is unacceptable. Reserving the column now costs nothing (NULL storage is negligible) and saves a future migration.
[Normal] Option 3 (minimal rules engine for v1.0) introduces scope creep. The conditions that v1.0 actually needs (amount limits, same-day editing) can be hardcoded in service layer for now, with a clean migration path to ABAC when the rules engine is built.
[Normal] The JSONB format allows flexible condition structures without schema changes. Conditions like {maxAmount: 5000}, {sameDayOnly: true}, {branchOnly: true} can all coexist in the same column.
[Normal] Consequences

[Heading 2] ADR-016: Effective Access Resolver (Single Service)
[Normal] Context
[Normal] Authorization logic — resolving what a user can do and see — is a cross-cutting concern that touches every module. If each module implements its own authorization checks, the logic becomes scattered, inconsistent, and impossible to audit. A bug in the Fee module's scope filter might leak data, while the Inventory module's correct filter goes unnoticed.
[Normal] Centralizing authorization in a single service solves this. Every module calls the same resolver, gets the same shaped response, and applies it the same way. Bugs are fixed in one place. New authorization rules (ABAC, time-based access) are added in one place. Auditing can point to one service as the source of truth for 'what was this user allowed to do at time T?'.
[Normal] Options Considered
[List Paragraph] Distributed authorization (each module implements its own): Maximum autonomy, minimum consistency. Authorization bugs scattered across 13 modules.
[List Paragraph] Centralized resolver, called per request: Single service, called on every request. Consistent but expensive if resolver is slow.
[List Paragraph] Centralized resolver with Redis cache: Single service, output cached in Redis. Consistent AND fast. Cache invalidation handled by ADR-032.
[Normal] Decision
[Normal] Adopt Option 3: Centralized Effective Access Resolver with Redis cache. The resolver is the single source of truth for authorization. Every module calls resolver.resolve(userId) at request entry, gets an EffectiveAccess object (permissions + scopes per module), and applies it as query filters.
[Normal] The resolver output is cached in Redis with version-based invalidation (ADR-022). Cache misses trigger a rebuild, which is the only expensive operation. Subsequent requests within the cache window (1 hour default, version-invalidated on changes) read from Redis in ~1ms.
[Normal] Rationale
[Normal] Centralization is non-negotiable for authorization. Scattered authorization logic is the #1 cause of security bugs in enterprise systems — a bug in one module's filter can leak data while other modules are correctly filtered. Single-resolver eliminates this class of bugs.
[Normal] Redis cache solves the performance concern. Without cache, calling the resolver on every request would add 50-100ms of database queries. With cache, hot-path requests pay ~1ms. Cache invalidation (ADR-032) ensures correctness when roles or assignments change.
[Normal] This pattern is used by AWS IAM (centralized policy evaluation), Microsoft Azure RBAC (centralized role assignment evaluation), and Okta (centralized identity provider). PreOne's resolver is a simplified version of these enterprise patterns.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Single source of authorization truth — bugs fixed in one place.
[List Paragraph] Auditable — every authorization decision traces to resolver call with timestamp.
[List Paragraph] Fast — Redis cache makes hot-path ~1ms.
[List Paragraph] Future ABAC rules engine plugs into resolver without touching modules.
[Heading 3] Negative / Trade-offs
[List Paragraph] Resolver is a single point of failure — mitigated by Redis LKG snapshot (ADR-031).
[List Paragraph] Cache invalidation complexity — mitigated by ADR-032 event-driven invalidation.
[Normal] Implementation Notes
[Normal] The resolver is a NestJS provider injected into every controller via guard. The guard calls resolver.resolve(userId), attaches the result to the request object, and downstream handlers apply the scopes as query filters.
[Normal] @Injectable()
[Normal] export class AccessResolver {
[Normal]   constructor(private redis: RedisService, private db: DatabaseService) {}
[Normal]   async resolve(userId: string): Promise<EffectiveAccess> {
[Normal]     const cacheKey = `user:${userId}:access`;
[Normal]     const version = await this.getCurrentVersion();
[Normal]     // Check cache
[Normal]     const cached = await this.redis.get(cacheKey);
[Normal]     if (cached) {
[Normal]       const access = JSON.parse(cached);
[Normal]       if (access.version === version) return access;  // cache hit, version match
[Normal]     }
[Normal]     // Cache miss or version mismatch — rebuild
[Normal]     const access = await this.buildEffectiveAccess(userId, version);
[Normal]     await this.redis.setex(cacheKey, 3600, JSON.stringify(access));
[Normal]     return access;
[Normal]   }
[Normal]   private async buildEffectiveAccess(userId: string, version: number): Promise<EffectiveAccess> {
[Normal]     // Implements ADR-012 formula:
[Normal]     // permissions = UNION across roles
[Normal]     // scopes = UNION per (role, module) of assignments
[Normal]     const roles = await this.userRoleRepo.findActive(userId);
[Normal]     const modules = new Map<Module, ModuleAccess>();
[Normal]     for (const role of roles) {
[Normal]       const perms = await this.rolePermissionRepo.findByRole(role.id);
[Normal]       for (const perm of perms) {
[Normal]         if (!perm.can_view) continue;
[Normal]         if (!modules.has(perm.module)) {
[Normal]           modules.set(perm.module, { permissions: new Set(), scopes: [] });
[Normal]         }
[Normal]         const entry = modules.get(perm.module)!;
[Normal]         this.mergePermissions(entry.permissions, perm);
[Normal]         const assignments = await this.assignmentRepo.findActive(userId, role.id);
[Normal]         for (const a of assignments) entry.scopes.push(...a.scopes);
[Normal]       }
[Normal]     }
[Normal]     return { userId, version, modules: Object.fromEntries(modules) };
[Normal]   }
[Normal] }

[Heading 1] Domain 4: Assignment
[Normal] This domain defines how users are linked to scopes: hybrid assignment model (master table + scope-specific child tables), whole-school scope convention (no child rows = global), temporal validity via valid_from/valid_to with status for soft states only, and overlapping assignments allowed with UI warning.
[Heading 2] ADR-017: Hybrid Assignment Model (Master + Scope Tables)
[Normal] Context
[Normal] Assignments link users to scopes — Pooja is Class Teacher for Jr.KG B, Activity Teacher for Art and Music, Inventory Incharge for the whole school. The assignment data model must support multiple scope types (branch, class, section, activity, transport route, shift) without becoming a polymorphic nightmare.
[Normal] A polymorphic assignment table (one row per assignment with scope_type and scope_id columns) loses referential integrity — scope_id cannot have a foreign key because it points to different tables. A separate table per scope type (UserBranchAssignment, UserClassAssignment, etc.) preserves referential integrity but proliferates tables (6+ scope types = 6+ tables).
[Normal] Options Considered
[List Paragraph] Polymorphic single table: assignment table with scope_type + scope_id. No referential integrity on scope_id. Simple schema, weak data integrity.
[List Paragraph] Separate table per scope type: user_branch_assignment, user_class_assignment, user_activity_assignment, etc. Strong referential integrity, but 6+ tables and union queries for cross-scope views.
[List Paragraph] Hybrid: assignment master + scope-specific child tables: assignment table holds common fields (user_id, role_id, valid_from, valid_to). Scope-specific child tables (assignment_branch, assignment_class, etc.) hold the scope reference with proper FK. Clean separation, strong integrity, queryable.
[Normal] Decision
[Normal] Adopt Option 3: Hybrid Assignment Model. The assignment table is the master record with common fields (id, user_id, role_id, assignment_type, valid_from, valid_to, status). Six scope-specific child tables hold the actual scope reference with proper foreign keys: assignment_branch, assignment_class, assignment_section, assignment_activity, assignment_transport_route, assignment_shift.
[Normal] An assignment with no child rows means 'global scope' (ADR-018). An assignment with child rows means 'scoped to those specific entities'. This pattern preserves referential integrity, supports clean querying, and allows new scope types to be added by creating a new child table.
[Normal] Rationale
[Normal] Hybrid model gives the best of both worlds: the master table provides a single query point for 'what assignments does this user have?', while child tables provide referential integrity for scope references. Polymorphic single-table sacrifices integrity; per-scope-type tables sacrifice query simplicity.
[Normal] The pattern is well-established in enterprise schema design — it's how Rails' polymorphic associations should have been designed, and how Django's contenttypes framework handles generic relations with integrity. PreOne's hybrid model is a clean implementation of this pattern.
[Normal] Adding a new scope type (e.g., 'department' in a future HR expansion) requires only creating a new assignment_department child table — no changes to the master table or existing child tables. This extensibility is critical for a SaaS product that will evolve over years.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Referential integrity preserved — every scope reference has a proper FK.
[List Paragraph] Single query point for user's assignments (master table).
[List Paragraph] New scope types added by creating new child tables — no schema changes to existing tables.
[List Paragraph] Clean DDD aggregate boundary — assignment is an aggregate root, child tables are within the aggregate.
[Heading 3] Negative / Trade-offs
[List Paragraph] Six child tables (initially) — manageable, and each is small (3-4 columns).
[List Paragraph] Cross-scope queries require joining master + specific child table — mitigated by repository pattern that abstracts the join.
[Normal] Implementation Notes
[Normal] The assignment master table holds common fields. Each child table has assignment_id (FK to assignment), the scope-specific ID (FK to the scoped entity), and a unique constraint to prevent duplicate scope assignments.

[Heading 2] ADR-018: Whole School Scope (No Rows = Global)
[Normal] Context
[Normal] Some assignments are school-wide — Pooja as Inventory Incharge has scope over the entire school's inventory, not just one branch or class. The schema must represent 'global scope' in a way that is unambiguous, queryable, and does not require inserting dummy rows for every branch.
[Normal] Three options exist: (1) insert a row in every assignment_branch for the school's branches (data duplication, breaks when new branches added), (2) explicit is_global_scope boolean on assignment (clear but adds a column that is mutually exclusive with child rows), (3) no child rows = global scope (simplest, relies on convention).
[Normal] Options Considered
[List Paragraph] Explicit rows for all branches: Insert assignment_branch row for every branch in the school. Breaks when new branches are added — must backfill. Storage waste.
[List Paragraph] is_global_scope boolean on assignment: Boolean flag. Clear, but must enforce mutual exclusivity (if true, no child rows allowed) and adds a column.
[List Paragraph] No child rows = global scope (convention): An assignment with no rows in any child table means 'applies to entire school'. Simplest schema, clear convention, no extra columns.
[Normal] Decision
[Normal] Adopt Option 3: No child rows = global scope. An assignment with zero rows in assignment_branch, assignment_class, assignment_section, assignment_activity, assignment_transport_route, and assignment_shift is interpreted as 'whole school scope'. This is documented in the schema and enforced by the repository layer.
[Normal] The repository's resolveScopes(assignmentId) method returns [{type: 'school', id: assignment.school_id}] when no child rows exist, and the specific scope rows when they do. This convention is consistent across all 6 scope types.
[Normal] Rationale
[Normal] Convention over configuration. Option 3 is the simplest schema — no extra columns, no mutual exclusivity constraints to enforce, no backfill when new branches are added. The convention is documented and the repository abstracts it, so application code never sees the difference.
[Normal] Option 2 (is_global_scope boolean) is tempting but adds a column that must be kept in sync with child rows. If is_global_scope=true but a child row exists, the data is inconsistent. Option 3 has no such inconsistency risk.
[Normal] Option 1 (explicit rows for all branches) is the worst — it duplicates data, breaks when branches are added, and wastes storage. It also makes 'global scope' queries expensive (must check all branches are present).
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Simplest schema — no extra columns, no constraints to enforce.
[List Paragraph] Global scope is automatically updated when new branches/classes are added — no backfill needed.
[List Paragraph] Clear convention documented in schema comments and repository methods.
[Heading 3] Negative / Trade-offs
[List Paragraph] Convention must be documented clearly — mitigated by schema comments, repository method names, and this ADR.
[List Paragraph] Risk of confusion if a developer doesn't know the convention — mitigated by code review checklist item.
[Normal] Implementation Notes
[Normal] The resolveScopes method in the AssignmentRepository enforces the convention. It returns the appropriate scope based on the presence or absence of child rows.
[Normal] async function resolveScopes(assignmentId: string): Promise<Scope[]> {
[Normal]   const assignment = await this.findById(assignmentId);
[Normal]   // Check each child table for this assignment
[Normal]   const [branches, classes, sections, activities, routes, shifts] = await Promise.all([
[Normal]     this.assignmentBranchRepo.findByAssignment(assignmentId),
[Normal]     this.assignmentClassRepo.findByAssignment(assignmentId),
[Normal]     this.assignmentSectionRepo.findByAssignment(assignmentId),
[Normal]     this.assignmentActivityRepo.findByAssignment(assignmentId),
[Normal]     this.assignmentTransportRepo.findByAssignment(assignmentId),
[Normal]     this.assignmentShiftRepo.findByAssignment(assignmentId),
[Normal]   ]);
[Normal]   const allChildRows = [...branches, ...classes, ...sections, ...activities, ...routes, ...shifts];
[Normal]   // CONVENTION: no child rows = GLOBAL scope (whole school)
[Normal]   if (allChildRows.length === 0) {
[Normal]     return [{ type: 'school', id: assignment.school_id }];
[Normal]   }
[Normal]   // Build scope list from child rows
[Normal]   const scopes: Scope[] = [];
[Normal]   for (const b of branches)   scopes.push({ type: 'branch', id: b.branch_id });

[Normal]   for (const s of sections)   scopes.push({ type: 'section', id: s.section_id });
[Normal]   for (const a of activities) scopes.push({ type: 'activity', id: a.activity_id });
[Normal]   for (const r of routes)     scopes.push({ type: 'route', id: r.route_id });
[Normal]   for (const s of shifts)     scopes.push({ type: 'shift', id: s.shift_id });

[Heading 2] ADR-019: Assignment Temporal Validity (ValidFrom/ValidTo, No Status)
[Normal] Context
[Normal] Assignments change over time. Pooja was Class Teacher for Jr.KG A from January to June 2025, then transferred to Jr.KG B from July 2025 onwards. The schema must support this temporal validity for two reasons: (1) historical audit (who taught Jr.KG A in March 2025?), (2) automatic scope resolution at any point in time.
[Normal] An initial proposal suggested a 'status' field (ACTIVE/INACTIVE/EXPIRED) on assignments. On review, status duplicates information already encoded in valid_from/valid_to — an assignment with valid_to < NOW() is expired, regardless of what status says. Status introduces a consistency risk (valid_to says expired, status says active).
[Normal] Options Considered
[List Paragraph] Status field only (ACTIVE/INACTIVE/EXPIRED): Single column, but requires manual status updates when assignments expire. Forgets historical validity.
[List Paragraph] Status + temporal fields: Both status and valid_from/valid_to. Redundant — status duplicates temporal info. Consistency risk.
[List Paragraph] Temporal validity only (valid_from/valid_to), status for soft states: valid_from/valid_to determine active period. status only captures soft states (SUSPENDED, DELETED) that temporal cannot represent.
[Normal] Decision
[Normal] Adopt Option 3: Temporal validity (valid_from, valid_to) as the primary active-state indicator. status is used only for soft states — SUSPENDED (admin paused the assignment) and DELETED (soft-deleted per ADR-034). An assignment is 'active' if valid_from <= NOW() AND (valid_to IS NULL OR valid_to >= NOW()) AND status = 'ACTIVE'.
[Normal] Historical assignments (valid_to < NOW()) are retained for audit, not deleted. When Pooja transfers from Jr.KG A to Jr.KG B, the system sets valid_to on the Jr.KG A assignment to yesterday and creates a new Jr.KG B assignment with valid_from = today.
[Normal] Rationale
[Normal] Temporal validity is the natural representation of time-bound assignments. Status fields that duplicate temporal info create consistency risks — if valid_to says expired but status says active, which is authoritative? Encoding active state purely from temporal fields eliminates this ambiguity.
[Normal] Status for soft states only (SUSPENDED, DELETED) is necessary because temporal fields cannot represent these. An admin can suspend an assignment (Pooja on leave for 2 weeks) without setting valid_to — the suspension ends and the assignment resumes. DELETED is for soft-delete per ADR-034.
[Normal] This pattern is called 'bi-temporal' or 'valid time' in database literature and is used by SAP, Workday, and other enterprise HR systems for employment history. PreOne applies the same pattern to assignments at smaller scale.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Single source of truth for active state — temporal fields, not redundant status.
[List Paragraph] Historical assignments retained for audit — no deletion of expired rows.
[List Paragraph] Transfers are non-destructive — old assignment gets valid_to, new assignment created.
[List Paragraph] Point-in-time queries supported — 'what were Pooja's assignments in March 2025?'
[Heading 3] Negative / Trade-offs
[List Paragraph] Active query must include temporal filter — mitigated by repository method findActive() that abstracts the filter.
[List Paragraph] Slightly more storage (expired rows retained) — mitigated by partitioning assignment table by year if it grows large.
[Normal] Implementation Notes

[Heading 2] ADR-020: Allow Overlapping Assignments with UI Warning
[Normal] Context
[Normal] A unique constraint on (user_id, role_id, scope) would prevent Pooja from being Class Teacher for both Jr.KG A and Jr.KG B simultaneously. But in small preschools, one teacher handling two sections is normal — team teaching, split sections, or covering for an absent colleague.
[Normal] On the other hand, accidental duplicate assignments (admin clicks 'assign' twice, or two admins assign the same teacher to the same class) produce confusing data and audit noise. The system should allow legitimate overlaps but warn about suspicious ones.
[Normal] Options Considered
[List Paragraph] Unique constraint, no overlaps allowed: Database-enforced uniqueness on (user_id, role_id, scope). Blocks legitimate multi-section assignments.
[List Paragraph] No constraint, no warning: Free-for-all. Allows duplicates but provides no protection against accidental double-assignment.
[List Paragraph] No constraint, UI warning on suspicious overlap: Database allows overlaps. UI warns admin when creating an assignment that overlaps an existing one. Admin confirms to proceed.
[Normal] Decision
[Normal] Adopt Option 3: No database unique constraint. UI shows a warning when an admin creates an assignment that overlaps an existing active assignment for the same user+role. Admin clicks 'Confirm' to proceed, and the overlap is logged for audit.
[Normal] The warning message is: 'This teacher already has 2 active class assignments. Continuing will add a 3rd. Confirm to proceed.' This makes overlaps deliberate, not accidental.
[Normal] Rationale
[Normal] Preschool reality requires overlap support. A 5-staff preschool cannot afford to assign one teacher per section — teachers must cover multiple sections. Blocking overlaps at the database level would force schools to create fake 'Combined Teacher' roles or other workarounds.
[Normal] UI warning catches accidental duplicates without blocking legitimate overlaps. The admin sees the existing assignments and can decide: 'yes, Pooja is covering for Neha who is on leave' vs 'oops, I already assigned her to Jr.KG A, this should be Jr.KG B'.
[Normal] Audit log records the confirmation, so if an overlap turns out to be a mistake, the audit trail shows who confirmed it and when. This is more accountable than a silent unique constraint that blocks legitimate work.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Multi-section assignments supported — matches preschool reality.
[List Paragraph] Accidental duplicates caught by UI warning — not silent.

[Heading 1] Domain 5: Permission Caching
[Normal] This domain defines the caching strategy for the Effective Access Resolver: mandatory Redis caching, version-based invalidation over TTL, and separate menu cache key for thin navigation API responses.
[Heading 2] ADR-021: Permission Caching Mandatory (Redis)
[Normal] Context
[Normal] The Effective Access Resolver (ADR-016) computes a user's permissions and scopes by joining user_roles, role_module_permission, and assignment tables. Without caching, every HTTP request triggers this computation — 5-10 database queries, 50-100ms latency. At 1000 concurrent users, this is 50,000-100,000 queries per second, overwhelming the database.
[Normal] Caching is not optional. The question is not 'should we cache?' but 'how do we cache correctly?' — preserving correctness when roles or assignments change, while keeping latency under 5ms for hot-path requests.
[Normal] Options Considered
[List Paragraph] In-memory cache (per-process): Each API process caches in its own memory. Fast but inconsistent across processes — role change on process A doesn't invalidate process B's cache.
[List Paragraph] Database-backed cache: Cache in a PostgreSQL table. Consistent but slow — defeats the purpose of caching.
[List Paragraph] Redis cache with version-based invalidation: Centralized cache, fast (~1ms), version-based invalidation (ADR-022) for correctness. Industry standard for multi-process caching.
[Normal] Decision
[Normal] Adopt Option 3: Redis cache with version-based invalidation. Every user's EffectiveAccess is cached in Redis under key user:{id}:access. Cache TTL is 1 hour (fallback expiry), but version-based invalidation (ADR-022) ensures changes are reflected within seconds, not hours.
[Normal] Cache structure: three keys per user — user:{id}:access (the EffectiveAccess JSON), user:{id}:menu (the UI navigation menu derived from access), user:{id}:scope-map (the per-module scope map for fast query filtering).
[Normal] Rationale
[Normal] Redis is the industry standard for distributed caching. It is fast (sub-millisecond reads), supports data structures (not just strings), and has built-in pub/sub for invalidation events. PreOne's deployment already includes Redis for session storage, so no new infrastructure is needed.
[Normal] In-memory per-process cache (Option 1) breaks consistency — a role change on process A leaves processes B, C, D with stale cache until their TTL expires. In a 10-process deployment, this means 90% of requests see stale data for up to 1 hour after a role change. Unacceptable for a security-critical system.
[Normal] Version-based invalidation (ADR-022) is the key to correctness. Each cache entry includes a version number; a separate current_version key tracks the global version. On role/assignment change, the version is bumped; cache entries with old versions are rebuilt on next access.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Hot-path permission checks: ~1ms (Redis read) instead of 50-100ms (DB queries).
[List Paragraph] Database load reduced by 95%+ for authorization queries.
[List Paragraph] Centralized cache — all API processes see the same cache state.
[Heading 3] Negative / Trade-offs
[List Paragraph] Redis is a new failure point — mitigated by ADR-031 (LKG snapshot for resilience).
[List Paragraph] Cache invalidation complexity — mitigated by ADR-032 (event-driven invalidation).
[Normal] Implementation Notes
[Normal] Three Redis keys per user store the cached access data. The resolver checks cache first, falls back to DB on miss or version mismatch, and writes the rebuilt cache back. TTL is 1 hour as a safety net; version-based invalidation is the primary correctness mechanism.
[Normal] // Redis key structure
[Normal] user:{userId}:access     → EffectiveAccess JSON (permissions + scopes per module)
[Normal] user:{userId}:menu       → UI navigation menu (derived from access)
[Normal] user:{userId}:scope-map  → Per-module scope map (for fast query filtering)
[Normal] global:permission-version → current version number (incremented on any RBAC change)
[Normal] // Resolver with cache
[Normal] async function resolve(userId: string): Promise<EffectiveAccess> {
[Normal]   const currentVersion = await redis.get('global:permission-version') || '1';
[Normal]   const cacheKey = `user:${userId}:access`;
[Normal]   const cached = await redis.get(cacheKey);
[Normal]   if (cached) {
[Normal]     const access = JSON.parse(cached);
[Normal]     if (access.version === parseInt(currentVersion)) {
[Normal]       return access;  // cache hit, version match
[Normal]     }

[Heading 2] ADR-022: Cache Versioning Over TTL
[Normal] Context
[Normal] TTL-based cache invalidation (cache expires after 1 hour) has a fatal flaw for authorization: a user whose role is revoked at 10:00 AM still has access until 11:00 AM when their cache expires. For a security-critical system, this 1-hour window of stale permissions is unacceptable.
[Normal] Manual cache invalidation (delete user:{id}:access on role change) is correct but fragile — every place in code that changes a role, assignment, or permission must remember to invalidate the cache. Missing one invalidation leaves stale permissions indefinitely.
[Normal] Options Considered
[List Paragraph] TTL only (1 hour expiry): Simple, but stale permissions for up to 1 hour after role change. Unacceptable for security.
[List Paragraph] Manual invalidation (delete key on change): Correct, but fragile — every code path that changes RBAC must invalidate. Easy to forget.
[List Paragraph] Version-based invalidation: Global permission-version key incremented on any RBAC change. Cache entries store their version; mismatch triggers rebuild. Correct and centralized.
[Normal] Decision
[Normal] Adopt Option 3: Version-based invalidation. A global Redis key global:permission-version holds the current version number (integer, incremented atomically). Every cached EffectiveAccess entry stores its version. On cache read, if entry.version !== current_version, the cache is stale and rebuilt.
[Normal] Version is bumped on any RBAC-relevant change: role creation/update/delete, role_module_permission change, assignment creation/update/delete, user_role change, permission_bundle change. A single INCR command bumps the version; all cache entries become stale simultaneously.
[Normal] TTL (1 hour) remains as a safety net for edge cases (e.g., bugs that fail to bump version), but version mismatch is the primary invalidation trigger.
[Normal] Rationale
[Normal] Versioning centralizes invalidation logic. Instead of every RBAC-change code path needing to know which cache keys to invalidate, they all just call bumpPermissionVersion() — one line of code, impossible to forget. The version check at cache read time handles the rest.
[Normal] Version mismatch detection is O(1) — one Redis GET to read global:permission-version, compare to cached entry's version. If they match, return cached; if not, rebuild. This is fast enough to do on every request.
[Normal] This pattern is used by Facebook's TAO cache, Twitter's cache invalidation, and Shopify's permission cache. PreOne's versioning is a simplified single-version implementation; future versions may use per-user versioning for finer-grained invalidation.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Stale cache window: seconds (next request after version bump) instead of hours (TTL).
[List Paragraph] Centralized invalidation — one INCR command invalidates all stale caches.
[List Paragraph] TTL remains as safety net for edge cases.
[Heading 3] Negative / Trade-offs
[List Paragraph] Mass invalidation on any RBAC change (all users' caches become stale) — mitigated by ADR-032 staggered rebuild.
[List Paragraph] Version number is global, not per-user — over-invalidation (some users' caches rebuilt unnecessarily). Acceptable trade-off for simplicity.
[Normal] Implementation Notes
[Normal] The bumpPermissionVersion function is called by every RBAC-change code path. It atomically increments the global version key. The next cache read for any user detects the version mismatch and rebuilds.
[Normal] // Bump version on any RBAC change
[Normal] async function bumpPermissionVersion(): Promise<number> {
[Normal]   return await redis.incr('global:permission-version');
[Normal] }

[Normal] // - RoleService.create / update / delete
[Normal] // - RolePermissionService.assign / revoke
[Normal] // - AssignmentService.create / update / delete
[Normal] // - UserRoleService.assign / revoke
[Normal] // - PermissionBundleService.create / update / delete

[Normal]   await bumpPermissionVersion();  // ← one line, invalidates all caches
[Normal]   await eventBus.publish('permission.version_bumped', { roleId });

[Heading 2] ADR-023: Menu Cache Key
[Normal] Context
[Normal] The UI navigation menu is derived from the EffectiveAccess object — it lists modules where the user has at least View permission, with their display names, icons, and URLs. Computing this menu on every page load from the EffectiveAccess JSON is fast (in-memory transform), but it still adds 2-5ms per request and produces identical output for the same user until their permissions change.
[Normal] Caching the derived menu separately from the raw access JSON avoids redundant computation and enables a thinner API response (just the menu, not the full access object) for frontend navigation requests.
[Normal] Options Considered
[List Paragraph] Compute menu from access on every request: Simple, no separate cache. 2-5ms per request for the transform — minor but adds up at scale.
[List Paragraph] Single cache key for access + menu combined: One Redis key holds both. Frontend extracts menu from the combined object. Slightly larger payload, no separate cache management.
[List Paragraph] Separate menu cache key, derived from access: user:{id}:menu stores the derived menu. Invalidated same as access (version-based). Frontend can request just the menu for thin navigation API.
[Normal] Decision
[Normal] Adopt Option 3: Separate menu cache key. user:{id}:menu stores the derived navigation menu (array of {module, label, icon, url, children?}). Invalidated by the same version mechanism as access (ADR-022). Frontend can call GET /me/menu to get just the menu, avoiding the larger access payload for routine navigation requests.
[Normal] The menu is derived from access on cache miss, then stored. Subsequent requests return the cached menu directly. On version bump, both access and menu caches are rebuilt on next access.
[Normal] Rationale
[Normal] Separate menu cache enables a thin navigation API. Frontend calls GET /me/menu (1KB response) instead of GET /me/access (10KB response with full permission/scopes JSON) when it just needs to render navigation. This reduces API payload size by 90% for routine navigation requests.
[Normal] The menu is a derived view of access — same version invalidation applies, no separate invalidation logic needed. The cost is one additional Redis key per user (~1KB), negligible compared to the access key (~5KB).
[Normal] Option 1 (compute on every request) is fast enough for v1.0 but doesn't scale — at 1000 RPS, 2-5ms per request is 2-5 seconds of CPU per second. Caching eliminates this entirely.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Thin navigation API — GET /me/menu returns ~1KB vs GET /me/access returns ~10KB.
[List Paragraph] No redundant menu computation on every page load.
[List Paragraph] Same invalidation as access — no separate logic.
[Heading 3] Negative / Trade-offs
[List Paragraph] One additional Redis key per user (~1KB) — negligible.
[Normal] Implementation Notes

[Heading 1] Domain 6: Multi-Tenancy & System Roles
[Normal] This domain defines multi-tenant isolation: mandatory columns (school_id, branch_id, created_by, updated_by, soft delete) on every aggregate root, repository-level auto-injection of tenant_id with PostgreSQL RLS as defense-in-depth, and system roles (Super Admin, Support Engineer, Migration Tool) with explicit scope bypass and full audit logging.
[Heading 2] ADR-024: Multi-Tenancy Mandatory (tenant_id, school_id, CreatedBy, UpdatedBy)
[Normal] Context
[Normal] PreOne is a multi-tenant SaaS where each School is a tenant. ADR-001 established shared-database-with-tenant_id as the architecture. This ADR defines the mandatory columns that every aggregate root table must include to support tenant isolation, audit, and ownership tracking.
[Normal] Without mandatory columns, individual tables may be created without tenant_id (data leakage risk), without created_by/updated_by (no accountability for changes), or without created_at/updated_at (no change tracking). Standardizing these columns at the schema level prevents omissions.
[Normal] Options Considered
[List Paragraph] Optional columns (per-table decision): Each table decides which audit/tenant columns to include. Flexible but error-prone — tables missing tenant_id leak data.
[List Paragraph] Mandatory on aggregate roots only: Aggregate root tables must have all columns. Child/lookup tables may skip some. Balance of safety and minimalism.
[List Paragraph] Mandatory on every table: Even lookup tables and junction tables carry tenant_id. Maximum safety but redundant for tables that already have FKs implying tenant.
[Normal] Decision
[Normal] Adopt Option 2: Mandatory columns on every aggregate root table. The mandatory columns are: id (UUID primary key), school_id (tenant isolation), branch_id (NULL for school-level aggregates), created_by, updated_by (accountability), created_at, updated_at (change tracking), and is_deleted + deleted_at + deleted_by (soft delete per ADR-034).
[Normal] Child tables within an aggregate (e.g., assignment_branch within assignment) inherit tenant isolation via FK to the aggregate root — they don't need their own school_id. Junction tables (e.g., user_user_group) carry school_id because they don't have a single FK to an aggregate root.
[Normal] Rationale
[Normal] Mandatory columns on aggregate roots is the DDD-correct approach. Aggregate roots are the consistency boundaries; they own their data including audit metadata. Child entities inherit tenant from the root via FK, so duplicating school_id on them is redundant and risks inconsistency (root.school_id ≠ child.school_id if one is updated).
[Normal] created_by and updated_by are essential for accountability — when a fee invoice is modified, the audit trail must show who modified it. These columns plus the AuditLog table (ADR-035) provide complete change tracking.
[Normal] branch_id is nullable because some aggregates are school-level (School itself, User, Role) rather than branch-level (Student, Class, Attendance). NULL branch_id means 'applies to all branches of this school'.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Consistent schema across all aggregate roots — predictable for developers.
[List Paragraph] Tenant isolation enforced at column level — every query can filter by school_id.
[List Paragraph] Complete audit trail — who created, who updated, when, with what changes (via AuditLog).
[Heading 3] Negative / Trade-offs
[List Paragraph] Slightly more storage per row (~80 bytes for the mandatory columns) — negligible.
[List Paragraph] Developers must remember to include these columns in new tables — mitigated by schema migration linter that fails if missing.
[Normal] Implementation Notes
[Normal] A schema migration linter (CI check) verifies every new aggregate root table includes all mandatory columns. The BaseRepository pattern provides default implementations for tenant filtering, audit column population, and soft delete handling.

[Heading 2] ADR-025: Repository-Level Auto-Inject tenant_id
[Normal] Context
[Normal] ADR-024 mandates school_id on every aggregate root. But if developers write raw SQL with WHERE clauses that forget school_id, tenant isolation is broken. A single forgotten WHERE school_id = ? clause can leak data across tenants — a critical security bug.
[Normal] Manual enforcement (code review, linting) is insufficient because the bug only surfaces under specific query patterns. The enforcement must be at the repository layer — every query automatically includes school_id filtering, and developers cannot bypass it.
[Normal] Options Considered
[List Paragraph] Manual WHERE clauses (developer discipline): Developers write WHERE school_id = ? in every query. Code review catches misses. Fragile — one miss leaks data.
[List Paragraph] PostgreSQL Row-Level Security (RLS) only: Database-level enforcement. Strong but opaque — developers don't see the filter, debugging is harder.
[List Paragraph] Repository auto-inject + RLS as defense-in-depth: Repository layer adds WHERE school_id = ? automatically. RLS catches any query that bypasses the repository (raw SQL, migrations). Defense in depth.
[Normal] Decision
[Normal] Adopt Option 3: Repository auto-injects school_id on every query, with PostgreSQL RLS as defense-in-depth. The BaseRepository class accepts a tenant context (school_id from the request) and automatically appends WHERE school_id = :tenantId to every SELECT, UPDATE, DELETE query.
[Normal] Developers never write WHERE school_id = ? manually. The repository's find(), update(), delete() methods all apply the filter. Raw SQL is discouraged; if needed, it goes through a tenantContext.sql() helper that injects the filter and sets the RLS session variable.
[Normal] Rationale
[Normal] Repository auto-injection makes tenant isolation the default — developers have to actively bypass it to cause a leak. This is the opposite of manual enforcement, where developers have to actively remember to add the filter. The default-safe pattern is the foundation of secure multi-tenant design.
[Normal] RLS as defense-in-depth catches queries that bypass the repository — raw SQL in migrations, ad-hoc queries by admins, bugs in the repository itself. Even if a query forgets school_id, RLS rejects rows from other tenants. This is the safety net that prevents catastrophic data leakage.
[Normal] This pattern is used by Rails' ActsAsTenant gem, Django's tenant-schemas, and Microsoft's EF Core multi-tenancy patterns. PreOne's implementation follows the same principles in NestJS.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Tenant isolation is the default — impossible to forget in normal code paths.
[List Paragraph] RLS catches bypass attempts — defense in depth.
[List Paragraph] Developers don't need to think about tenant isolation — it just works.
[Heading 3] Negative / Trade-offs
[List Paragraph] Repository pattern is mandatory — no raw SQL without going through tenant context helper.
[List Paragraph] RLS adds slight query overhead (~5%) — acceptable for security.
[List Paragraph] Cross-tenant queries (Super Admin analytics) require explicit RLS bypass — mitigated by ADR-026 System Roles.
[Normal] Implementation Notes
[Normal] The BaseRepository class wraps every query with tenant filtering. The tenant context is set per-request from the authenticated user's school_id. RLS policies on PostgreSQL enforce isolation at the database level as a safety net.

[Heading 2] ADR-026: System Roles (Super Admin as System Role, Not Bypass)
[Normal] Context
[Normal] Super Admin (PreOne SaaS operator) needs cross-tenant access to manage all schools on the platform. The initial proposal was to bypass RBAC entirely for Super Admin — a single is_super_admin flag that skips all permission checks and scope filters.
[Normal] On review, full bypass is dangerous. It produces an unauditable god-mode account that can access any data without leaving a trace of what permissions were exercised. Compliance audits (SOC2, ISO27001, DPDP Act 2023) require explicit access controls even for administrators. A bypass flag is the wrong pattern.
[Normal] Options Considered
[List Paragraph] Full bypass (is_super_admin flag): Super Admin skips all RBAC. Fast and simple but unauditable, non-compliant, and a single point of catastrophic risk.
[List Paragraph] Super Admin as regular role with all permissions: Super Admin is a role with every permission granted. Auditable but permission matrix is huge and brittle.
[List Paragraph] System Roles with explicit scope bypass: Super Admin is a System Role (is_system_role=true) that bypasses scope filtering but goes through permission checks and audit logging. Explicit, auditable, compliant.
[Normal] Decision
[Normal] Adopt Option 3: System Roles. The role table has an is_system_role boolean. System Roles (Super Admin, Support Engineer, Migration Tool) bypass tenant isolation and scope filtering, but still go through permission checks (they have a defined permission set) and all actions are audit-logged with the system role identity.
[Normal] Super Admin's permission set is 'all modules, all actions' — but this is explicit in the database, not a bypass flag. The audit log records 'Super Admin accessed Student data on School A' with the system role identity, providing full accountability.
[Normal] Rationale
[Normal] System Roles make Super Admin's access explicit in the architecture. There is no hidden bypass flag — Super Admin's capabilities are visible in the role_module_permission table, auditable in the audit log, and revocable by changing the role's permissions. This is the compliant pattern for privileged access.
[Normal] Full bypass (Option 1) is the anti-pattern. It produces a god-mode account that compliance auditors flag immediately. Even if the bypass is logged, the log shows 'Super Admin did X' without explaining what permissions were exercised — making it impossible to detect misuse vs legitimate use.
[Normal] This pattern is used by AWS root account (which still has explicit IAM policies), Microsoft Azure Global Admin (which has defined role permissions), and Google Cloud Organization Admin. PreOne's System Roles follow the same principle at smaller scale.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Super Admin access is explicit and auditable — compliance-friendly.
[List Paragraph] System Roles can have restricted permissions (e.g., Support Engineer with read-only access).
[List Paragraph] Bypass is scoped to tenant isolation only — permissions still enforced.
[List Paragraph] Audit log records system role identity — full accountability.
[Heading 3] Negative / Trade-offs
[List Paragraph] Super Admin permission matrix must be maintained — mitigated by 'all permissions' bundle that auto-grants new modules.
[List Paragraph] Slight performance cost (permission check runs even for system roles) — negligible.
[Normal] Implementation Notes
[Normal] The role table has is_system_role column. The TenantContext (ADR-025) sets is_system_role from the user's primary role. The BaseRepository checks is_system_role — if true, skips school_id filter (system roles see all tenants) but still applies other filters. The AccessResolver (ADR-016) returns 'global scope' for system roles, bypassing per-module scope computation.

[Heading 2] ADR-027: Inventory as Separate Business Domain
[Normal] Context
[Normal] Most school ERPs treat inventory as a sub-module of Administration — a few tables for stationery, a basic stock register, and an afterthought in the admin portal. This design under-serves preschools where toys, Montessori materials, art supplies, kitchen supplies, and hygiene items are daily operational consumables that directly affect the quality of childcare.
[Normal] PreOne's v2 database schema (142 models) initially placed inventory under Administration. On review, the operational importance of preschool inventory justified promoting it to a first-class Business Domain, parallel to Finance, HR, and Academics.
[Normal] Options Considered
[List Paragraph] Keep inventory as Administration sub-module: Smaller scope, fewer tables. But under-serves preschool inventory needs and couples inventory changes with Administration changes.
[List Paragraph] Inventory as separate domain: Full business domain with purchase requests, POs, GRN, issue register, returns, damaged items, stock audit, low-stock alerts, and preschool-specific categories.
[List Paragraph] Inventory as separate domain + Activity integration: Separate domain PLUS integration with Teacher Portal's Curriculum and Activity modules, so activity planning can check material availability.
[Normal] Decision
[Normal] Adopt Option 3: Inventory as a separate Business Domain with Activity integration. The Inventory domain includes 15 models: InventoryCategory, InventoryItem, PurchaseRequest, PurchaseOrder, GRN, StockIssue, StockReturn, DamagedItem, StockAudit, StockAuditItem, LowStockAlert, Vendor, plus supporting lookup tables. Preschool-specific categories (Toys, Montessori Materials, Art & Craft, Sports Equipment, Kitchen Supplies, Hygiene Items, First Aid, Nap Mats, Smart Devices) are first-class.
[Normal] Integration with Activity module: when a teacher plans an activity (e.g., 'Clay Modeling'), the system checks inventory for clay availability and shows low-stock warnings. This connects material planning with activity execution.
[Normal] Rationale
[Normal] Preschool inventory is operationally critical. A preschool without art supplies, Montessori materials, or kitchen supplies cannot operate. Treating inventory as an afterthought produces stockouts, emergency purchases at premium prices, and operational disruptions. A first-class Inventory domain enables proactive procurement and consumption tracking.
[Normal] Preschool inventory categories differ from K-12 schools. Toys, Montessori materials, nap mats, hygiene items, and first aid are preschool-specific. Burying these in a generic 'stationery' category loses operational visibility — the principal cannot answer 'do we have enough Montessori materials for next month's enrollment?'
[Normal] Activity integration is the killer feature that justifies a separate domain. When the Curriculum module plans a 'Water Play' activity for Friday, the system checks inventory for water play materials (towels, toys, soap) and alerts the inventory manager to restock if needed. This closed-loop integration is impossible if inventory is a passive Administration sub-module.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Proactive procurement — low-stock alerts before stockouts.
[List Paragraph] Activity-inventory integration — material availability checked during activity planning.
[List Paragraph] Preschool-specific categories — operational visibility into toys, Montessori, art supplies, etc.
[List Paragraph] Independent release cadence — Inventory can iterate without Administration coordination.
[Heading 3] Negative / Trade-offs
[List Paragraph] More tables and code to maintain — mitigated by dedicated inventory team ownership.
[List Paragraph] Integration touchpoints with Curriculum and Activity modules — mitigated by domain events (ADR-036).
[Normal] Implementation Notes
[Normal] The Inventory domain is implemented as a NestJS module with 15 entities, separate from Administration. Preschool-specific categories are seeded as system categories. Integration with Curriculum module is via domain events — when an activity is planned, an event is published that the Inventory module consumes to check material availability.

[Heading 2] ADR-028: Modular Monolith Initially
[Normal] Context
[Normal] PreOne has 13 business domains, each a candidate for microservice extraction. The temptation is to start with microservices — each domain as a separate service with its own database. But microservices add operational complexity: service discovery, distributed tracing, inter-service authentication, distributed transactions, and deployment orchestration.
[Normal] For a startup engineering team of 5-10 developers, microservices on day 1 are over-engineering. The team would spend more time on infrastructure than on features. The alternative is a modular monolith — a single deployable unit with strong module boundaries that can be extracted to microservices later if scale demands.
[Normal] Options Considered
[List Paragraph] Microservices from day 1: 13 services, each with own database. Maximum isolation but massive operational overhead. Wrong for early-stage startup.
[List Paragraph] Modular monolith, never extract: Single deployable, strong module boundaries, no extraction plan. Simple but limits scaling options.
[List Paragraph] Modular monolith with extraction paths: Single deployable initially, but modules designed for clean extraction (independent schemas, domain events for inter-module communication). Extract to microservices only when scale demands.
[Normal] Decision
[Normal] Adopt Option 3: Modular Monolith with extraction paths. The application is a single NestJS deployable with 13 domain modules. Each module has its own schema namespace, its own service layer, and communicates with other modules only via domain events (ADR-036) — never via direct database access to another module's tables.
[Normal] Extraction paths are preserved: when a module's scale (traffic, team size, release cadence) justifies extraction, it can be pulled out as a microservice with minimal code changes. The domain event contracts remain the same; only the transport changes (in-process event bus → message broker).
[Normal] Rationale
[Normal] Microservices are an architectural response to organizational scale (Conway's Law), not technical scale. A 5-developer team cannot effectively own 13 services — they will spend all time on coordination and none on features. Modular monolith lets the team build the product first, extract services when team growth justifies it.
[Normal] Extraction paths prevent the common failure mode of 'monolith that cannot be split'. By enforcing module boundaries (separate schemas, domain events, no cross-module repository calls), the monolith remains extractable. When Finance grows to need its own team and deployment cadence, it can be extracted in a focused migration, not a rewrite.
[Normal] This pattern is used by Shopify (modular monolith with Dequeue for inter-module), GitHub (Rails monolith with clear module boundaries), and Basecamp (single Rails app). All started as monoliths and selectively extracted services as they scaled. PreOne follows the same path.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Single deployment — simple CI/CD, no orchestration overhead.
[List Paragraph] Single database — no distributed transactions, no saga complexity.
[List Paragraph] Module boundaries preserve extraction options — no rewrite when scaling.
[List Paragraph] Small team can ship features fast — no service coordination tax.
[Heading 3] Negative / Trade-offs
[List Paragraph] Single point of failure — one module's crash can take down all. Mitigated by process isolation for critical modules (e.g., background jobs separate from API).
[List Paragraph] Scaling is all-or-nothing — cannot scale Finance independently of Settings. Mitigated by extracting when needed.
[List Paragraph] Module boundaries must be enforced by discipline, not the runtime. Mitigated by CI tests that ban cross-module repository imports.
[Normal] Implementation Notes
[Normal] Each domain is a NestJS module with its own controllers, services, repositories, and entity definitions. Modules export only services (never repositories) — other modules call exported services, which internally use repositories. Domain events (ADR-036) handle cross-module workflows.

[Heading 2] ADR-029: DDD (Domain-Driven Design)
[Normal] Context
[Normal] PreOne's 13 business domains each have complex business logic — fee calculation rules, admission workflow states, attendance policies, inventory reorder logic. Without a structured approach to modeling these domains, the codebase becomes a tangle of services calling services, with business logic scattered across controllers, services, and repositories.
[Normal] Domain-Driven Design (DDD) provides a structured approach: each domain is a Bounded Context with its own Ubiquitous Language (shared vocabulary between domain experts and code), Aggregates (consistency boundaries), Entities (with identity), Value Objects (without identity), and Domain Events (facts about domain state changes).
[Normal] Options Considered
[List Paragraph] Anemic domain model (CRUD-focused): Services contain all logic; entities are just data bags. Simple but scatters business logic across services.
[List Paragraph] Rich domain model (full DDD): Entities contain business logic; aggregates enforce consistency; value objects for concepts like Money, Address. More upfront design but cleaner long-term.
[List Paragraph] Pragmatic DDD (rich where it matters): Rich domain model for complex domains (Finance, Admission, Inventory); anemic for simple CRUD domains (Settings, Reports). Balance of design effort and clarity.
[Normal] Decision
[Normal] Adopt Option 3: Pragmatic DDD. Complex domains (Finance, Admission, Student Lifecycle, Inventory, Daily Operations) get rich domain models with aggregates, value objects, and domain events. Simple domains (Settings, Reports, Platform Management) use anemic models with CRUD services.
[Normal] Ubiquitous Language is enforced: code uses the same terms as domain experts. A 'Fee Invoice' is called FeeInvoice in code, not 'BillingRecord' or 'PaymentEntry'. This requires a glossary document per domain, maintained by the domain team.
[Normal] Rationale
[Normal] Full DDD on simple CRUD domains is over-engineering. Settings module (CRUD for configuration key-value pairs) does not need aggregates or value objects — it's a lookup table. Forcing DDD patterns there adds ceremony without value. Pragmatic DDD applies DDD where domain complexity justifies it.
[Normal] Rich domain model for complex domains prevents the 'anemic model anti-pattern' where business logic ends up in services because entities are just data. FeeInvoice entity should know how to calculate its own total, mark itself as paid, and apply late fees — not delegate all this to a FeeService.
[Normal] Ubiquitous Language is the foundation of DDD. When code uses different terms than domain experts ('FeeInvoice' vs 'the bill'), communication breaks down and bugs creep in. Maintaining a glossary per domain forces alignment between business and engineering vocabulary.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Business logic localized to entities — easier to test, understand, and modify.
[List Paragraph] Aggregates enforce consistency — no half-paid invoices, no invalid state transitions.
[List Paragraph] Ubiquitous Language reduces business-engineering communication friction.
[List Paragraph] Domain events enable loose coupling between domains (ADR-036).
[Heading 3] Negative / Trade-offs
[List Paragraph] More upfront design for complex domains — mitigated by event storming sessions with domain experts.
[List Paragraph] Learning curve for engineers new to DDD — mitigated by pair programming and code review with DDD-experienced engineers.
[Normal] Implementation Notes
[Normal] Each complex domain has Aggregate Roots (e.g., FeeInvoice in Finance, Admission in Admission domain, Student in Student Lifecycle). Aggregates enforce consistency boundaries — all modifications go through the aggregate root. Value Objects (Money, Address, DateRange) encapsulate concepts with validation.
[Normal] // Finance domain — FeeInvoice aggregate root
[Normal] export class FeeInvoice {
[Normal]   private constructor(
[Normal]     public readonly id: string,
[Normal]     public readonly studentId: string,
[Normal]     public readonly academicYearId: string,
[Normal]     public items: FeeInvoiceItem[],
[Normal]     public status: InvoiceStatus,
[Normal]     public totalAmount: Money,  // Value Object
[Normal]     public paidAmount: Money,
[Normal]     public dueDate: Date,
[Normal]   ) {}
[Normal]   // Factory method — enforces invariants on creation
[Normal]   static create(props: CreateInvoiceProps): FeeInvoice {
[Normal]     if (props.items.length === 0) throw new Error('Invoice must have items');
[Normal]     const total = props.items.reduce((sum, i) => sum.add(i.amount), Money.zero());
[Normal]     return new FeeInvoice(uuid(), props.studentId, props.academicYearId,
[Normal]       props.items, 'ISSUED', total, Money.zero(), props.dueDate);
[Normal]   }
[Normal]   // Business logic lives on the aggregate
[Normal]   applyPayment(amount: Money, paymentDate: Date): DomainEvent[] {
[Normal]     if (this.status === 'PAID') throw new Error('Already paid');
[Normal]     if (paymentDate > this.dueDate) {
[Normal]       const lateFee = this.calculateLateFee(paymentDate);
[Normal]       this.totalAmount = this.totalAmount.add(lateFee);
[Normal]     }
[Normal]     this.paidAmount = this.paidAmount.add(amount);
[Normal]     if (this.paidAmount.equals(this.totalAmount)) {
[Normal]       this.status = 'PAID';
[Normal]       return [new FeeInvoicePaidEvent(this.id)];
[Normal]     }
[Normal]     this.status = 'PARTIALLY_PAID';
[Normal]     return [new FeeInvoicePartiallyPaidEvent(this.id, amount)];
[Normal]   }
[Normal]   private calculateLateFee(date: Date): Money {
[Normal]     // Domain logic — late fee calculation rules
[Normal]     const daysLate = Math.floor((date.getTime() - this.dueDate.getTime()) / 86400000);
[Normal]     return this.totalAmount.multiply(0.01).multiply(daysLate);  // 1% per day
[Normal]   }
[Normal] }
[Normal] // Money value object — encapsulates currency and prevents float errors
[Normal] export class Money {
[Normal]   private constructor(private amountInPaise: number) {}
[Normal]   static fromRupees(r: number): Money { return new Money(Math.round(r * 100)); }
[Normal]   static zero(): Money { return new Money(0); }
[Normal]   add(other: Money): Money { return new Money(this.amountInPaise + other.amountInPaise); }
[Normal]   multiply(factor: number): Money { return new Money(Math.round(this.amountInPaise * factor)); }
[Normal]   equals(other: Money): boolean { return this.amountInPaise === other.amountInPaise; }
[Normal]   toRupees(): number { return this.amountInPaise / 100; }
[Normal] }

[Heading 2] ADR-030: Clean Architecture + CQRS Where Justified
[Normal] Context
[Normal] PreOne's modules need a consistent internal structure. Without structure, business logic ends up in controllers (mixed with HTTP concerns), in repositories (mixed with SQL), or in service classes that become god-objects. Clean Architecture separates concerns into layers: Presentation (controllers), Application (use cases), Domain (entities, business rules), Infrastructure (databases, external APIs).
[Normal] CQRS (Command Query Responsibility Segregation) separates read operations from write operations. Most PreOne modules are read-heavy (dashboards, lists, reports) with relatively few writes (create fee invoice, mark attendance). Separating reads from writes enables independent optimization — reads can use denormalized views, writes go through rich domain models.
[Normal] Options Considered
[List Paragraph] No architecture (logic in controllers): Fast to start, unmaintainable as complexity grows. Business logic mixed with HTTP, SQL, validation.
[List Paragraph] Clean Architecture everywhere: Strict layering for all modules. Consistent but adds ceremony to simple CRUD modules.
[List Paragraph] Clean Architecture + CQRS for complex modules, simple CRUD for the rest: Pragmatic — full Clean Architecture + CQRS for read-heavy/complex domains (Finance, Reports, Daily Operations); simple service+repository for CRUD domains (Settings).
[Normal] Decision
[Normal] Adopt Option 3: Clean Architecture + CQRS where justified. Complex, read-heavy, or high-stakes modules (Finance, Reports & Analytics, Daily Operations, Admission) use full Clean Architecture with CQRS — separate Command handlers (writes through domain model) and Query handlers (reads via optimized read models). Simple CRUD modules (Settings, Platform Management) use a simpler service+repository pattern.
[Normal] CQRS enables read models to be denormalized for query performance. For example, the Reports module's 'Fee Collection Summary' query reads from a denormalized fee_collection_summary table updated by domain events, not from the normalized fee_invoice table — 10x faster for dashboard rendering.
[Normal] Rationale
[Normal] Clean Architecture's dependency rule (dependencies point inward toward domain) keeps business logic isolated from infrastructure. This means business logic is testable without a database, and infrastructure can be swapped (e.g., PostgreSQL to MongoDB for a specific read model) without touching business logic.
[Normal] CQRS is justified where read/write asymmetry is high. Reports module is 99% reads — building a separate read model optimized for reporting queries is worth the complexity of maintaining two models. Finance module has both heavy reads (statements, dashboards) and complex writes (fee calculation, payment application) — CQRS lets each side be optimized independently.
[Normal] CQRS is NOT justified for simple CRUD. Settings module has balanced reads/writes and simple logic — adding CQRS ceremony there would double the code for no benefit. Pragmatic application prevents over-engineering.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Business logic testable without infrastructure — fast unit tests.
[List Paragraph] Read models optimized independently — dashboard queries 10x faster.
[List Paragraph] Write side enforces business rules via domain aggregates — consistency guaranteed.
[List Paragraph] Simple modules stay simple — no ceremony where not justified.
[Heading 3] Negative / Trade-offs
[List Paragraph] Two models (read + write) for CQRS modules — eventual consistency between them. Mitigated by synchronous event handling for critical updates, async for non-critical.
[List Paragraph] More code for CQRS modules — Command + Handler + Query + Handler + Read Model Updater. Justified by complexity.
[Normal] Implementation Notes
[Normal] Clean Architecture layers (inside-out): Domain (entities, value objects, domain events), Application (commands, queries, handlers), Infrastructure (repositories, external API clients), Presentation (controllers, DTOs). Dependencies point inward — Infrastructure depends on Application, Application depends on Domain, never the reverse.
[Normal] CQRS modules have separate Command handlers (write path through domain aggregates) and Query handlers (read path through read models). Read models are updated by domain event handlers.

[Heading 2] ADR-031: Authorization Cache Resilience (Fail-Closed + LKG)
[Normal] Context
[Normal] ADR-021 mandates Redis for permission caching. But Redis is infrastructure that can fail — network partitions, OOM kills, maintenance restarts. When Redis is unavailable, the Effective Access Resolver must decide: fail-open (allow access, may be wrong) or fail-closed (deny access, user locked out)?
[Normal] Fail-open is convenient but a critical security risk — a Redis outage becomes a 'permission bypass' for all users. Fail-closed is secure but produces a 'system down' experience — users cannot log in or access anything during Redis outage. Neither extreme is acceptable.
[Normal] Options Considered
[List Paragraph] Fail-open (allow on Redis failure): Convenient, but a Redis outage grants all users all permissions. Critical security risk.
[List Paragraph] Fail-closed (deny on Redis failure): Secure, but a Redis outage locks out all users. Operational disaster.
[List Paragraph] Fail-closed + Last Known Good (LKG) snapshot: If Redis is down, try to use a snapshot of the user's last known good permissions (max age 1 hour). If no snapshot, deny. Security risk limited to 1-hour window of stale permissions.
[Normal] Decision
[Normal] Adopt Option 3: Fail-Closed + Last Known Good (LKG) snapshot. When Redis is unavailable, the resolver attempts to use a LKG snapshot of the user's permissions, stored locally (filesystem or separate storage) with a max age of 1 hour. If no LKG snapshot exists or it's older than 1 hour, access is denied and an admin alert is generated.
[Normal] Critical rule: NEVER fail-open. Even if LKG snapshot is stale, it represents actual permissions the user had at most 1 hour ago — far safer than granting all permissions. LKG is a degradation strategy, not a bypass.
[Normal] Rationale
[Normal] Fail-open is the single most dangerous pattern in authorization systems. A Redis outage would grant every user Super Admin permissions, leaking all tenant data. This is unacceptable under any compliance framework (SOC2, ISO27001, DPDP Act 2023).
[Normal] Fail-closed without LKG is operationally unacceptable. A 5-minute Redis maintenance window would lock out every user in every school — a multi-tenant SaaS incident affecting thousands of customers. LKG provides graceful degradation: users keep working with their most recent permissions, with a 1-hour safety window.
[Normal] The 1-hour max age on LKG is a security/availability tradeoff. Longer (24 hours) increases availability during outages but increases stale-permission risk. Shorter (5 minutes) reduces risk but provides less availability benefit. 1 hour balances both: enough to cover typical Redis outages, short enough to limit damage from stale permissions.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Redis outage degrades gracefully — users continue working with LKG permissions.
[List Paragraph] Security risk bounded to 1-hour stale window — far better than fail-open's total bypass.
[List Paragraph] Never fail-open — compliance-friendly.
[Heading 3] Negative / Trade-offs
[List Paragraph] LKG snapshot storage adds complexity — mitigated by simple filesystem or separate DB table.
[List Paragraph] Stale permissions risk during outage — mitigated by 1-hour max age and admin alerting.
[Normal] Implementation Notes
[Normal] On every successful cache write (Redis), a LKG snapshot is written to local filesystem (or a separate database table). On Redis failure, the resolver reads the LKG snapshot, checks its age (< 1 hour), and uses it if valid. If invalid or missing, access is denied and admin alert is sent.

[Heading 2] ADR-032: Permission Cache Invalidation (Event-Driven)
[Normal] Context
[Normal] ADR-022 bumps the global permission-version on any RBAC change, invalidating all cached EffectiveAccess entries. The next request from any user rebuilds their cache from the database. For a single user, this is fine — one rebuild per permission change.
[Normal] But for a school with 500 active users, a single role change (e.g., adding a new permission to the Teacher role) triggers 500 cache rebuilds on the next 500 requests. If all 500 users are active simultaneously (e.g., morning login rush at 9 AM), this is 500 concurrent database-heavy cache rebuilds — a cache stampede that can overwhelm the database.
[Normal] Options Considered
[List Paragraph] Synchronous rebuild on every cache miss: Each request rebuilds its own cache. Simple but causes stampede — 500 concurrent rebuilds on version bump.
[List Paragraph] Flush all caches on version bump: Redis FLUSHDB on version bump. Forces mass rebuild but no control over timing — stampede guaranteed.
[List Paragraph] Event-driven staggered rebuild: Version bump publishes event. Background worker picks up event, identifies affected users, rebuilds their caches proactively with jitter (staggered timing). Frontend requests during rebuild use single-flight pattern.
[Normal] Decision
[Normal] Adopt Option 3: Event-driven staggered rebuild. When the permission-version is bumped, a CacheInvalidationEvent is published to the event bus. A background worker subscribes, identifies users whose caches are stale (via version check), and proactively rebuilds their caches with random jitter (0-30 seconds) to spread the load.
[Normal] Single-flight pattern: if multiple requests for the same user arrive during rebuild, only one rebuild runs; others wait for its result. This prevents stampede per user even if many requests arrive simultaneously.
[Normal] Rationale
[Normal] Cache stampede is a well-known failure mode in distributed caching. A naive 'invalidate on change, rebuild on demand' approach produces thundering herd: all users' caches expire simultaneously, all rebuild simultaneously, database is overwhelmed. Event-driven proactive rebuild with jitter spreads the load over time.
[Normal] Single-flight pattern prevents per-user stampede. Without it, 100 concurrent requests from the same user (e.g., page load triggering multiple API calls) would each rebuild the cache independently — 100x the database load for one user. Single-flight ensures one rebuild, 99 waiters.
[Normal] This pattern is used by Facebook's cache invalidation (Memcached stampede prevention), Netflix's Hystrix (collapsing requests), and Shopify's permission cache (proactive rebuild on role change). PreOne's implementation follows the same principles at smaller scale.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] No cache stampede — proactive rebuild with jitter spreads database load.
[List Paragraph] Single-flight prevents per-user thundering herd.
[List Paragraph] Stale cache window bounded by rebuild completion (seconds), not TTL (hours).
[Heading 3] Negative / Trade-offs
[List Paragraph] Background worker adds infrastructure — mitigated by simple in-process worker initially, extractable to separate service later.
[List Paragraph] Rebuild consumes database resources even for users not actively requesting — mitigated by prioritizing recently-active users.
[Normal] Implementation Notes
[Normal] The background worker subscribes to CacheInvalidationEvent. It identifies affected users (those with stale caches), prioritizes by recent activity, and rebuilds their caches with random jitter. Single-flight is implemented via in-flight rebuild tracking.

[Normal] @OnEvent('cache_invalidation')
[Normal] export class CacheRebuildWorker {
[Normal]   private inFlightRebuilds = new Map<string, Promise<EffectiveAccess>>();
[Normal]   async handle(event: CacheInvalidationEvent) {
[Normal]     // Find users with stale caches (version < event.newVersion)
[Normal]     const staleUsers = await this.findStaleUsers(event.newVersion);
[Normal]     // Prioritize by recent activity (last 24 hours)
[Normal]     staleUsers.sort((a, b) => b.lastActiveAt - a.lastActiveAt);
[Normal]     // Rebuild with jitter (0-30 seconds)
[Normal]     for (const user of staleUsers) {
[Normal]       const jitter = Math.random() * 30 * 1000;  // 0-30s
[Normal]       setTimeout(() => this.rebuildUserCache(user.id, event.newVersion), jitter);
[Normal]     }
[Normal]   }
[Normal]   // Single-flight: one rebuild per user, others wait
[Normal]   async rebuildUserCache(userId: string, version: number): Promise<EffectiveAccess> {
[Normal]     if (this.inFlightRebuilds.has(userId)) {
[Normal]       return this.inFlightRebuilds.get(userId)!;  // wait for in-flight
[Normal]     }
[Normal]     const promise = (async () => {
[Normal]       try {
[Normal]         const access = await this.resolver.buildEffectiveAccess(userId, version);
[Normal]         await redis.setex(`user:${userId}:access`, 3600, JSON.stringify(access));
[Normal]         return access;
[Normal]       } finally {
[Normal]         this.inFlightRebuilds.delete(userId);
[Normal]       }
[Normal]     })();
[Normal]     this.inFlightRebuilds.set(userId, promise);
[Normal]     return promise;
[Normal]   }
[Normal] }
[Normal] // Resolver uses single-flight for on-demand rebuild
[Normal] async function resolve(userId: string): Promise<EffectiveAccess> {
[Normal]   const cached = await redis.get(`user:${userId}:access`);
[Normal]   if (cached) {
[Normal]     const access = JSON.parse(cached);
[Normal]     if (access.version === await this.getCurrentVersion()) return access;
[Normal]   }
[Normal]   // Cache miss or stale — use single-flight rebuild
[Normal]   return this.rebuildWorker.rebuildUserCache(userId, await this.getCurrentVersion());
[Normal] }

[Heading 2] ADR-033: Migration Identity (SYSTEM_MIGRATION Role)
[Normal] Context
[Normal] PreOne's production launch requires data migration from existing preschool systems (spreadsheets, legacy ERPs, paper records). Migration scripts insert thousands of rows — students, fees, attendance history — and these rows need audit metadata (created_by, updated_by per ADR-024).
[Normal] Using an admin user's account for migration ties audit trails to a specific person, which is misleading (the admin didn't manually insert 4,826 students). Using NULL created_by violates ADR-024. Anonymous 'system' users lose accountability — 'who migrated this data?' becomes unanswerable.
[Normal] Options Considered
[List Paragraph] Use admin account for migration: Ties migration to a person. Audit shows 'Kavita created 4826 students' — misleading and not useful for debugging migration issues.
[List Paragraph] Anonymous 'system' user: All migrations attributed to 'system'. No accountability, no traceability to specific migration run.
[List Paragraph] Dedicated SYSTEM_MIGRATION role with correlation IDs: Migration runs as SYSTEM_MIGRATION system role. Each migration run has a correlation ID (e.g., MIG-2026-001) stored in audit log. Full traceability without tying to a person.
[Normal] Decision
[Normal] Adopt Option 3: Dedicated SYSTEM_MIGRATION system role. Migration scripts authenticate as SYSTEM_MIGRATION (a system role per ADR-026). Every migrated row has created_by=SYSTEM_MIGRATION_USER_ID, updated_by=SYSTEM_MIGRATION_USER_ID. Audit log entries include a correlation_id (e.g., MIG-2026-001) linking rows to a specific migration run.
[Normal] SYSTEM_MIGRATION has special capabilities: scope bypass (can write to any tenant), validation bypass (configurable — can skip unique constraints during bulk import for performance), mandatory audit logging, dry-run mode, and rollback support.
[Normal] Rationale
[Normal] Migration is a distinct operational concern that deserves its own identity. SYSTEM_MIGRATION role makes migration explicit in the architecture — there is no ambiguity about 'was this row created by a human or by migration?'. This clarity is essential for debugging migration issues and for compliance audits.
[Normal] Correlation IDs link rows to migration runs. When a school reports 'we have duplicate students after migration', the correlation_id in the audit log identifies which migration run created the duplicates, enabling targeted investigation and rollback.
[Normal] Dry-run and rollback are essential for safe migration. Dry-run validates data without persisting; rollback undoes a migration run if issues are discovered post-import. These capabilities require SYSTEM_MIGRATION to have elevated permissions (validation bypass for dry-run, scope bypass for rollback across tenants).
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Migration identity is explicit — no ambiguity in audit trail.
[List Paragraph] Correlation IDs enable targeted debugging and rollback.
[List Paragraph] Dry-run catches migration issues before production impact.
[List Paragraph] Rollback undoes bad migrations without manual cleanup.
[Heading 3] Negative / Trade-offs
[List Paragraph] SYSTEM_MIGRATION has elevated permissions — must be tightly controlled.
[List Paragraph] Validation bypass during migration can let bad data in — mitigated by post-migration validation reports.
[Normal] Implementation Notes
[Normal] Migration scripts use a dedicated SYSTEM_MIGRATION user account (one per environment — staging, production). Scripts pass a correlation_id (e.g., MIG-2026-001) that is stored in every audit log entry for that migration run. Dry-run mode validates without persisting; rollback mode deletes rows matching the correlation_id.

[Heading 2] ADR-034: Soft Delete Policy
[Normal] Context
[Normal] Physical deletion of records causes two problems: (1) audit trail gaps — if a student record is deleted, the audit log references a non-existent entity; (2) accidental data loss — a misplaced DELETE FROM student WHERE clause can wipe a school's data with no recovery. Soft delete (marking records as deleted, not removing them) solves both.
[Normal] However, not all data should be soft-deletable. Audit logs must be immutable (append-only, no deletes ever) for compliance. Transactional data (fee invoices, attendance) needs soft delete with approval workflow to prevent casual deletion. Master data (students, staff, classes) needs straightforward soft delete.
[Normal] Options Considered
[List Paragraph] Physical delete everywhere: DELETE FROM removes rows. Audit trail gaps, accidental data loss risk. Non-compliant.
[List Paragraph] Soft delete everywhere (including audit log): is_deleted on all tables. Audit log becomes mutable — violates compliance.
[List Paragraph] Tiered soft delete policy: Master data: soft delete. Transactions: soft delete + approval workflow. Audit log: immutable (append-only). System config: versioned (no delete, only supersede).
[Normal] Decision
[Normal] Adopt Option 3: Tiered Soft Delete Policy. Master data (Student, Staff, Class, Branch) uses soft delete via is_deleted, deleted_at, deleted_by columns (already in ADR-024). Transactional data (FeeInvoice, Attendance, DailySheet) uses soft delete but requires approval workflow — only admin-approved deletions proceed. Audit log is immutable (no delete, no update, append-only). System configuration uses versioning (no delete, only supersede with previous version retained).
[Normal] Rationale
[Normal] Tiered policy recognizes that different data types have different deletion requirements. Master data soft delete is straightforward — when a student transfers out, mark is_deleted=true; the record remains for historical reference and can be restored if the student returns.
[Normal] Transactional data needs approval workflow because deletions have financial or operational impact. Deleting a fee invoice affects outstanding balances; deleting attendance affects report cards. Approval workflow ensures deletions are deliberate and reviewable.
[Normal] Audit log immutability is non-negotiable for compliance. SOC2, ISO27001, and DPDP Act 2023 all require tamper-evident audit trails. If audit logs could be deleted or modified, the audit trail cannot be trusted. Append-only (INSERT only, no UPDATE or DELETE) is the standard pattern.
[Normal] System configuration versioning preserves history of changes. When a fee head is 'deleted', it's actually superseded by a new version with is_active=false; the old version remains for historical fee invoice references. This prevents 'fee invoice references non-existent fee head' errors.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] No audit trail gaps — soft-deleted records remain referenceable.
[List Paragraph] Accidental data loss prevented — soft delete is reversible.
[List Paragraph] Compliance-friendly — audit log immutable, configuration versioned.
[List Paragraph] Transactional deletions controlled — approval workflow prevents casual deletion.
[Heading 3] Negative / Trade-offs
[List Paragraph] Tables grow with soft-deleted rows — mitigated by indexing is_deleted and partitioning if needed.
[List Paragraph] Queries must filter is_deleted=false — mitigated by repository pattern that auto-applies filter.
[Normal] Implementation Notes
[Normal] All aggregate root tables have is_deleted, deleted_at, deleted_by columns (ADR-024). BaseRepository's find() auto-applies WHERE is_deleted=false. Soft delete via BaseRepository.softDelete() sets all three columns. Audit log table has no UPDATE or DELETE permissions granted to any role — only INSERT.

[Heading 2] ADR-035: Audit Strategy
[Normal] Context
[Normal] Naive audit strategies log every CRUD operation, producing audit tables that grow by millions of rows per day. At scale, this overwhelms storage and makes audit queries slow. Worse, the noise of 'user viewed student record X' drowns out the signal of 'admin changed fee structure Y'.
[Normal] PreOne needs a focused audit strategy that logs business-critical and compliance-relevant events, not every database write. Indian DPDP Act 2023 requires logging of PII access (parent contact, child medical records), data exports, and authentication events. SOC2 requires logging of authorization changes and security-relevant actions.
[Normal] Options Considered
[List Paragraph] Log everything (every CRUD): Complete trail but overwhelming volume. Audit table grows by millions of rows/day. Storage and query performance issues.
[List Paragraph] Log only explicit audit calls: Developers add auditLog.record() calls in code. Incomplete — easy to forget. No guarantee of coverage.
[List Paragraph] Log by category (6 audit categories): Define 6 categories of mandatory audit events. Logging is automatic for these categories via decorators/middleware. Optional explicit logging for custom events.
[Normal] Decision
[Normal] Adopt Option 3: Audit by category. Six mandatory audit categories cover compliance and security requirements without logging every CRUD:
[Normal] 1. Authentication events (login, logout, failed login, password reset, MFA changes)
[Normal] 2. Authorization changes (role creation/update/delete, role_module_permission change, assignment change, user_role change)
[Normal] 3. PII access (parent contact info, child medical records, child documents, parent financial info)
[Normal] 4. Financial transactions (fee invoice create/update/delete, payment, refund, expense, payroll)
[Normal] 5. Critical state changes (admission approval, student transfer, student graduation, student archive)
[Normal] 6. Data exports (any list/report download, bulk export, API access to large datasets)
[Normal] Rationale
[Normal] Audit by category focuses logging on what matters for compliance and security. Authentication events catch intrusion attempts. Authorization changes catch privilege escalation. PII access is mandated by DPDP Act. Financial transactions are business-critical. Critical state changes are irreversible or hard to reverse. Data exports are the primary vector for data exfiltration.
[Normal] Automatic logging via decorators/middleware prevents gaps. @AuditLog(category='PII_ACCESS') decorator on controller methods automatically logs without developer intervention. Middleware on authentication routes logs all auth events. This ensures coverage even when developers forget explicit audit calls.
[Normal] Excluding routine CRUD (viewing a class list, updating a daily sheet) keeps audit table manageable. These operations are tracked in change history (old_values/new_values on update) but not in the audit log. The audit log is for security and compliance, not operational change tracking.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Audit log volume manageable — only security/compliance-relevant events.
[List Paragraph] DPDP Act compliance — PII access logged.
[List Paragraph] SOC2 compliance — authorization changes logged.
[List Paragraph] Query performance — audit log stays small enough for fast queries.
[Heading 3] Negative / Trade-offs
[List Paragraph] Operational change tracking is separate (in entity history tables) — minor additional complexity.
[List Paragraph] Risk of missing a category if compliance requirements change — mitigated by annual audit strategy review.
[Normal] Implementation Notes
[Normal] Audit logging is implemented via decorators on controllers and middleware on auth routes. The @AuditLog decorator accepts a category and optional description; the middleware inspects the request/response and logs automatically. Audit log entries include actor, action, entity, entity_id, old_values, new_values, and timestamp.
[Normal] // Decorator for automatic audit logging
[Normal] export function AuditLog(category: AuditCategory, description?: string) {
[Normal]   return function (target: any, propertyKey: string, descriptor: PropertyDescriptor) {
[Normal]     const originalMethod = descriptor.value;
[Normal]     descriptor.value = async function (...args: any[]) {
[Normal]       const request = args[0];  // assume first arg is request
[Normal]       const result = await originalMethod.apply(this, args);
[Normal]       // Log after successful execution
[Normal]       await this.auditLog.record({
[Normal]         category,
[Normal]         actor: 'USER',
[Normal]         actorUserId: request.user.id,
[Normal]         actorRole: request.user.primaryRole,
[Normal]         schoolId: request.user.schoolId,
[Normal]         action: `${request.method}_${propertyKey.toUpperCase()}`,
[Normal]         entity: request.params.entity || 'N/A',
[Normal]         entityId: request.params.id || 'N/A',
[Normal]         description: description || `${propertyKey} called`,
[Normal]         ipAddress: request.ip,
[Normal]         userAgent: request.headers['user-agent'],
[Normal]       });
[Normal]       return result;
[Normal]     };
[Normal]   };
[Normal] }

[Heading 2] ADR-036: Domain Events
[Normal] Context
[Normal] PreOne's 13 business domains need to communicate — Admission domain creates a Student, which Finance needs to bill, which Communication needs to notify parents about, which Reports needs to track. Without a decoupling mechanism, domains would call each other's services directly, producing tight coupling that breaks the modular monolith (ADR-028).
[Normal] Direct integration (Admission service calls Finance service) couples domains at compile time. A change to Finance's createInvoice signature requires updating Admission. Adding a new consumer (e.g., Analytics wants to track admissions) requires modifying Admission to call Analytics. This coupling is the death of modular architecture.
[Normal] Options Considered
[List Paragraph] Direct service-to-service calls: Admission calls Finance.createInvoice(studentId). Tight coupling, brittle to change.
[List Paragraph] Shared database (Finance reads Admission tables): Finance queries admission table directly. Couples at database layer, breaks module boundaries.
[List Paragraph] Domain events (publish-subscribe): Admission publishes StudentCreated event. Finance, Communication, Analytics subscribe. Decoupled — new subscribers don't require Admission changes.
[Normal] Decision
[Normal] Adopt Option 3: Domain Events. When a domain changes state, it publishes a domain event (e.g., StudentCreated, FeeInvoicePaid, AttendanceMarked). Other domains subscribe to events they care about. Publishers don't know who subscribes; subscribers don't know who publishes. Decoupled at the event contract level.
[Normal] Event schema is versioned (version field in event payload). Subscribers handle version negotiation — they can consume events they understand and ignore or transform versions they don't. This allows publishers to evolve event schemas without breaking all subscribers simultaneously.
[Normal] Rationale
[Normal] Domain events enable loose coupling between domains. Admission publishes StudentCreated and forgets about it — Finance, Communication, Analytics each handle the event independently. Adding a new consumer (e.g., AI domain for growth prediction) requires only subscribing to the event, not modifying Admission.
[Normal] Versioned event schemas handle evolution. When Admission adds a new field to StudentCreated (e.g., scholarshipEligibility), they bump the version. Existing subscribers that don't need the new field continue working with v1. New subscribers that need the field consume v2. This avoids big-bang migrations when event schemas change.
[Normal] This pattern is fundamental to DDD (event-driven architecture between bounded contexts) and is used by enterprise systems like Shopify (Dequeue), Netflix (Eureka + EventBus), and Amazon (EventBridge). PreOne's in-process event bus is a simplified version, extractable to Kafka/RabbitMQ when needed (ADR-028).
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Loose coupling — domains evolve independently.
[List Paragraph] Extensible — new consumers don't require publisher changes.
[List Paragraph] Versioned — schema evolution without big-bang migrations.
[List Paragraph] Extractable — in-process bus can swap to message broker for microservices.
[Heading 3] Negative / Trade-offs
[List Paragraph] Eventual consistency — subscribers react asynchronously, slight lag between publisher and subscriber state.
[List Paragraph] Event schema governance — versioning requires discipline to avoid breaking consumers.
[List Paragraph] Debugging harder — event chains span multiple domains, harder to trace than direct calls.
[Normal] Implementation Notes
[Normal] Domain events extend a base DomainEvent class with version, occurredAt, and payload. Publishers use eventBus.publish(event). Subscribers use @OnEvent('event.name') decorator. In-process event bus is synchronous by default (publisher waits for subscribers); async mode is available for non-critical events.

[Heading 2] ADR-037: Feature Flags (3-Level: Platform/School/Plan)
[Normal] Context
[Normal] PreOne is a SaaS with multiple pricing tiers (Free, Standard, Premium) and per-school feature enablement. The AI Observation Writer might be globally available (platform level), enabled for School A but not School B (school level), and only available on Premium plan (plan level). All three conditions must be TRUE for the feature to activate for a given user.
[Normal] Without feature flags, enabling a feature for one school requires a code deployment. With 1000+ schools, deploying per-school configuration is operationally infeasible. Feature flags decouple feature rollout from deployment, enabling per-school, per-plan, per-platform configuration without code changes.
[Normal] Options Considered
[List Paragraph] Code-based feature flags (if statements): if (school.id === 'X') enableFeature(). Requires deployment per change. Doesn't scale.
[List Paragraph] Single-level flags (school only): Per-school enable/disable. Doesn't handle plan-level restrictions (Free vs Premium) or platform-wide rollout control.
[List Paragraph] 3-level flags (platform/school/plan): All three levels must be TRUE. Platform controls global availability, school controls per-customer enablement, plan controls tier-based access.
[Normal] Decision
[Normal] Adopt Option 3: 3-Level Feature Flags. A feature is active for a user only if ALL three levels return TRUE:
[Normal] 1. Platform Level — 'Is this feature globally available on PreOne?' (e.g., AI features may be disabled platform-wide during beta)
[Normal] 2. School Level — 'Has this school enabled the feature?' (per-customer toggle, controlled by school admin)
[Normal] 3. Plan Level — 'Does the school's subscription plan include this feature?' (Free/Standard/Premium tier restrictions)
[Normal] If any level is FALSE, the feature is inactive. This prevents Free plan schools from using Premium features even if school-level flag is on.
[Normal] Rationale
[Normal] 3-level flags model real-world SaaS feature gating. Platform level allows PreOne to disable features globally (e.g., disable AI features during outage or beta). School level allows per-customer enablement (e.g., enable WhatsApp integration for School A that has paid for it). Plan level enforces subscription tier restrictions (e.g., AI Observation Writer is Premium-only).
[Normal] The AND logic (all three must be TRUE) is the safest default. If platform disables a feature, no school or plan can override. If school disables, plan doesn't matter. If plan doesn't include, school can't enable. This prevents 'Free plan school enables Premium feature via school flag' mistakes.
[Normal] This pattern is used by SaaS platforms like Shopify (platform/app/shop levels), Slack (workspace/plan levels), and Atlassian (platform/site/plan levels). PreOne's 3-level model is a simplified version tuned for preschool SaaS.
[Normal] Consequences
[Heading 3] Positive
[List Paragraph] Feature rollout decoupled from deployment — no code changes per school/plan.
[List Paragraph] Platform-wide kill switch for emergencies (disable feature globally).
[List Paragraph] Plan enforcement prevents Free tier from accessing Premium features.
[List Paragraph] Per-school enablement allows custom customer configurations.
[Heading 3] Negative / Trade-offs
[List Paragraph] Three-level evaluation adds slight latency per feature check — mitigated by caching flags.
[List Paragraph] Configuration complexity — admins must understand three levels — mitigated by UI that shows effective state per feature.
[Normal] Implementation Notes
[Normal] Feature flags are stored in three tables: platform_feature_flag (global), school_feature_flag (per-school), plan_feature_flag (per-plan). The FeatureFlagService evaluates all three levels for a given (user, feature) pair and returns the effective boolean. Flags are cached in Redis with 5-minute TTL.
[Normal] -- Platform level (global)
[Normal] CREATE TABLE platform_feature_flag (
[Normal]   feature VARCHAR(50) PRIMARY KEY,  -- 'ai_observation_writer'
[Normal]   enabled BOOLEAN NOT NULL DEFAULT FALSE,
[Normal]   updated_by UUID NOT NULL,
[Normal]   updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
[Normal] );
[Normal] -- School level (per-customer)
[Normal] CREATE TABLE school_feature_flag (
[Normal]   school_id UUID NOT NULL REFERENCES school(id),
[Normal]   feature VARCHAR(50) NOT NULL,
[Normal]   enabled BOOLEAN NOT NULL DEFAULT FALSE,
[Normal]   updated_by UUID NOT NULL,
[Normal]   updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
[Normal]   PRIMARY KEY (school_id, feature)
[Normal] );
[Normal] -- Plan level (per-subscription-tier)
[Normal] CREATE TABLE plan_feature_flag (
[Normal]   plan VARCHAR(20) NOT NULL,  -- 'FREE', 'STANDARD', 'PREMIUM'
[Normal]   feature VARCHAR(50) NOT NULL,
[Normal]   enabled BOOLEAN NOT NULL DEFAULT FALSE,
[Normal]   PRIMARY KEY (plan, feature)
[Normal] );
[Normal] -- Feature flag evaluation service
[Normal] @Injectable()
[Normal] export class FeatureFlagService {
[Normal]   async isEnabled(userId: string, feature: string): Promise<boolean> {
[Normal]     const cacheKey = `flag:${userId}:${feature}`;
[Normal]     const cached = await this.redis.get(cacheKey);
[Normal]     if (cached !== null) return cached === 'true';
[Normal]     // Evaluate all three levels
[Normal]     const user = await this.userRepo.findById(userId);
[Normal]     const school = await this.schoolRepo.findById(user.schoolId);
[Normal]     const [platform, schoolFlag, planFlag] = await Promise.all([
[Normal]       this.platformFlagRepo.find(feature),
[Normal]       this.schoolFlagRepo.find(user.schoolId, feature),
[Normal]       this.planFlagRepo.find(school.subscriptionPlan, feature),
[Normal]     ]);
[Normal]     const enabled = platform?.enabled && schoolFlag?.enabled && planFlag?.enabled;
[Normal]     await this.redis.setex(cacheKey, 300, String(enabled));  // 5 min TTL
[Normal]     return enabled;
[Normal]   }
[Normal] }
[Normal] // Usage in controllers
[Normal] @Get('observations/ai-suggest')
[Normal] async suggestObservation(@User() user: AuthUser) {
[Normal]   if (!await this.featureFlags.isEnabled(user.id, 'ai_observation_writer')) {
[Normal]     throw new ForbiddenException('AI Observation Writer not available on your plan');
[Normal]   }
[Normal]   return this.aiService.suggestObservation(...);
[Normal] }

[Heading 2] ADR Lifecycle
[Normal] Architecture Decision Records follow a defined lifecycle to ensure deliberate evolution of the architecture:
[List Paragraph] Draft: Engineer writes a draft ADR using the standard format (Context, Options Considered, Decision, Rationale, Consequences, Implementation Notes, Diagrams, Risk & Mitigation, Cross-References).
[List Paragraph] Review: Draft is circulated to the engineering team for review. Feedback is addressed, ADR is updated.
[List Paragraph] Approval: Architecture review board reviews and approves the ADR. Approval requires consensus that the decision is sound, well-documented, and aligned with existing ADRs.
[List Paragraph] Publication: Approved ADR is added to this document with the next ADR number. Status is set to LOCKED.
[List Paragraph] Supersession: If a future ADR invalidates an existing one, the old ADR is marked SUPERSEDED with a reference to the new ADR. The old ADR remains in the document for historical reference.
[List Paragraph] Amendment: Minor additions to an existing ADR (without changing the decision) are made inline, with a note in the ADR's metadata indicating the amendment date and nature.
