[00051] [Normal] PreOne API security 12 layers मध्ये structured आहे. हे layers defense-in-depth principle follow करतात — एक layer compromised झाला तरी इतर layers attack prevent करतात. Security architecture ADR-009 (Security Architecture) आणि DPDP Act 2023 compliance requirements वर based आहे.

[00053] [Heading 2] 3.2 Security Layers
[00054] [Heading 2] 3.3 PII Encryption at Rest
[00055] [Normal] PII fields (student name, parent contact, medical info, Aadhaar) PostgreSQL pgcrypto द्वारे encrypted आहेत. Encryption key HashiCorp Vault मध्ये stored, quarterly rotated. Application layer transparent decrypt करते (Prisma extension), पण database dump compromise झाला तरी PII ciphertext मिळेल — useless without key.
[00056] [Heading 2] 3.4 Webhook Security
[00057] [Normal] Outbound webhooks HMAC-SHA256 ने signed आहेत. Recipients MUST signature verify करावी — नाही तर replay attack possible. X-PreOne-Signature header मध्ये hex-encoded signature. Replay window 5 minutes — older payloads recipient ने reject करावे.

[00069] [Normal] Access token expiry झाल्यावर client refresh token वापरून new access token fetch करतो. Refresh token rotation — प्रत्येक refresh मध्ये old refresh token blacklist, new refresh token issue. ह्यामुळे stolen refresh token quickly useless होतो.

[00072] [Heading 2] 4.5 Token Lifecycle
[00073] [Normal] Token lifecycle खालील steps मध्ये managed होतो:
[00074] [Normal] •  User logs in via OTP/password → server issues accessToken (15min) + refreshToken (30day)
[00075] [Normal] •  Client uses accessToken for all API calls in Authorization: Bearer <token> header
[00076] [Normal] •  When accessToken expires, client calls POST /auth/refresh with refreshToken → new accessToken
[00077] [Normal] •  Refresh token is rotated on each use (one-time use) — old refresh token blacklisted
[00078] [Normal] •  User logs out → POST /auth/logout → both tokens blacklisted server-side
[00079] [Normal] •  After 30 days of inactivity → refreshToken expires → full re-authentication required
[00080] [Normal] •  Compromised token? Admin can force-invalidate all sessions for a user via POST /users/{id}/sessions/invalidate

[00164] [Heading 2] 10.2 Pagination Rules
[00165] [Normal] Page-based pagination साठी rules:
[00166] [Normal] •  Default pageSize: 20. Max pageSize: 100. Larger sizes rejected with VALIDATION_001.
[00167] [Normal] •  Page numbering is 1-indexed (page=1 is first page). page=0 returns 400.
[00168] [Normal] •  For datasets >10k rows, prefer cursor-based pagination (?cursor=abc&limit=20) for performance.
[00169] [Normal] •  OFFSET pagination becomes O(n) slow at deep pages — cursor is O(1).
[00170] [Normal] •  Response meta always includes: page, pageSize, total, totalPages, hasMore, nextCursor.
[00171] [Normal] •  Empty result set returns 200 with data:[] and meta.total:0 (NOT 404).
[00172] [Normal] •  Sorting + pagination: sort must be deterministic (include id as tiebreaker).
[00173] [Heading 2] 10.3 Cursor-Based Pagination (Large Datasets)
[00174] [Normal] Datasets >10k rows साठी cursor-based pagination recommended आहे. OFFSET pagination O(n) slow होते deep pages वर — cursor O(1) राहतो. Cursor opaque string आहे (client ला decode करायची गरज नाही), server-side last-seen-ID encode करतो. Request: ?cursor=abc&limit=20. Response: meta.nextCursor field.

[00194] [Normal] Sorting ?sort query param द्वारे done आहे. Single-field sort: ?sort=firstName. Descending: prefix with - (e.g., -createdAt). Multi-field: comma-separated (e.g., lastName,-createdAt). Legacy order=asc|desc param सुद्धा supported आहे (single-field only, backward compat).
[00195] [Heading 2] 12.2 Sorting Rules
[00196] [Normal] •  Sort parameter accepts comma-separated field names.
[00197] [Normal] •  Prefix with - for descending order (e.g., -createdAt).
[00198] [Normal] •  Legacy order=asc|desc param still supported for backward compat (single-field only).
[00199] [Normal] •  Sortable fields are whitelisted per endpoint — unknown fields rejected with VALIDATION_001.
[00200] [Normal] •  Sort is always deterministic — primary sort + id as final tiebreaker.
[00201] [Normal] •  Sort field must be indexed for performance — endpoint metadata exposes indexed sortable fields.
[00202] [Normal] •  Multi-field sort respects field order — first field has highest precedence.
[00204] [Heading 1] 13. Versioning
[00205] [Heading 2] 13.1 Versioning Policy
[00206] [Normal] URI path versioning वापरले जाते — /api/v1/, /api/v2/. हा approach clients साठी सर्वात intuitive आहे (URL ने version visible), routing सोपा आहे (load balancer prefix route), आणि debugging easier आहे (logs मध्ऍे version visible).
[00207] [Heading 2] 13.2 Versioning Rules
[00208] [Normal] Versioning strict rules follow करतात — long-term client compatibility साठी:
[00209] [Normal] •  URI path versioning: /api/v1/, /api/v2/. Most intuitive for clients, easiest to route.
[00210] [Normal] •  Within a major version, NO breaking changes — only additive (new fields, new endpoints, new optional params).
[00211] [Normal] •  Breaking changes (rename field, remove field, change type, change semantics) require new major version.
[00212] [Normal] •  New major version has 12-month overlap with prior version before sunset.
[00213] [Normal] •  Deprecation headers sent on deprecated endpoints: Sunset: Wed, 30 Jun 2027 00:00:00 GMT; Deprecation: true; Link: </api/v2/students>; rel="successor-version".
[00214] [Normal] •  Version support policy: current major + 1 prior major. v3 release → v1 sunset.
[00215] [Normal] •  Migration guide published with each major release: field mapping, behavior changes, SDK updates.
[00216] [Normal] •  Versioning is per-API-surface, not per-endpoint — all endpoints under /api/v1/ move together.
[00217] [Heading 2] 13.3 Breaking vs Non-Breaking Changes
[00218] [Normal] Breaking changes (new major version required):
[00219] [Normal] •  Remove a field from response
[00220] [Normal] •  Rename a field (e.g., studentName → firstName)
[00221] [Normal] •  Change field type (e.g., string → number)
[00222] [Normal] •  Change field semantic (e.g., amount in rupees → paise)
[00223] [Normal] •  Change required-ness of a request field (optional → required)
[00224] [Normal] •  Change error code (e.g., AUTH_001 → AUTH_BAD)
[00225] [Normal] •  Change default behavior (e.g., default pageSize 20 → 50)
[00226] [Normal] Non-breaking changes (within same major version OK):
[00227] [Normal] •  Add new field to response
[00228] [Normal] •  Add new optional field to request
[00229] [Normal] •  Add new endpoint
[00230] [Normal] •  Add new enum value (clients should handle unknown enums gracefully)
[00231] [Normal] •  Add new error code (clients should fallback to message)
[00233] [Heading 1] 14. Idempotency
[00234] [Heading 2] 14.1 Idempotency-Key Header
[00235] [Normal] Idempotency-Key header (UUID v4) POST/PUT/PATCH methods वर supported आहे. हा header server ला सांगतो की हा request retry आहे — same key + same payload आला तर cached response return करा. Payment, refund, admission submission, file upload यांसारख्या critical endpoints साठी MANDATORY आहे.
[00236] [Heading 2] 14.2 Idempotency Rules
[00237] [Normal] •  Idempotency-Key header (UUID v4) supported on POST, PUT, PATCH methods.
[00238] [Normal] •  MANDATORY for: payments, refunds, admissions submission, file uploads, invoice generation.
[00239] [Normal] •  Server stores key + response hash for 24 hours in Redis.
[00240] [Normal] •  Same Idempotency-Key + same payload → returns cached response (HTTP 200/201 + identical body).
[00241] [Normal] •  Same Idempotency-Key + DIFFERENT payload → returns 409 CONFLICT_001 (key reuse with different body).
[00242] [Normal] •  Scope: per-user, per-endpoint. Different users can use same key without conflict.
[00243] [Normal] •  Client MUST regenerate key for genuinely new requests, not retry same key for different operations.
[00244] [Normal] •  Network failure / timeout: client retries with SAME key — server deduplicates automatically.
[00245] [Normal] •  After 24h TTL, key expires — same key can be reused for new operation (uncommon, but safe).
[00246] [Heading 2] 14.3 Example: Payment with Idempotency
[00247] [Normal] Network failure झाला तर client समान Idempotency-Key ने retry करतो — server automatically deduplicates. ह्यामुळे duplicate payment होण्याचा risk zero.
[00249] [Heading 1] 15. Rate Limiting
[00250] [Heading 2] 15.1 Rate Limit Tiers
[00251] [Normal] Rate limiting tiered आहे — different client types साठी different limits. Anonymous requests (login, OTP) सर्वात कमी, authenticated users standard, service accounts high, premium tenants custom.
[00252] [Heading 2] 15.2 Rate Limit Headers
[00253] [Normal] प्रत्येक response मध्ऍे rate limit info headers include होतात — clients ला limit awareness मिळतो, proactive backoff करता येते.
[00254] [Heading 2] 15.3 429 Response
[00255] [Normal] Rate limit exceeded झाल्यावर 429 Too Many Requests response return होतो, Retry-After header सह. Clients MUST wait Retry-After seconds before retrying. Aggressive retry (ignoring Retry-After) additional rate limit penalty trigger करतो.
[00256] [Heading 2] 15.4 Rate Limit Rules
[00257] [Normal] •  Burst allowance: 2x the limit for 10 seconds (allows for page-load burst).
[00258] [Normal] •  Per-IP limit prevents single-client DDoS. Per-user limit prevents single-account abuse.
[00259] [Normal] •  Cloudflare WAF in front for volumetric DDoS (1M+ req/s mitigation).
[00260] [Normal] •  AI APIs have additional per-tenant daily quota — enforced via token counter in Redis.
[00261] [Normal] •  Premium tenants can request custom limits via Platform Admin UI — billed accordingly.
[00262] [Normal] •  Rate limit windows: 1 minute (sliding window algorithm in Redis).
[00263] [Normal] •  Long-polling/WebSocket connections do not count against rate limit (different resource model).

[00433] [Heading 1] 17. WebSocket APIs
[00434] [Heading 2] 17.1 WebSocket Channels
[00435] [Normal] PreOne 5 WebSocket channels provide करतो real-time communication साठी — chat, attendance live updates, notifications, dashboard KPIs, आणि bus tracking. WebSocket API REST API साठी complementary आहे — same auth (JWT), same multi-tenant headers (passed via query param or initial auth message).
[00436] [Heading 2] 17.2 Connection Lifecycle
[00437] [Normal] WebSocket connection lifecycle 8 steps मध्ऍे managed होतो:
[00438] [Normal] •  1. Client opens WebSocket: wss://api.preone.in/ws/chat?token=eyJ...
[00439] [Normal] •  2. Server validates JWT, sends 'connected' event with sessionId
[00440] [Normal] •  3. Client subscribes to channels: { event: 'subscribe', channel: 'room:01HROOM' }
[00441] [Normal] •  4. Server confirms subscription, starts forwarding events for that channel
[00442] [Normal] •  5. Client sends messages, receives messages in real-time
[00443] [Normal] •  6. Heartbeat: client sends ping every 30s, server responds pong (timeout 60s = disconnect)
[00444] [Normal] •  7. Disconnect: client sends 'close' or TCP connection drops — server cleans up subscriptions
[00445] [Normal] •  8. Reconnect: client uses exponential backoff (1s, 2s, 4s, 8s, 16s, max 30s) with jitter
[00446] [Heading 2] 17.3 Message Envelope
[00447] [Normal] प्रत्येक WebSocket message consistent envelope structure follow करतो — event name, payload, eventId (for dedup), timestamp. ह्या consistency मुळे client-side parsing uniform राहतो.
[00448] [Heading 2] 17.4 Authentication
[00449] [Normal] WebSocket authentication JWT द्वारे — initial connection request मध्ऍे ?token=eyJ... query param, किंवा connected event नंतर auth message. Server validates JWT, returns connected event with sessionId. Invalid/expired token = 401 close code.
[00450] [Heading 2] 17.5 Subscription Scoping
[00451] [Normal] Clients subscribe to specific channels (room:01HROOM, class:01HCL, branch:01HBR). Server ने scope validate करावे — teacher फक्त स्वतःच्या classroom च room subscribe करू शकतो, सर्व branches नाही. Subscription scope JWT scope शी consistent असणे mandatory.
[00452] [Heading 2] 17.6 Heartbeat & Reconnection
[00453] [Normal] Heartbeat ping/pong दर 30 seconds — client ping, server pong. 60 seconds timeout = disconnect. Reconnection exponential backoff (1s, 2s, 4s, 8s, 16s, max 30s) with random jitter — thundering herd problem prevent करण्यासाठी.

[00462] [Heading 2] 18.4 Webhook Delivery
[00463] [Normal] Webhook delivery semantics आणि retry policy:
[00464] [Normal] •  At-least-once delivery — recipients MUST be idempotent (dedupe by eventId).
[00465] [Normal] •  HMAC-SHA256 signature in X-PreOne-Signature header — recipient MUST verify.
[00466] [Normal] •  Replay window: 5 minutes (timestamp in payload — older payloads rejected by recipient).
[00467] [Normal] •  Retry policy: 5 attempts with exponential backoff (1min, 5min, 30min, 2h, 12h).
[00468] [Normal] •  Dead letter queue: after 5 failed attempts, event lands in DLQ for manual review.
[00469] [Normal] •  Webhook audit log: every delivery attempt logged (timestamp, status code, latency).
[00470] [Normal] •  Recipient response: 2xx = success, 4xx = do not retry (recipient error), 5xx = retry.
[00471] [Normal] •  Webhook testing: Postman Echo endpoint + integration test environment.

[00493] [Heading 1] 20. File Upload APIs
[00494] [Heading 2] 20.1 Supported Formats
[00495] [Normal] PreOne 5 file formats supports करते — PDF, DOCX, PNG, JPEG, MP4. प्रत्येक format चा size limit + processing pipeline defined आहे. Unsupported formats VALIDATION_001 ने rejected.
[00496] [Heading 2] 20.2 Upload Flow
[00497] [Normal] File upload 6-step flow मध्ऍे done आहे — pre-signed URL pattern वापरून direct-to-S3 upload (PreOne server bandwidth वाचवण्यासाठी).
[00498] [Normal] •  Step 1: Client calls POST /api/v1/files with { filename, contentType, size, purpose } — server validates format/size.
[00499] [Normal] •  Step 2: Server responds with { uploadUrl, fileId, expiresAt } — pre-signed S3 URL valid for 15 min.
[00500] [Normal] •  Step 3: Client uploads file directly to S3 via PUT (no PreOne server in path — saves bandwidth).
[00501] [Normal] •  Step 4: Client calls POST /api/v1/files/{fileId}/complete — server verifies upload, triggers virus scan + processing.
[00502] [Normal] •  Step 5: Server responds with { status: 'READY', downloadUrl, variants: { thumbnail, medium } }.
[00503] [Normal] •  Step 6: File is now ready for use — referenced by fileId in other API calls (e.g., observation.photos).
[00504] [Normal] •  Multipart upload: for files >5MB, S3 multipart upload API is used (10MB parts, max 1000 parts = 10GB).
[00505] [Normal] •  DPDP compliance: file deleted when parent withdraws consent or student archived (7y retention then hard-delete).
[00506] [Heading 2] 20.3 Virus Scan
[00507] [Normal] सर्व uploads ClamAV द्वारे scanned. Infected files immediately deleted + flagged. Upload marked REJECTED, client ला SYSTEM_001 error. Repeated infected uploads from same user trigger security investigation.
[00508] [Heading 2] 20.4 Image Optimization
[00509] [Normal] Image uploads (PNG, JPEG) automatic optimization — Sharp library द्वारे resize variants (thumbnail 100x100, medium 400x400, large 1200x1200), EXIF strip (privacy), quality compress (80% default). Original + variants S3 मध्ऍे stored.
[00510] [Heading 2] 20.5 Video Transcoding
[00511] [Normal] Video uploads (MP4) HLS transcoding — adaptive bitrate (240p, 480p, 720p, 1080p), thumbnail extraction at 10% mark. HLS playlist + segments S3 मध्ऍे stored, CloudFront CDN द्वारे delivered.
[00512] [Heading 2] 20.6 DPDP Compliance
[00513] [Normal] Files DPDP compliant — (1) Consent logged at upload time (parent consent for child photos); (2) Retention 7 years post-student-archival; (3) Withdraw consent = cascade delete file + variants + DB reference; (4) Breach notification 72h अंतर्गत if file storage compromised.

[00534] [Normal] AI safety paramount आहे — child data privacy + teacher oversight. खालील principles enforced:

[00544] [Heading 1] 22. Notification APIs
[00545] [Heading 2] 22.1 Notification Channels
[00546] [Normal] PreOne 5 notification channels supports करतो — Push (mobile), SMS, WhatsApp, Email, In-App. प्रत्येक event चा routing rule define करतो कोणत्या channel(s) मध्ऍे deliver होईल. Multi-channel delivery supported — same notification SMS + Push + In-App मध्ऍे एकाच वेळी.
[00547] [Heading 2] 22.2 Notification Routing
[00548] [Normal] Notification routing rules complex आहेत — per-event, per-recipient, per-time-window, per-priority. Routing engine BullMQ queue मध्ऍे events ला subscribe करतो, applies routing rules, fans out to channels.
[00549] [Normal] •  Per-event routing rules: each domain event (StudentCreated, FeePaid, etc.) maps to ≥1 channel + template.
[00550] [Normal] •  Per-recipient rules: parent vs teacher vs admin get different notification content for same event.
[00551] [Normal] •  Per-time-window rules: quiet hours (10pm–7am) — non-critical notifications batched and sent at 7am.
[00552] [Normal] •  Priority tiers: CRITICAL (absence alert — all channels immediately), HIGH (fee due — SMS+Push), NORMAL (announcement — In-App+Email), LOW (birthday greeting — In-App only).
[00553] [Normal] •  Opt-in/opt-out management: per-channel consent per user, DPDP compliant (default opt-out for marketing).
[00554] [Normal] •  DND/DNS compliance: TRAI DLT registration for SMS templates, WhatsApp Business opt-in required.
[00555] [Normal] •  Delivery receipts: FCM/APNS delivery status, SMS DLR, Email bounce tracking — all logged for analytics.
[00556] [Normal] •  Notification queue: BullMQ (Redis-backed) — failed notifications retried 3x, then DLQ.
[00557] [Normal] •  Notification analytics: delivery rate, open rate (email/push), click-through rate — per template per channel.
[00558] [Heading 2] 22.3 Opt-In/Opt-Out Management
[00559] [Normal] Per-channel consent per user — DPDP compliant. Default opt-out for marketing communications. Transactional (fee due, absence alert) opt-out नाही. Parents channel preference UI मध्ऍे manage करू शकतात. Consent changes logged audit_log मध्ऍे.
[00560] [Heading 2] 22.4 Delivery Analytics
[00561] [Normal] Notification analytics per template per channel — delivery rate, open rate (email/push), click-through rate (links). Dashboard द्वारे trends visible. Low delivery rate trigger करतो investigation (e.g., SMS DLT template rejected → fix template).
[00563] [Heading 1] 23. Event Contracts
[00564] [Heading 2] 23.1 Domain Events
[00565] [Normal] PreOne 8 internal domain events define करतो — core state changes जे multiple consumers care करतात. Events event sourcing pattern चा basis आहेत — aggregates emit events, event_store मध्ऍे persistently stored, consumers subscribe. यामुळे loose coupling + replay capability मिळते.
[00566] [Heading 2] 23.2 Event Envelope
[00567] [Normal] Event envelope consistent structure follow करतो — eventId (UUID for dedup), eventName, aggregateId, aggregateType, timestamp, version, payload, metadata. Metadata मध्ऍे tenant context + correlation ID + source + schema version.
[00568] [Heading 2] 23.3 Event Rules
[00569] [Normal] •  Event schema registry: JSON Schema per event, versioned (semver), published to internal Schema Store.
[00570] [Normal] •  Event versioning: backward-compatible changes (add fields) increment minor; breaking changes increment major + new event name.
[00571] [Normal] •  Event sourcing pattern: events stored immutably in event_store table; aggregates rebuild state from event log.
[00572] [Normal] •  Consumers: Notification Service (fan-out to channels), Analytics Service (aggregations), Webhook Dispatcher (outbound).
[00573] [Normal] •  Delivery semantics: at-least-once. Consumers MUST be idempotent — dedupe by eventId.
[00574] [Normal] •  Consumer offset tracking: per-consumer per-partition offset in Redis (Kafka-style consumer groups).
[00575] [Normal] •  Dead letter queue: events that fail processing after 5 retries → DLQ for manual review.
[00576] [Normal] •  Event replay: admin can replay events from event_store to rebuild read models or retrigger notifications.
[00577] [Heading 2] 23.4 Event Consumers
[00578] [Normal] 3 primary consumers: (1) Notification Service — events ला notification routing शी map करतो; (2) Analytics Service — events aggregate करून read models build करतो; (3) Webhook Dispatcher — events outbound webhooks द्वारे external systems ला deliver करतो. Each consumer independent — एक consumer down असला तरी इतर consumers affected होत नाहीत.
[00579] [Heading 2] 23.5 Event Replay
[00580] [Normal] Admin UI द्वारे event replay capability — उदा.: bug fix झाल्यावर last 7 days events replay करून read models rebuild. Replay idempotent — consumers eventId द्वारे dedupe. Replay operation audit_log मध्ऍे logged.

[00592] [Normal] •  SDK Generation: openapi-generator outputs TypeScript + Python + Java SDKs (published to internal npm/PyPI/Maven).
[00593] [Normal] •  Contract testing: Schemathesis runs fuzz tests against staging using the spec (random valid inputs).
[00594] [Normal] •  Versioning: spec versioned with API version — /api/v1/openapi.json, /api/v2/openapi.json.
