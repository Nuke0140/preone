   127→[Heading 2] ADR-008: User Groups Only (Drop Role Groups)
   128→[Normal] Context
   129→[Normal] An initial proposal suggested two grouping concepts: User Groups (organizational classification like Teaching Staff, Office, Operations) and Role Groups (grouping of roles like Teaching Group = Class Teacher + Activity Teacher + Coordinator). The intent was to use User Groups for HR reporting and Role Groups for permission bundle assignment.
   130→[Normal] On review, Role Groups add a redundant layer. A role is already a capability bundle — grouping roles into meta-bundles produces three levels of indirection (User → Role Group → Roles → Permissions) when two levels (User → Roles → Permissions) suffice. Role Groups would not add capability, only complexity.
   131→[Normal] Options Considered
   132→[List Paragraph] Both User Groups and Role Groups: Two parallel grouping concepts. More flexibility but redundant — Role Group's job is already done by Permission Bundles (ADR-013).
   133→[List Paragraph] User Groups only: One grouping concept for organizational classification. Roles are flat — no meta-grouping. Permission Bundles handle the role-permission assignment optimization.
   134→[List Paragraph] Role Groups only (no User Groups): Drop organizational classification. Loses HR reporting capability (Teaching Staff count, Office Staff count, etc.).
   135→[Normal] Decision
   136→[Normal] Adopt Option 2: User Groups only. User Groups are an HR/organizational classification (Teaching Staff, Office, Operations, Management, Support) that serves workflow routing, dashboard defaults, and batch operations. Roles remain flat — there is no Role Group concept in the schema.
   137→[Normal] Permission Bundle optimization (ADR-013) replaces what Role Groups would have done — bundling common permission sets for easy role assignment. Role Groups would have been a poor man's Permission Bundles, and the real Permission Bundles make them redundant.
   138→[Normal] Rationale
   139→[Normal] User Groups and Roles serve different purposes. User Groups answer 'who is this person in the organization?' (HR classification). Roles answer 'what can this person do in the system?' (capability). Conflating them produces confusion — a Reception staff member who is also handling Accounts has the User Group 'Office' but two distinct Roles.
   140→[Normal] Role Groups would have introduced a third concept that overlaps with Permission Bundles. The 3-Level Permission Bundle (Business → Module → Action) already provides the 'group of permissions' abstraction that Role Groups would have provided at the role level. Adding Role Groups would force the system to maintain two parallel bundling mechanisms.
   141→[Normal] User Groups as workflow routing inputs (ADR-009) is the killer feature — when an incident is reported, the system routes a notification to the Operations Group; when a leave request is submitted, it routes to the user's reporting manager within their User Group. This makes User Groups an active workflow participant, not just an HR label.

   143→[Heading 3] Positive
   144→[List Paragraph] Single grouping concept — simpler mental model for admins and engineers.
   145→[List Paragraph] User Groups serve double duty as HR classification and workflow routing input.
   146→[List Paragraph] No redundant Role Group layer to maintain or migrate.
   147→[Heading 3] Negative / Trade-offs
   148→[List Paragraph] Schools wanting to group roles (e.g., 'all teaching roles get a default bundle') must use Permission Bundles — which is actually the correct abstraction anyway.

   204→[Heading 2] ADR-009: User Groups as Workflow Engine Input
   205→[Normal] Context
   206→[Normal] User Groups were originally conceived as an HR classification label — Teaching Staff, Office, Operations, etc. The label would be useful for reporting (count of teaching staff) but would not actively drive system behavior.
   207→[Normal] On review, this underutilizes User Groups. Enterprise workflow engines use organizational groupings as routing inputs — when an incident is reported, route to Operations; when a fee is overdue, route to Finance; when a leave request is submitted, route to the user's reporting manager (resolved via group hierarchy). User Groups should be active workflow participants, not passive labels.
   208→[Normal] Options Considered
   209→[List Paragraph] User Groups as HR labels only: Passive classification for reporting. No workflow integration. Simplest but underutilizes the data.
   210→[List Paragraph] User Groups as workflow routing inputs: Active participation — workflow engine queries User Groups to determine notification recipients, approval chains, and default dashboard layouts.
   211→[List Paragraph] Both, but with separate HR and Workflow group tables: Two parallel grouping systems — one for HR, one for workflow. Maximum flexibility but redundant and confusing.
   212→[Normal] Decision
   213→[Normal] Adopt Option 2: User Groups are active workflow routing inputs. The workflow engine queries User Groups to determine: (1) notification recipients for events (incident → Operations Group), (2) approval chains (leave request → reporting manager in user's group), (3) default dashboard layouts (Teaching Staff group → Teacher dashboard), (4) batch operations (broadcast message to Support Staff group).
   214→[Normal] User Groups become a first-class workflow entity, not just an HR tag. Group membership changes trigger workflow re-evaluation (e.g., when Pooja leaves the Operations group, she stops receiving incident notifications).
   215→[Normal] Rationale
   216→[Normal] Workflow routing is the most common reason enterprise systems need organizational groupings. Without workflow integration, User Groups are decorative — admins fill them in but the system never uses them. With workflow integration, User Groups become operational infrastructure that drives daily system behavior.
   217→[Normal] The four workflow use cases (notification routing, approval chains, dashboard defaults, batch operations) cover the majority of real-world preschool workflows. Hardcoding these routes per role would be brittle (role changes break routing); routing via User Groups is resilient (group membership can change without touching workflow definitions).
   218→[Normal] This pattern aligns with industry workflow engines (Camunda, Bonita, Microsoft Power Automate) where organizational units are first-class routing inputs. PreOne's workflow engine will use the same pattern at smaller scale.
   219→[Normal] Consequences
   220→[Heading 3] Positive
   221→[List Paragraph] User Groups actively drive system behavior — admins have incentive to keep them accurate.
   222→[List Paragraph] Workflow routing is decoupled from role definitions — role changes do not break routing.
   223→[List Paragraph] Four workflow use cases covered: notifications, approvals, dashboards, batch operations.
   224→[Heading 3] Negative / Trade-offs
   225→[List Paragraph] Group membership changes must trigger workflow re-evaluation — adds complexity to the membership update API.
   226→[List Paragraph] Misconfigured groups can break workflows (e.g., empty Operations Group means incident notifications go nowhere) — mitigated by validation that prevents empty system groups.
   227→[Normal] Implementation Notes
   228→[Normal] The workflow engine subscribes to group membership change events. Workflow definitions reference groups by name (e.g., route_to: 'operations') and the engine resolves names to user lists at runtime.

   287→[Heading 2] ADR-010: Context-Based Access (Role Switch UI Only)
   288→[Normal] Context
   289→[Normal] When a user holds multiple roles (Pooja is Class Teacher + Activity Teacher + Inventory Incharge), two questions arise: (1) Should the user explicitly switch roles to access different modules? (2) Should permissions be the union of all roles, or scoped to the currently-active role? Early enterprise systems forced explicit role switching — the user clicked a dropdown, selected 'Inventory Incharge', and only then could access the Inventory module.
   290→[Normal] Role switching creates friction. Pooja, mid-task in the Today module (Class Teacher context), wants to glance at Inventory stock levels. Forcing a role switch is a context-breaking interaction. The system should know from her role union that she has Inventory access and apply the correct scope automatically based on the module she's accessing.
   291→[Normal] Options Considered
   292→[List Paragraph] Explicit role switching: User selects active role from a dropdown. Only that role's permissions apply. Friction for multi-role users; impossible to access Inventory while in Class Teacher mode.
   293→[List Paragraph] Union of all roles, broadest scope wins: All role permissions unioned; data scope is broadest among roles. Security risk — Inventory Incharge role's whole-school scope leaks into Class Teacher queries.
   294→[List Paragraph] Union of permissions, scope-per-role-per-module: Permissions unioned across all roles. Data scope is determined per-module by finding all roles that grant access to that module and unioning their assignments.
   295→[Normal] Decision
   296→[Normal] Adopt Option 3: Context-Based Access. Permissions are unioned across all of the user's roles. Data scope is computed per-module at query time: for each module the user accesses, the system finds all roles granting access to that module, retrieves their assignments, and unions the scopes.
   297→[Normal] Role switching exists only as UI convenience — a user can filter their dashboard to show 'only Class Teacher view' or 'only Inventory view', but this is a presentation choice, not a security boundary. The user always has their full role union's permissions, regardless of UI role switcher state.
   298→[Normal] Rationale
   299→[Normal] Context-Based Access is the natural model for multi-role users. Pooja accessing the Inventory module sees whole-school inventory because her Inventory Incharge role grants that scope. Pooja accessing the Today module sees only Jr.KG B because her Class Teacher role is scoped to that class. The system determines scope from the module context, not from an explicit role selection.
   300→[Normal] Explicit role switching (Option 1) is a usability regression — it forces the user to think about 'which role am I in?' instead of just doing their work. Union-with-broadest-scope (Option 2) is a security regression — it leaks scope from one role into another. Context-Based Access avoids both pitfalls.
   301→[Normal] This pattern matches how Microsoft Azure and AWS IAM handle multi-role users — permissions are unioned, scopes are determined by the resource being accessed, and role switching is a UI convenience only.
   302→[Normal] Consequences
   303→[Heading 3] Positive
   304→[List Paragraph] Multi-role users get seamless access without role switching.
   305→[List Paragraph] Data scope is always correct per module — no scope leakage between roles.
   306→[List Paragraph] UI role switcher is optional, not mandatory — users who never switch roles still get correct behavior.
   307→[Heading 3] Negative / Trade-offs
   308→[List Paragraph] Permission evaluation is more complex than single-role — mitigated by ADR-016 Effective Access Resolver.
   309→[List Paragraph] Caching is essential because per-module scope computation is expensive — mitigated by ADR-021 Permission Caching.

   375→[Heading 2] ADR-011: Scope-Per-Role-Per-Module
   376→[Normal] Context
   377→[Normal] ADR-010 established that data scope is computed per-module, not globally. The remaining question is the exact algorithm: when a user holds multiple roles that all grant access to the same module, how are their scopes combined? Pooja holds Class Teacher (scope: Jr.KG B) and Activity Teacher (scope: Art + Music activities) — both roles grant access to the Observations module. When Pooja views Observations, what does she see?
   378→[Normal] A naive implementation might pick the broadest scope (whole school), but that would leak Activity Teacher's broad activity scope into Class Teacher's class-scoped queries. A stricter implementation might require the user to pick which role's scope to use, but that reintroduces explicit role switching. The correct algorithm must union scopes from all relevant roles without leaking scope between roles.
   379→[Normal] Options Considered
   380→[List Paragraph] Broadest scope wins: Pick the broadest scope among all roles granting module access. Pooja sees all observations school-wide. Simple but insecure — scope leaks across roles.
   381→[List Paragraph] Scope intersection: Find the intersection of scopes across roles. Pooja sees only observations that are both in Jr.KG B AND in Art/Music. Likely empty — too strict.
   382→[List Paragraph] Scope union per role, then union across roles: For each role granting module access, gather its scope. Union all scopes across all relevant roles. Pooja sees Jr.KG B observations + Art observations + Music observations.
   383→[Normal] Decision
   384→[Normal] Adopt Option 3: Scope-Per-Role-Per-Module. For each module the user accesses, the system finds all roles that grant access to that module. For each such role, it retrieves the role's assignments (which classes, branches, activities, etc.). All these scopes are unioned to produce the effective scope for that module.
   385→[Normal] Critical rule: scopes are unioned, but they are never 'promoted'. A class-scoped role does not become school-scoped just because another role has school scope. Pooja's Class Teacher scope (Jr.KG B) and Activity Teacher scope (Art + Music activities) are unioned — she sees observations from Jr.KG B, Art, and Music. She does not see observations from other classes or other activities.
   386→[Normal] Rationale
   387→[Normal] Scope-Per-Role-Per-Module is the only algorithm that respects both multi-role reality and security boundaries. Pooja, as Class Teacher for Jr.KG B and Activity Teacher for Art + Music, legitimately needs to see observations from all three contexts — she teaches in all three. But she does not need to see observations from Sr.KG A or from Sports class, because she has no role there.
   388→[Normal] The 'never promote scope' rule prevents the most dangerous failure mode — a low-scope role gaining high-scope access because the user happens to also hold a high-scope role. This would violate the principle of least privilege and create audit nightmares (Pooja viewing Sr.KG A observations would be inexplicable from her role assignments).
   389→[Normal] This algorithm is what enterprise document management systems (SharePoint, Box) use for permission inheritance — each role contributes its scope, scopes are unioned, but a role's scope never expands beyond what was explicitly assigned.
   390→[Normal] Consequences
   391→[Heading 3] Positive
   392→[List Paragraph] Multi-role users see exactly what they need — no more, no less.
   393→[List Paragraph] No scope leakage between roles — security boundary is preserved.
   394→[List Paragraph] Audit trail is clear: every observation Pooja sees can be traced to a specific role assignment.
   395→[Heading 3] Negative / Trade-offs
   396→[List Paragraph] Scope computation is more complex than naive broadest-scope — mitigated by caching (ADR-021).
   397→[List Paragraph] Edge case: a role with 'whole school' scope (e.g., Inventory Incharge) effectively grants that scope for the Inventory module only, not for other modules — implementation must enforce per-module scope isolation.
